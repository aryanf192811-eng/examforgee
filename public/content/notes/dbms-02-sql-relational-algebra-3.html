Chapter 01
Relational Algebra — Core Operators

1.1   What Is Relational Algebra? Core

Relational Algebra (RA) is a procedural query language for the relational model. It is procedural because you specify how to retrieve the data — a sequence of operations applied to relations. This contrasts with SQL, which is declarative (you specify what you want, not how to get it).

RA forms the theoretical foundation of SQL. Every SQL query you write is internally translated by the database engine into a relational algebra expression (a query tree), which is then optimized and executed. Understanding RA means understanding what SQL actually does underneath.

Every RA operation takes one or two relations as input and produces a new relation as output. This closure property — relations in, relation out — lets you compose operations freely.

The Six Fundamental Operators

Selection (σ), Projection (π), Union (∪), Set Difference (−), Cartesian Product (×), Rename (ρ). All other RA operators can be derived from these six.

1.2   Selection — σ GATE Tested

σ
Selection
Filters rows that satisfy a condition. Operates on one relation.

Syntax: σ<condition>(R)

Selection is the horizontal filter — it keeps only the rows (tuples) for which the condition evaluates to true. The output relation has the same schema as the input but fewer (or equal) rows.

The condition can use comparison operators (=, ≠, <, >, ≤, ≥), logical connectives (∧ AND, ∨ OR, ¬ NOT), and can reference attribute names or constants.

Query: Find all CSE students with CGPA above 8.5.

σ(Dept = 'CSE' ∧ CGPA > 8.5)(Student)

Result: Tuples for Aryan (101, CSE, 8.9) and Karan (103, CSE, 9.1).

Important Property

Selection does not change the schema (columns). It only filters rows. The degree of the output equals the degree of the input.

1.3   Projection — π GATE Tested

π
Projection
Keeps only specified columns. Eliminates duplicates automatically.

Syntax: π<attr list>(R)

Projection is the vertical filter — it keeps only the specified columns and discards the rest. Because the result is a relation (a set), any duplicate rows that arise from dropping columns are automatically eliminated.

Query: Get the names and departments of all students.

π(Name, Dept)(Student)

Result: {(Aryan, CSE), (Meera, ECE), (Karan, CSE), (Priya, ME)}

Note: if two students had the same name AND department, the duplicate would be removed.

GATE Trap — Duplicate Elimination

In formal RA, projection always eliminates duplicates (relation = set). In SQL, SELECT without DISTINCT does NOT eliminate duplicates. This distinction is tested explicitly in GATE.

Combining Selection and Projection

These two operators are almost always used together. Convention: selection first (reduce rows), then projection (reduce columns). Example: get names of CSE students with CGPA > 8: π(Name)(σ(Dept='CSE' ∧ CGPA>8)(Student))

1.4   Set Operations — ∪, −, ∩

The set operations treat relations as mathematical sets of tuples. For ∪ and − to be valid, both relations must be union-compatible: same number of attributes, and corresponding attributes must come from the same domain. Intersection (∩) is derived and also requires union-compatibility.

Union — ∪

Definition

R ∪ S produces all tuples that appear in R, or in S, or in both. Duplicates are eliminated.

If Relation A has SIDs {101, 102} and Relation B has SIDs {102, 103}, then A ∪ B = {101, 102, 103}. The 102 duplicate is eliminated.

Set Difference — −

Definition

R − S produces all tuples that appear in R but NOT in S.

Query: Find students who are NOT enrolled in any course.

π(SID)(Student) − π(SID)(Enrollment)

Project all SIDs from Student, subtract those that appear in Enrollment. Result: {104} (Priya).

Intersection — ∩

Definition

R ∩ S produces tuples that appear in BOTH R and S. Derived: R ∩ S = R − (R − S).

Union Compatibility

All three set operations require the operand relations to have the same number of attributes with compatible domains. Applying them to relations with different schemas is undefined. GATE sometimes presents such a case — recognize it as invalid immediately.

1.5   Cartesian Product — × GATE Tested

×
Cartesian Product
Every tuple from R paired with every tuple from S. No condition.

Syntax: R × S

Cartesian product generates every possible combination of a tuple from R with a tuple from S. If R has m tuples and degree p, and S has n tuples and degree q, then R × S has m × n tuples and degree p + q.

Student has 4 rows, Enrollment has 4 rows. Student × Enrollment has 4 × 4 = 16 rows and 4 + 3 = 7 columns. Most of these 16 rows are meaningless — they pair, say, Priya with a course she never enrolled in.

Cartesian product by itself is rarely useful. Its value is as a building block: when followed by a selection on the join condition, it becomes a join. This is exactly how join is defined in formal RA.

Naming Conflict

When both relations have an attribute with the same name (e.g., both have SID), the resulting relation has two columns both named SID. To disambiguate, use the relation name as a prefix: Student.SID and Enrollment.SID. This is where the Rename operator becomes necessary.

1.6   Rename — ρ

ρ
Rename
Assigns a new name to a relation or its attributes. No data change.

Syntax: ρ<new_name>(R)   or   ρ<new_name(a1, a2, ...)>(R)

Rename is necessary in two situations: (1) when a Cartesian product creates ambiguous attribute names, and (2) when you need to self-join a relation — join a table with itself. Without rename, you cannot distinguish the two copies of the same relation in a self-join.

Query: Find pairs of students in the same department.

ρ(S1)(Student) × ρ(S2)(Student) — then filter: σ(S1.Dept = S2.Dept ∧ S1.SID ≠ S2.SID)(...)

Without rename, you can't distinguish the two copies of Student.

Chapter 02
Extended RA — Joins & More

2.1   Natural Join — ⋈ Most Tested

⋈
Natural Join
Joins on all common attribute names. Keeps one copy of each common attribute.

Definition: R ⋈ S = π<unique attrs>(σ<equal common attrs>(R × S))

Natural join automatically finds all attributes with the same name in both relations, matches tuples where those attributes are equal, and returns a single copy of the common attributes in the result.

Student and Enrollment both have SID. Natural join matches on SID, combining each student's info with their enrollment records.

Student ⋈ Enrollment:

SIDNameDeptCGPACourseIDGrade
101AryanCSE8.9CS301A
101AryanCSE8.9CS302B
102MeeraECE7.4CS301A
103KaranCSE9.1CS302A

Note: Priya (104) does not appear — she has no enrollment record. Natural join is an inner join by default.

GATE Trap — Natural Join Pitfall

Natural join joins on ALL attributes with the same name. If two unrelated relations happen to share an attribute name (say both have a column called ID), natural join will incorrectly join on it. Always be explicit about which attributes you intend to join on — this is why SQL uses JOIN ... ON instead of relying on name matching alone.

2.2   Theta Join and Equijoin

Theta Join — ⋈θ

Definition

R ⋈θ S = σθ(R × S). It is a Cartesian product followed by selection on condition θ. The condition can be any comparison — not just equality.

This is the most general join. Any condition works: equalities, inequalities, range conditions.

Equijoin

Definition

A theta join where θ uses only equality conditions. Unlike natural join, equijoin keeps both copies of the join attributes (one from each relation).

Natural join vs equijoin on Student ⋈ Enrollment on SID:

  • Natural Join: Result has one SID column (common attribute merged).
  • Equijoin: Result has two columns — Student.SID and Enrollment.SID — both with the same values.

Natural join = equijoin + projection to remove duplicate join columns.

2.3   Outer Joins GATE Tested

All joins discussed so far are inner joins — they discard tuples that don't find a matching partner. Outer joins preserve unmatched tuples by padding them with NULLs.

Left Outer Join ⟕

All tuples from the left relation are kept. Unmatched left tuples are padded with NULL for the right relation's attributes.

Right Outer Join ⟖

All tuples from the right relation are kept. Unmatched right tuples are padded with NULL for the left relation's attributes.

Full Outer Join ⟗

All tuples from both relations are kept. Unmatched tuples from either side are padded with NULL.

Student LEFT OUTER JOIN Enrollment on SID:

SIDNameDeptCGPACourseIDGrade
101AryanCSE8.9CS301A
101AryanCSE8.9CS302B
102MeeraECE7.4CS301A
103KaranCSE9.1CS302A
104PriyaME6.8NULLNULL

Priya is preserved with NULLs for the missing enrollment data.

2.4   Division — ÷ Frequently Asked

÷
Division
"Find all X that are associated with every Y." The universal quantification operator.

Division answers the class of queries that use "for all" or "every." Given R(A, B) ÷ S(B), the result contains every value of A in R such that for every tuple in S, there exists a matching tuple in R.

Formal definition: R ÷ S = πA(R) − πA((πA(R) × S) − R)

Query: Find students who are enrolled in ALL courses that Aryan (101) is enrolled in.

Aryan is in {CS301, CS302}. We want SIDs such that for every course in that set, there is an enrollment record.

Let R = π(SID, CourseID)(Enrollment) Let S = π(CourseID)(σ(SID = 101)(Enrollment)) = {CS301, CS302} R ÷ S → {101} (Only Aryan himself is enrolled in both CS301 and CS302.)
Recognition Pattern

Whenever a query has "all", "every", "each", or "for all" — think division. It is the only RA operator that naturally expresses universal quantification.

2.5   RA Equivalences — Query Optimization Advanced

The database query optimizer uses algebraic equivalences to rewrite a query into a more efficient form. These are tested in GATE in the form of "which two RA expressions are equivalent?"

RuleEquivalenceWhy It Matters
Selection cascadeσc1 ∧ c2(R) = σc1(σc2(R))Apply tighter filter first to reduce intermediate size
Selection commutativityσc1(σc2(R)) = σc2(σc1(R))Order of selection doesn't matter
Projection cascadeπL1(πL2(R)) = πL1(R) if L1 ⊆ L2Outer projection dominates
Selection before joinσc(R ⋈ S) = σc(R) ⋈ S if c involves only R's attrsPush selection down — key optimization
Join commutativityR ⋈ S = S ⋈ RJoin order can be swapped
Join associativity(R ⋈ S) ⋈ T = R ⋈ (S ⋈ T)Grouping of multi-way joins is flexible
The Key Optimization Principle

Push selections and projections as far down the query tree as possible — closer to the leaves. This reduces the size of intermediate relations before expensive operations like joins happen. A Cartesian product of two 1-million-row tables is catastrophic; a Cartesian product of a 10-row filtered result is trivial.

Chapter 03
SQL Foundations — DDL & DML

3.1   SQL — Nature and Classification

SQL (Structured Query Language) is a declarative language — you describe what data you want, not how to retrieve it. The database engine's query optimizer decides the execution strategy. SQL is built on the relational algebra but extends it significantly: it allows duplicate rows (multiset semantics), NULLs, aggregation, ordering, and much more.

DDL — Data Definition

CREATE, ALTER, DROP, TRUNCATE. Defines and modifies schema structure.

DML — Data Manipulation

SELECT, INSERT, UPDATE, DELETE. Operates on the data itself.

DCL — Data Control

GRANT, REVOKE. Manages permissions and access rights.

TCL — Transaction Control

COMMIT, ROLLBACK, SAVEPOINT. Manages transactions (covered in File 3).

3.2   DDL — Defining Structure

CREATE TABLE

Defines a new relation with its attributes, data types, and constraints.

CREATE TABLE Student ( SID INT PRIMARY KEY, Name VARCHAR(50) NOT NULL, Dept VARCHAR(10), CGPA DECIMAL(3,1) CHECK (CGPA >= 0 AND CGPA <= 10), FOREIGN KEY (Dept) REFERENCES Department(DeptName) );

ALTER TABLE

Modifies an existing table — add, drop, or modify columns and constraints.

ALTER TABLE Student ADD Email VARCHAR(100); ALTER TABLE Student DROP COLUMN Email; ALTER TABLE Student MODIFY Name VARCHAR(100);

DROP vs TRUNCATE vs DELETE

CommandWhat It DoesRollback Possible?Removes Schema?
DROP TABLERemoves table + all data + schemaNo (DDL — auto-committed)Yes
TRUNCATE TABLERemoves all rows, resets auto-incrementNo (DDL in most DBs)No
DELETE FROMRemoves rows matching a conditionYes (DML — transactional)No
GATE Trap

TRUNCATE is DDL in most databases (cannot be rolled back), but DELETE is DML (can be rolled back within a transaction). TRUNCATE is faster because it doesn't log individual row deletions. This distinction appears in GATE transaction and recovery questions.

3.3   The SELECT Statement — Anatomy GATE Critical

The full structure of a SQL SELECT statement, in the order you write it:

SELECT [DISTINCT] <attribute list or expressions> FROM <table list or join expressions> [WHERE <condition>] [GROUP BY <grouping attributes>] [HAVING <group condition>] [ORDER BY <sort attributes> [ASC|DESC]] [LIMIT <n>];
Logical Execution Order — Different from Writing Order

SQL is written in one order but executed in a different order. GATE frequently tests this by asking what's valid in WHERE vs HAVING, or whether an alias defined in SELECT can be used in WHERE.

StepClauseWhat Happens
1FROMIdentify source tables; compute joins/Cartesian products
2WHEREFilter individual rows — no aggregates allowed here
3GROUP BYGroup remaining rows by specified attributes
4HAVINGFilter groups — aggregates allowed here
5SELECTCompute output columns and expressions
6DISTINCTRemove duplicate rows from output
7ORDER BYSort the final result
8LIMITReturn only the first n rows

Why this matters: The query below is INVALID:

SELECT Dept AS Department, COUNT(*) AS cnt FROM Student WHERE cnt > 2; -- ERROR: 'cnt' is not defined yet when WHERE executes

WHERE executes before SELECT, so the alias cnt doesn't exist yet. Use HAVING instead, which runs after GROUP BY.

DISTINCT

SELECT DISTINCT removes duplicate rows from the result. Without DISTINCT, SQL operates in multiset (bag) semantics — duplicates are preserved. Formal relational algebra always eliminates duplicates. This is a fundamental SQL vs RA difference.

SELECT DISTINCT Dept FROM Student; -- Returns: CSE, ECE, ME (3 rows, not 4 — CSE appears only once) SELECT Dept FROM Student; -- Returns: CSE, ECE, CSE, ME (4 rows — both CSE rows preserved)

3.4   DML — INSERT, UPDATE, DELETE

INSERT

-- Insert a single row with explicit column list INSERT INTO Student (SID, Name, Dept, CGPA) VALUES (105, 'Rahul', 'CSE', 7.8); -- Insert result of a SELECT query INSERT INTO CSE_Students SELECT * FROM Student WHERE Dept = 'CSE';

UPDATE

-- Update specific rows UPDATE Student SET CGPA = CGPA + 0.1 WHERE Dept = 'CSE'; -- Without WHERE: updates ALL rows (common mistake) UPDATE Student SET CGPA = 10.0; -- Every student now has CGPA = 10

DELETE

-- Delete specific rows DELETE FROM Student WHERE CGPA < 6.0; -- Without WHERE: deletes ALL rows (table structure remains) DELETE FROM Student; -- Empty table, schema intact
The Missing WHERE Clause Trap

UPDATE and DELETE without a WHERE clause affect every row in the table. GATE has presented this as a reasoning question: "After executing UPDATE Student SET CGPA = 0, how many rows have CGPA = 0?" Answer: all rows.

Chapter 04
Joins in SQL

4.1   All Join Types — Complete Reference GATE Tested

SQL joins combine rows from two or more tables based on a related column. Each join type handles unmatched rows differently.

INNER JOIN

Returns only rows where the join condition is satisfied in both tables. Equivalent to natural join / equijoin in RA. Unmatched rows from either table are discarded.

SELECT S.Name, E.CourseID, E.Grade FROM Student S INNER JOIN Enrollment E ON S.SID = E.SID;

Result: 4 rows — Priya (104) is excluded because she has no enrollment.

LEFT OUTER JOIN

Returns all rows from the left table, plus matched rows from the right. Unmatched left rows get NULL for right table's columns.

SELECT S.Name, E.CourseID FROM Student S LEFT JOIN Enrollment E ON S.SID = E.SID;

Result: 5 rows — Priya appears with CourseID = NULL.

RIGHT OUTER JOIN

Mirror of LEFT JOIN — all rows from right table are preserved, unmatched right rows get NULL for left table columns.

FULL OUTER JOIN

Union of LEFT and RIGHT JOIN. All rows from both tables appear. Unmatched rows on either side get NULL for the other table's columns.

CROSS JOIN

Cartesian product. Every row in left table is paired with every row in right table. No join condition. Rarely useful in practice, but often tested for result-size calculations.

SELECT * FROM Student CROSS JOIN Enrollment; -- 4 students × 4 enrollments = 16 rows

NATURAL JOIN

Automatically joins on all columns with the same name. Keeps one copy of the join column. Dangerous in practice — avoid if column name overlap is coincidental.

SELECT * FROM Student NATURAL JOIN Enrollment; -- Joins on SID (only common column), returns 4 rows
Join TypeUnmatched rows in LeftUnmatched rows in Right
INNER JOINDiscardedDiscarded
LEFT JOINKept (NULLs for right cols)Discarded
RIGHT JOINDiscardedKept (NULLs for left cols)
FULL OUTER JOINKept (NULLs)Kept (NULLs)
CROSS JOINNo condition — every pair is produced

4.2   Self Join

A self-join joins a table with itself. You must use aliases to distinguish the two copies. This is used to compare rows within the same table.

Query: Find all pairs of students in the same department.

SELECT A.Name AS Student1, B.Name AS Student2, A.Dept FROM Student A JOIN Student B ON A.Dept = B.Dept AND A.SID < B.SID; -- prevents (Aryan,Karan) AND (Karan,Aryan)

The condition A.SID < B.SID ensures each pair appears only once and no student is paired with themselves.

Chapter 05
Subqueries & Nested SQL

5.1   What Is a Subquery?

A subquery (also called an inner query or nested query) is a complete SQL SELECT statement embedded inside another SQL statement. The outer query uses the result of the inner query. Subqueries can appear in the SELECT, FROM, WHERE, or HAVING clauses.

Scalar Subquery

Returns exactly one row and one column — a single value. Can be used anywhere a single value is expected.

-- Find students with above-average CGPA SELECT Name, CGPA FROM Student WHERE CGPA > (SELECT AVG(CGPA) FROM Student);

Row Subquery

Returns one row with multiple columns. Used with row-level comparison operators.

Table Subquery

Returns multiple rows and columns. Used in FROM clause (as a derived table) or with IN, ANY, ALL, EXISTS operators.

-- Subquery in FROM clause (derived table / inline view) SELECT Dept, avg_cgpa FROM ( SELECT Dept, AVG(CGPA) AS avg_cgpa FROM Student GROUP BY Dept ) AS dept_avg WHERE avg_cgpa > 8.0;

5.2   IN, NOT IN, ANY, ALL, EXISTS GATE Critical

IN and NOT IN

-- Students enrolled in CS301 SELECT Name FROM Student WHERE SID IN (SELECT SID FROM Enrollment WHERE CourseID = 'CS301'); -- Students NOT enrolled in any course SELECT Name FROM Student WHERE SID NOT IN (SELECT SID FROM Enrollment);
NOT IN with NULLs — Killer Trap

If the subquery in NOT IN returns any NULL value, the entire NOT IN condition evaluates to UNKNOWN (not FALSE), which means no rows are returned. This is one of the most dangerous SQL NULL traps. Always use NOT EXISTS when NULLs may be present, or add WHERE col IS NOT NULL inside the subquery.

ANY and ALL

Definition

val op ANY (subquery) — true if the condition holds for AT LEAST ONE value in the subquery result. val op ALL (subquery) — true if the condition holds for EVERY value in the subquery result.

-- Students with CGPA greater than at least one ECE student SELECT Name FROM Student WHERE CGPA > ANY (SELECT CGPA FROM Student WHERE Dept = 'ECE'); -- Students with CGPA greater than ALL ECE students (> max) SELECT Name FROM Student WHERE CGPA > ALL (SELECT CGPA FROM Student WHERE Dept = 'ECE');
= ANY

Equivalent to IN. True if the value equals at least one in the set.

<> ALL

Equivalent to NOT IN. True if the value differs from every value in the set.

> ALL

Equivalent to > MAX(...). Must exceed the largest value.

> ANY

Equivalent to > MIN(...). Must exceed the smallest value.

EXISTS and NOT EXISTS

Definition

EXISTS (subquery) returns TRUE if the subquery produces at least one row. FALSE if the result is empty. Does not care about the actual values — just whether any row exists.

-- Students who have at least one enrollment SELECT Name FROM Student S WHERE EXISTS ( SELECT 1 FROM Enrollment E WHERE E.SID = S.SID -- correlated subquery ); -- Students with NO enrollments SELECT Name FROM Student S WHERE NOT EXISTS ( SELECT 1 FROM Enrollment E WHERE E.SID = S.SID );
EXISTS vs IN — When to Use Which

Use IN when the subquery is small and returns a list you want to match against. Use EXISTS when the subquery is large, when NULLs may be present (EXISTS handles NULLs safely), or when the subquery is correlated. NOT EXISTS is almost always safer than NOT IN when NULLs are possible.

5.3   Correlated Subqueries Advanced

A correlated subquery references a column from the outer query. It is re-evaluated once for each row processed by the outer query — making it conceptually like a nested loop. This is different from an uncorrelated subquery, which is evaluated once and its result reused.

Query: Find students whose CGPA is above the average CGPA of their own department.

SELECT Name, Dept, CGPA FROM Student S1 WHERE CGPA > ( SELECT AVG(CGPA) FROM Student S2 WHERE S2.Dept = S1.Dept -- correlated: references outer S1.Dept );

For each row in S1, the inner query computes the average CGPA for that row's department. This cannot be done with a simple uncorrelated subquery.

Chapter 06
Aggregation, GROUP BY & HAVING

6.1   Aggregate Functions GATE Tested

Aggregate functions compute a single value from a set of rows. They collapse multiple rows into one summary value. They appear in SELECT and HAVING clauses (never in WHERE — WHERE runs before grouping).

FunctionWhat It ComputesNULL Behavior
COUNT(*)Total number of rowsCounts ALL rows including NULLs
COUNT(col)Number of non-NULL values in colIgnores NULL values
COUNT(DISTINCT col)Number of distinct non-NULL valuesIgnores NULLs and duplicates
SUM(col)Sum of all values in colIgnores NULLs
AVG(col)Average: SUM / COUNT of non-NULLsIgnores NULLs (NOT divided by total rows)
MIN(col)Smallest valueIgnores NULLs
MAX(col)Largest valueIgnores NULLs
AVG and NULLs — Classic GATE Trap

If a column has values {8, 9, NULL, NULL, NULL} for 5 rows, AVG(col) = (8+9)/2 = 8.5, NOT (8+9)/5 = 3.4. AVG ignores NULLs in both numerator and denominator. But COUNT(*) = 5, while COUNT(col) = 2. These differences are a recurring GATE question source.

6.2   GROUP BY and HAVING GATE Critical

GROUP BY

Partitions rows into groups based on one or more attributes. All rows with the same value(s) for the GROUP BY attribute(s) form one group. Aggregates are then computed per group.

HAVING

Filters groups after GROUP BY is applied. Can use aggregate functions. HAVING is to groups what WHERE is to rows.

Query: For each department with more than 1 student, find the average CGPA.

SELECT Dept, COUNT(*) AS num_students, AVG(CGPA) AS avg_cgpa FROM Student GROUP BY Dept HAVING COUNT(*) > 1;

Execution:

  1. 1
    FROM: load Student.
  2. 2
    GROUP BY Dept: form groups — {CSE: Aryan, Karan}, {ECE: Meera}, {ME: Priya}.
  3. 3
    HAVING: keep groups with COUNT(*) > 1. Only CSE qualifies.
  4. 4
    SELECT: compute AVG(CGPA) for CSE = (8.9 + 9.1)/2 = 9.0.

Result:

Deptnum_studentsavg_cgpa
CSE29.0
The GROUP BY Rule — Most Violated in GATE

Every attribute in the SELECT clause that is NOT inside an aggregate function MUST appear in the GROUP BY clause. Violating this is a syntax error. Example: SELECT Dept, Name, COUNT(*) FROM Student GROUP BY Dept is INVALID because Name is not aggregated and not in GROUP BY.

WHERE vs HAVING — Exact Difference

AspectWHEREHAVING
FiltersIndividual rows (before grouping)Groups (after GROUP BY)
Can use aggregates?NO — aggregates not yet computedYES — aggregates are computed by this point
Can use non-grouped attrs?YESOnly grouped attrs or aggregates
PerformanceFilters early → fasterFilters late → more work done before filter
Optimization Rule

If a condition does not involve an aggregate, put it in WHERE, not HAVING. WHERE filters rows before grouping, reducing the amount of work GROUP BY has to do. HAVING on a non-aggregate condition is valid but inefficient.

Chapter 07
Views & Integrity Constraints

7.1   Views Core

Definition

A view is a named query stored in the database. It appears to the user as a virtual table — you can query it like a regular table — but it does not store data itself. Its data is computed from the underlying base tables each time the view is queried.

-- Create a view CREATE VIEW CSE_Students AS SELECT SID, Name, CGPA FROM Student WHERE Dept = 'CSE'; -- Query the view like a table SELECT * FROM CSE_Students WHERE CGPA > 8.0; -- Drop a view DROP VIEW CSE_Students;
Purpose 1: Simplicity

Hide complex joins and conditions behind a simple name. Users query the view without knowing the underlying schema.

Purpose 2: Security

Expose only certain columns or rows to certain users. Grant SELECT on the view, not the base table.

Purpose 3: Logical Independence

If base table schema changes, update only the view definition — applications using the view are unaffected.

Updatable vs Non-Updatable Views

Not all views can be used for INSERT, UPDATE, DELETE. A view is updatable only if each row in the view maps uniquely to exactly one row in a single base table. A view becomes non-updatable if it contains any of: JOIN, GROUP BY, HAVING, DISTINCT, aggregate functions, subqueries in SELECT, UNION, or derived columns (expressions).

Materialized Views

A materialized view physically stores the result of the query (unlike a regular view). It must be refreshed periodically to stay current. Used for performance on expensive queries that are run frequently. This is a trade-off: stale data vs query speed.

7.2   Integrity Constraints GATE Tested

Integrity constraints are rules enforced by the DBMS to maintain the correctness and consistency of data. They are defined at schema creation time and checked on every INSERT, UPDATE, and DELETE.

Domain Constraint

Restricts the values an attribute can hold — enforced through data types and CHECK constraints.

CGPA DECIMAL(3,1) CHECK (CGPA >= 0.0 AND CGPA <= 10.0)

NOT NULL Constraint

Ensures an attribute cannot have a NULL value. Applied to columns where a value must always exist.

UNIQUE Constraint

Ensures all values in a column (or combination) are distinct. Unlike PRIMARY KEY, UNIQUE columns can contain NULL (and multiple NULLs are typically allowed — each NULL is considered distinct).

PRIMARY KEY Constraint

Combination of UNIQUE + NOT NULL. Exactly one per table. Uniquely identifies each row.

FOREIGN KEY Constraint (Referential Integrity)

Ensures that a value in a column references a valid value in another table's primary key. When a referenced row is deleted or updated, the DBMS handles it according to the defined action:

ActionOn DELETE / UPDATE of Referenced Row
RESTRICTPrevent the delete/update if any referencing rows exist.
CASCADEAutomatically delete/update all referencing rows.
SET NULLSet the FK column to NULL in all referencing rows.
SET DEFAULTSet the FK column to its default value.
NO ACTIONSimilar to RESTRICT but check is deferred to end of transaction.
CREATE TABLE Enrollment ( SID INT, CourseID VARCHAR(10), PRIMARY KEY (SID, CourseID), FOREIGN KEY (SID) REFERENCES Student(SID) ON DELETE CASCADE ON UPDATE RESTRICT );

Assertion

A general constraint not tied to a specific table. Checked against the entire database state. Rarely supported in practice but theoretically important.

CREATE ASSERTION max_courses CHECK (NOT EXISTS ( SELECT SID FROM Enrollment GROUP BY SID HAVING COUNT(*) > 6 )); -- No student can be enrolled in more than 6 courses

Chapter 08
GATE Traps & Quick Reference

8.1   The Critical GATE Traps for SQL & RA

#TrapThe Correct Understanding
1 RA vs SQL duplicate semantics RA projection always eliminates duplicates (set semantics). SQL SELECT without DISTINCT preserves duplicates (bag semantics). Two different behaviors on the same-looking operation.
2 NOT IN with NULLs returns no rows If the subquery in NOT IN returns any NULL, the result is empty. Use NOT EXISTS instead for safety.
3 AVG ignores NULLs in denominator AVG(col) = SUM(non-null values) / COUNT(non-null values). Not divided by total row count. COUNT(*) ≠ COUNT(col) when NULLs exist.
4 WHERE cannot use aggregates WHERE runs before GROUP BY. Aggregates are computed after. Use HAVING for conditions on aggregates.
5 SELECT alias not available in WHERE SELECT runs after WHERE in the logical execution order. An alias defined in SELECT doesn't exist when WHERE is evaluated.
6 GROUP BY rule violation Every non-aggregated attribute in SELECT must appear in GROUP BY. Forgetting this is a syntax error.
7 Natural join on coincidental column names Natural join joins on ALL common column names. If two unrelated attributes have the same name, the join is incorrect.
8 CROSS JOIN result size Result has m × n rows where m and n are the cardinalities of the two tables. Often used to test if you can calculate intermediate result sizes.
9 TRUNCATE vs DELETE vs DROP DROP removes schema. TRUNCATE empties the table (DDL, no rollback in most DBs). DELETE is DML — transactional and can be rolled back.
10 Division operator recognition "Find X associated with ALL Y" → think division. "Enrolled in all courses," "rated by every user" — these are division queries.
11 = ANY vs IN, <> ALL vs NOT IN These pairs are equivalent but NOT IN is dangerous with NULLs. ≠ ANY is NOT equivalent to NOT IN.
12 Outer join and NULL generation Left join preserves all left rows; missing right columns become NULL. Selecting only rows WHERE right.col IS NULL after a LEFT JOIN gives you rows with NO match — this is how you find "not in" without using NOT IN.

Division in SQL — The NOT EXISTS Pattern

Since SQL has no division operator, "find all X associated with every Y" is written as:

-- Find students enrolled in ALL courses that SID 101 is enrolled in SELECT DISTINCT S.SID FROM Student S WHERE NOT EXISTS ( SELECT CourseID FROM Enrollment WHERE SID = 101 EXCEPT SELECT CourseID FROM Enrollment WHERE SID = S.SID ); -- "There is no course that 101 is enrolled in that S is NOT enrolled in"

The double negation pattern: "there does not exist a Y such that X is NOT associated with Y" is the standard SQL translation of universal quantification.
