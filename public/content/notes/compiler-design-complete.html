Front End vs Back End

Front end: Phases that are source-language dependent — lexical, syntax, semantic analysis, IC generation. Output is language-independent intermediate code.
Back end: Phases that are target-machine dependent — optimization, code generation. Input is the intermediate code.
This separation allows one front end + multiple back ends (compile same language for multiple architectures) or multiple front ends + one back end (compile different languages for same architecture).

Chapter 02
Lexical Analysis — Tokens, Patterns & DFA

2.1   Tokens, Patterns & Lexemes Core

Three Key Terms

Token: A category of lexical units. Example: IDENTIFIER, NUMBER, PLUS, KEYWORD_IF.
Lexeme: The actual string of characters that matches a token pattern. Example: "count", "3.14", "+", "if".
Pattern: A rule (usually a regular expression) describing what strings are valid lexemes for a token. Example: pattern for IDENTIFIER = [a-zA-Z_][a-zA-Z0-9_]*

Source: count = count + 3.14;

Tokens and their lexemes: (IDENTIFIER, "count") (ASSIGN, "=") (IDENTIFIER, "count") (PLUS, "+") (FLOAT_LITERAL, "3.14") (SEMICOLON, ";")

Regular Expressions for Token Patterns

2.2   RE → NFA → DFA → Minimized DFA GATE Tested

A lexer (scanner) is built by converting token patterns (regular expressions) into a Deterministic Finite Automaton (DFA) using the standard pipeline: RE → NFA (Thompson's construction) → DFA (subset construction) → Minimal DFA (table filling). The resulting DFA is then used to scan the input character by character, grouping characters into tokens. This pipeline is covered in depth in the TOC File 1.

Longest Match Rule (Maximal Munch)

When multiple tokens could match at the current position, the lexer takes the longest match. Example: input "==" — don't tokenize as "=" then "="; tokenize as "==" (EQUALS_EQUALS). If a tie exists (two different tokens of equal length match), the token listed first in the specification wins (priority rule).

Lexer vs Parser Responsibility

The lexer handles regular language recognition (tokens) — things describable by regular expressions. It does NOT handle nested structures (parentheses, brackets) — those require a context-free grammar and a parser. "Is 'if' a keyword or an identifier?" is a lexer question. "Is this expression syntactically valid?" is a parser question.

Chapter 03
CFGs, Derivations & Ambiguity

3.1   CFG in Compiler Context — Eliminating Ambiguity GATE Critical

Programming language syntax is specified using CFGs. Unlike theoretical automata where ambiguity is sometimes tolerable, compilers require unambiguous grammars — each valid program must have exactly one parse tree, ensuring one unambiguous interpretation.

Ambiguous grammar for arithmetic expressions:

E → E + E | E * E | (E) | id The string "id + id * id" has two parse trees: Tree 1: (id + id) * id (wrong precedence) Tree 2: id + (id * id) (correct precedence) This grammar is ambiguous. Parsers cannot decide which tree to build.

Unambiguous grammar encoding precedence and associativity:

E → E + T | T (+ is left-associative, lowest precedence) T → T * F | F (* is left-associative, higher precedence) F → (E) | id (parentheses, base case) Now "id + id * id" has exactly one parse tree: E → E + T → T + T → F + T → id + T → id + T*F → id + F*F → id + id*id The tree forces * to bind tighter than + (precedence). Left-recursion (E→E+T) ensures left-associativity (a+b+c = (a+b)+c).
Eliminating Left Recursion

LL parsers cannot handle left-recursive grammars (they loop forever). Left recursion A → Aα | β must be converted:
A → βA'
A' → αA' | ε
For "E → E+T | T": E → TE', E' → +TE' | ε

Left Factoring

When two productions for the same nonterminal start with the same terminal, LL parsers cannot choose between them. Factor out the common prefix:
A → αβ | αγ becomes A → αA', A' → β | γ

The Dangling Else Problem

Grammar: stmt → if expr then stmt | if expr then stmt else stmt | other

Ambiguous: "if E1 then if E2 then S1 else S2" has two trees: (a) if E1 then (if E2 then S1 else S2) ← else belongs to inner if (b) if E1 then (if E2 then S1) else S2 ← else belongs to outer if Most languages use rule (a) — "else matches closest if." This is the classic dangling-else ambiguity, resolved by convention in parsers.

Chapter 04
Top-Down Parsing — LL(1) & Predictive Parsing

4.1   FIRST and FOLLOW Sets Most Tested

FIRST(α)

The set of terminals that can begin strings derived from α.
If α ⟹* ε (derives empty), then ε ∈ FIRST(α).
Rules:
FIRST(a) = {a} for terminal a.
FIRST(ε) = {ε}.
FIRST(A) for nonterminal A: for each production A → X₁X₂...Xₙ, add FIRST(X₁)−{ε}. If ε ∈ FIRST(X₁), add FIRST(X₂)−{ε}, and so on. If ε ∈ FIRST(Xᵢ) for all i, add ε to FIRST(A).

FOLLOW(A)

The set of terminals that can appear immediately after A in a sentential form. $ (end-of-input) is in FOLLOW(S) (start symbol).
Rules:
If B → αAβ: add FIRST(β)−{ε} to FOLLOW(A).
If B → αAβ and ε ∈ FIRST(β): add FOLLOW(B) to FOLLOW(A).
If B → αA: add FOLLOW(B) to FOLLOW(A).

Compute FIRST and FOLLOW for the expression grammar:

Grammar: E → T E' E' → + T E' | ε T → F T' T' → * F T' | ε F → ( E ) | id FIRST: FIRST(E) = FIRST(T) = FIRST(F) = { (, id } FIRST(E') = { +, ε } FIRST(T') = { *, ε } FIRST(F) = { (, id } FOLLOW: FOLLOW(E) = { $, ) } (start symbol gets $; E appears before ) in F→(E)) FOLLOW(E') = FOLLOW(E) = { $, ) } (E' at end of E → T E') FOLLOW(T) = FIRST(E')−{ε} ∪ FOLLOW(E') = { +, $, ) } FOLLOW(T') = FOLLOW(T) = { +, $, ) } FOLLOW(F) = FIRST(T')−{ε} ∪ FOLLOW(T') = { *, +, $, ) }

4.2   LL(1) Parsing Table Construction GATE Critical

LL(1) Parsing Table Construction Rule

For each production A → α:
1. For each terminal a ∈ FIRST(α) − {ε}: add A → α to M[A, a].
2. If ε ∈ FIRST(α): for each b ∈ FOLLOW(A): add A → α to M[A, b].
If any cell gets more than one production → the grammar is NOT LL(1).

LL(1) table for the expression grammar:

| id | + | * | ( | ) | $ | ──────┼──────┼─────┼─────┼─────┼─────┼─────┤ E | E→TE'| | |E→TE'| | | E' | |E'→+TE'| | |E'→ε |E'→ε | T | T→FT'| | |T→FT'| | | T' | |T'→ε |T'→*FT'| |T'→ε |T'→ε | F | F→id | | |F→(E)| | | All cells have at most one entry → grammar IS LL(1). ✓
LL(1) Parsing — Stack Algorithm

Initialize stack with [S, $] (top = S). Input pointer at first token.
While stack not empty:
  If top = terminal a = current input: pop stack, advance input.
  If top = terminal a ≠ current input: ERROR.
  If top = nonterminal A: look up M[A, current_input] = A → α. Pop A, push α reversed (so leftmost symbol is on top).
  If M[A, current_input] = error: ERROR.
Accept when stack = [$] and input = $.

Parse "id + id * id" using LL(1) table:

Stack Input Action [E, $] id+id*id$ M[E,id]=E→TE'. Push T,E' [T, E', $] id+id*id$ M[T,id]=T→FT'. Push F,T' [F, T', E', $] id+id*id$ M[F,id]=F→id. Push id [id, T', E', $] id+id*id$ Match id. Pop, advance. [T', E', $] +id*id$ M[T',+]=T'→ε. Pop T'. [E', $] +id*id$ M[E',+]=E'→+TE'. Push +,T,E' [+, T, E', E', $] +id*id$ Match +. Pop, advance. [T, E', E', $] id*id$ M[T,id]=T→FT'. Push F,T' [F, T', E', E', $] id*id$ M[F,id]=F→id. Push id. [id,T',E',E',$] id*id$ Match id. Pop, advance. [T', E', E', $] *id$ M[T',*]=T'→*FT'. Push *,F,T' ... [E', $] $ M[E',$]=E'→ε. Pop. [$] $ ACCEPT ✓
A Grammar is LL(1) iff No Table Cell Has Multiple Entries

A grammar is LL(1) if and only if no two productions of the same nonterminal have overlapping FIRST sets (or both derive ε). Left-recursive grammars are never LL(1). Ambiguous grammars are never LL(1). Every LL(1) grammar is unambiguous, but not every unambiguous grammar is LL(1).

Chapter 05
Bottom-Up Parsing — SLR, CLR & LALR

5.1   LR Parsing — Concept & Handle Detection GATE Critical

LR Parsing Idea

LR (Left-to-right, Rightmost derivation) parsers build the parse tree from leaves to root. They maintain a stack of grammar symbols and states, and make two decisions at each step:
Shift: Push the next input token onto the stack.
Reduce: Replace the top of the stack (matching a production's RHS) with the LHS nonterminal.
The parser reads tokens left-to-right and produces a rightmost derivation in reverse.

Handle

A handle is a substring of the stack that (a) matches the RHS of some production A → α and (b) reducing it is the correct next step in the rightmost derivation. The key challenge in bottom-up parsing is efficiently identifying the handle — the LR automaton does this using states.

LR(0) Items

LR(0) Item

An LR(0) item is a production with a dot (•) indicating how much has been seen. A → α•β means "we've seen α on the stack and expect to see β from the input."
A → •αβ: haven't started yet.
A → α•β: seen α, expect β.
A → αβ•: seen the complete RHS — this is a complete item — potential reduce action.

LR(0) items for production E → E + T:

E → • E + T (initial — expect to see E + T) E → E • + T (have seen E, expect +, then T) E → E + • T (have seen E+, expect T) E → E + T • (complete — REDUCE using E → E+T)

Power Hierarchy

Every LL(1) grammar is also LR(1). Not every LR(1) grammar is LL(1). The hierarchy:
LR(0) ⊂ SLR(1) ⊂ LALR(1) ⊂ LR(1) = CLR(1)
All these are proper subsets of unambiguous CFLs.
LALR(1) ≡ LR(1) for most practical programming languages. LALR(1) is what yacc/bison implement.

SLR Conflicts — Shift-Reduce and Reduce-Reduce

Shift-Reduce conflict: In a state, for some lookahead token a, both SHIFT a and REDUCE (A→α, a∈FOLLOW(A)) are valid actions. Ambiguity in what to do.
Reduce-Reduce conflict: In a state, two different complete items A→α• and B→β• both want to reduce on the same lookahead. Which reduction to take?
If a grammar produces conflicts in SLR but not in LALR(1): the grammar is LALR(1) but not SLR(1). If LALR(1) also has conflicts: try CLR(1). If CLR(1) has conflicts: the grammar is ambiguous or not LR(1).

SLR parsing table construction steps:

1. Augment grammar: add S' → S to make unique start. 2. Compute LR(0) item sets (states) using CLOSURE and GOTO: CLOSURE(I): If A→α•Bβ in I, add B→•γ for all B→γ. GOTO(I, X): Items in I with • before X, advance •. 3. Build ACTION table: If A→α•aβ in state i (shift on a): ACTION[i,a] = SHIFT(GOTO(i,a)) If A→α• in state i (reduce): ACTION[i,b] = REDUCE(A→α) for all b∈FOLLOW(A) If S'→S• in state i: ACTION[i,$] = ACCEPT 4. Build GOTO table: GOTO[i,A] = GOTO(state_i, A) for nonterminals A. 5. Any undefined cells = ERROR.
yacc/bison Conflict Resolution Defaults

yacc/bison use defaults to resolve conflicts: Shift-Reduce conflict → prefer SHIFT (this correctly handles the dangling else). Reduce-Reduce conflict → prefer the production listed first in the grammar. These defaults make yacc accept slightly more than pure LALR(1), but the extra programs may be unintended — always eliminate conflicts explicitly.

Chapter 06
Semantic Analysis & Syntax-Directed Translation

6.1   What Semantic Analysis Does Core

Syntactically valid programs can still be semantically wrong. "int x = 'a' + true;" may be syntactically correct but semantically problematic in a strongly-typed language. Semantic analysis enforces language-specific rules that context-free grammars cannot express.

  • Type checking: Operands have compatible types. Functions called with correct argument types. Return types match declarations.
  • Scope resolution: Every identifier used is declared in the appropriate scope before use.
  • Undeclared variable detection: Using a variable not in the current symbol table = error.
  • Array bounds / index type: Array index must be an integer type.
  • Function call matching: Correct number and types of arguments.
  • Label existence: GOTO targets a label that exists.
Symbol Table

A data structure mapping identifiers to their attributes (type, scope, size, address, number of parameters for functions). Built during semantic analysis. Implemented as a hash table with scope chaining — each scope has its own table linked to its enclosing scope. Lookup traverses from innermost to outermost scope.

6.2   Syntax-Directed Translation (SDT) GATE Tested

Syntax-Directed Definition (SDD)

Each grammar production has associated semantic rules (attribute computations). Attributes are values associated with grammar symbols — each symbol at each node in the parse tree has values for its attributes.
Synthesized attribute: Value computed from children's attributes (bottom-up). Computed on the way up the tree.
Inherited attribute: Value passed from parent or siblings (top-down). Computed on the way down the tree.

SDD for evaluating arithmetic expressions (synthesized attributes):

Production Semantic Rule E → E₁ + T E.val = E₁.val + T.val E → T E.val = T.val T → T₁ * F T.val = T₁.val * F.val T → F T.val = F.val F → ( E ) F.val = E.val F → digit F.val = digit.lexval For "3 + 4 * 5": F.val=3, T.val=3, F.val=4, F.val=5, T.val=20, E.val=23 ✓
S-attributed vs L-attributed SDDs

S-attributed: Uses only synthesized attributes. Evaluation is bottom-up — can be integrated directly into LR parsers. Simplest case.
L-attributed: Uses synthesized and inherited attributes, but inherited attributes of a symbol depend only on: (a) attributes of the parent, (b) attributes of left siblings. Can be evaluated in a left-to-right traversal of the parse tree. Compatible with LL parsers and top-down passes.

Intermediate Code & Optimization

7.1   Three-Address Code & DAG GATE Tested

Three-Address Code (TAC)

The most common intermediate representation. Each instruction has at most one operator and at most three addresses (operands/result). General form: result = operand₁ op operand₂. Temporary variables (t₁, t₂, ...) hold intermediate results. Maps closely to machine instructions.

Generate TAC for: a = b * -c + b * -c

Naïve TAC: t1 = -c t2 = b * t1 t3 = -c t4 = b * t3 t5 = t2 + t4 a = t5 After CSE (Common Subexpression Elimination): t1 = -c t2 = b * t1 t3 = t2 + t2 ← reuse t2 instead of recomputing a = t3

DAG (Directed Acyclic Graph)

DAG for Expressions

A DAG compactly represents expressions by sharing nodes for common subexpressions. Each unique subexpression appears exactly once as a node. Leaves are identifiers/constants. Internal nodes are operators. If an expression is computed twice, both references point to the same node — this is automatic common subexpression elimination.

DAG for a = b * -c + b * -c:

+ / \ / \ ← same node used twice (CSE) * * ← both * nodes are identical → merged into one / \ / \ b uminus b uminus ← both uminus point to same c | | c c ← shared leaf After merging shared nodes: a │ + / \ / \ * (same) ← single * node, used twice in + / \ b uminus │ c

Basic Block & Control Flow Graph (CFG)

A basic block is a maximal sequence of statements with: one entry point (first statement, or target of a jump) and one exit point (last statement, or jump/return). No branches in the middle. Optimizations within a basic block are local optimizations. Optimizations across multiple basic blocks are global optimizations. The Control Flow Graph (CFG) has basic blocks as nodes and edges for possible control flow between them.

Register Allocation — Graph Coloring

Variables that are live simultaneously cannot share a register. Build an interference graph: nodes = variables, edges between variables that are simultaneously live. Register allocation = graph coloring with k colors (k = number of registers). If k-coloring is impossible: spill some variables to memory. NP-complete in general, but linear-time heuristics work well in practice.

Parameter Passing

Call by value: Copy of actual parameter passed. Callee cannot modify caller's variable. C default.
Call by reference: Address of actual parameter passed. Callee can modify caller's variable. C++ references, C with pointers.

8.1   All the Traps in One Place

#TrapCorrect Understanding
1LL(1) implies LR(1)Every LL(1) grammar IS also LR(1) (and LALR(1)). LR parsers are strictly more powerful. An LR(1) grammar may NOT be LL(1) (e.g., left-recursive grammars are LR(1) but NOT LL(1)). LL(1) ⊂ LALR(1) ⊂ LR(1).
2Left-recursive grammars cannot be parsedLeft-recursive grammars CANNOT be parsed by LL parsers (infinite loop). But they CAN be parsed by LR parsers — LR handles left recursion naturally. LR is more powerful than LL precisely because of this.
3FIRST and FOLLOW are computed the same wayFIRST(α) = terminals that can BEGIN strings derived from α. FOLLOW(A) = terminals that can FOLLOW A in any sentential form. FIRST is defined for strings of symbols. FOLLOW is defined only for nonterminals. Never mix them up in LL(1) table construction.
4ε should be in FOLLOW setsε is NEVER in a FOLLOW set. FOLLOW contains only terminals (including $). ε belongs in FIRST sets (when a symbol can derive the empty string). This distinction is critical for LL(1) table construction — ε in FIRST triggers FOLLOW-based table entries.
5Shift-Reduce conflict always means grammar is ambiguousA grammar causing SR conflict in SLR does NOT necessarily mean the grammar is ambiguous. The grammar might be LALR(1) or LR(1) — SLR just uses coarser lookaheads (FOLLOW sets) which cause false conflicts. Ambiguity causes conflicts in ALL LR parsers including LR(1).
6Synthesized attributes flow top-downSynthesized attributes flow BOTTOM-UP (computed from children). Inherited attributes flow TOP-DOWN (passed from parent). S-attributed SDDs use only synthesized — computed during LR reduction (bottom-up parse). L-attributed SDDs may use inherited — evaluated left-to-right in the tree.
7Semantic analysis handles type checking and scoping — these are syntacticType errors and scope errors are SEMANTIC, not syntactic. CFGs cannot express "variable must be declared before use" or "operand types must match" — these require semantic analysis using the symbol table. If GATE asks "which phase catches type errors?" → semantic analysis.
8Dead code elimination removes unreachable codeDead code elimination removes code whose RESULTS are never used (variables computed but never read). Unreachable code elimination (code after return, else branches never taken) is a separate optimization. GATE distinguishes these: "variable x is assigned but never read" → dead code. "Code after return statement" → unreachable code.
9LALR(1) and CLR(1) have different parsing powerLALR(1) and CLR(1)/LR(1) recognize the same class of languages — they differ in whether LALR(1) can be built without reduce-reduce conflicts. If LALR(1) produces a reduce-reduce conflict, the grammar is still LR(1) and CLR(1) can handle it without conflicts. They differ in table size, not expressive power in theory.
10Three-address code uses three addresses alwaysThree-address code uses AT MOST three addresses. Unary operations (t1 = -a) use two. Copy instructions (a = b) use two. Jump instructions (goto L) use one. The "three" is a maximum, not a requirement for every instruction.
11Compiler back end is language-dependentThe compiler BACK END (optimization, code generation) is machine/target-dependent, NOT source language-dependent. The front end is language-dependent. Intermediate code provides the clean separation — same IR can be processed by different back ends for different target machines.
12Loop-invariant code motion is always safeMoving loop-invariant code out of a loop is safe only if the code ALWAYS executes in the loop (it's in the loop body, not inside a conditional that might not execute). If the invariant code is inside an if-statement in the loop, moving it out changes semantics (it might now execute even when it shouldn't).

8.3   GATE Question Patterns

  • FIRST/FOLLOW computation NAT: "Compute FOLLOW(A) for grammar G." Trace all occurrences of A in productions, apply rules systematically. Very common 2-mark question.
  • LL(1) table cell MCQ: "Which production goes in M[E', +]?" Find production E'→+TE'. + ∈ FIRST(+TE')={+}. So M[E',+] = E'→+TE'. 1 mark.
  • Grammar classification MCQ: "Is this grammar LL(1)? SLR(1)? LALR(1)?" Check for conflicts. Left-recursive → not LL(1). Build SLR table → check for conflicts. Most common 2-mark classification question.
  • Parser power MCQ: "Which is more powerful: LL(1) or LALR(1)?" LALR(1). "Which handles left recursion?" LR parsers (SLR/LALR/CLR). 1 mark.
  • SDT evaluation: "Given grammar with semantic rules, what is the synthesized attribute value at the root for input X?" Trace the parse, evaluate attributes bottom-up. 2-mark computation.
  • TAC generation: "Generate three-address code for expression E." Apply standard translation scheme — introduce temporaries, one operation per instruction. 2 marks.
  • Optimization MCQ: "Which optimization replaces 2*x with x+x?" Strength reduction. "Which eliminates x=5 if x is never read?" Dead code elimination. 1-mark classification.
  • Phase identification MCQ: "Which phase detects undefined variables?" Semantic analysis. "Which phase produces tokens?" Lexical analysis. "Which phase detects mismatched parentheses?" Syntax analysis. 1 mark each.
