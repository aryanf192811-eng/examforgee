CPU Datapath & Control Unit

1.1   CPU Architecture — Core Components Core

The CPU is responsible for fetching, decoding, and executing instructions. Its core components form the datapath and control path.

Simplified CPU Datapath ┌──────────────────────────────────────────────────────────┐ │ CPU │ │ │ │ ┌──────┐ instruction ┌──────────┐ control signals │ │ │ PC │──────────────→│ Control │──────────────────→ │ │ └──┬───┘ │ Unit │ │ │ │ address └──────────┘ │ │ ↓ │ │ ┌──────────────┐ data/instr ┌─────────┐ │ │ │ Instruction │──────────────→│Register │ │ │ │ Memory │ │ File │←── WB │ │ └──────────────┘ └────┬────┘ │ │ │ operands │ │ ┌──────────────┐ result ┌────↓────┐ │ │ │ Data │←──────────────│ ALU │ │ │ │ Memory │──────────────→│ │ │ │ └──────────────┘ └─────────┘ │ └──────────────────────────────────────────────────────────┘ Key registers: PC (Program Counter): address of next instruction IR (Instruction Reg): currently executing instruction MAR (Memory Address Reg): address being accessed in memory MDR (Memory Data Reg): data read from / to be written to memory ACC (Accumulator): ALU result in accumulator-based ISAs
Instruction Cycle — Fetch-Decode-Execute

Fetch: PC → MAR. Read instruction from memory → MDR → IR. PC ← PC + instruction_size.
Decode: Control unit interprets opcode in IR. Determines operation, operand locations, addressing mode.
Execute: ALU performs operation. Result written to register or memory. PC updated (for branches).

Microprogramming

Each machine instruction is implemented as a sequence of microinstructions stored in control memory. The microprogram counter (μPC) steps through the microinstructions for each machine instruction. Each microinstruction generates the exact combination of control signals for one clock cycle. Allows complex instructions to be implemented without complex combinational logic.

RISC Philosophy — Simplicity Enables Speed

RISC designers recognized that compilers rarely use complex CISC instructions, so hardware complexity was wasted. By making every instruction simple (one clock cycle, fixed format, load/store only), every instruction could be pipelined efficiently. The "missing" complex operations are synthesized by the compiler as sequences of simple instructions — and the simple instructions execute so fast that the overall performance is better. Modern x86 chips actually translate CISC instructions to RISC-like micro-ops internally.

2.2   Instruction Format & Encoding GATE Tested

How many instructions can a 32-bit ISA encode if opcode is 6 bits?

2⁶ = 64 distinct opcodes (operation types). Each opcode then uses remaining bits for operands. RISC typically uses 6-bit opcode + addressing in remaining 26 bits.

Chapter 04
CPU Performance & Amdahl's Law

4.1   Performance Metrics & Formulas GATE Critical

Core CPU Performance Equation

CPU Time = IC × CPI × T_clock

IC = Instruction Count (number of instructions executed)
CPI = Cycles Per Instruction (average)
T_clock = Clock cycle time = 1 / Clock frequency

CPU Time = IC × CPI / f
where f = clock frequency in Hz.

MIPS — Millions of Instructions Per Second

MIPS = f / (CPI × 10⁶) = IC / (CPU_Time × 10⁶)
MIPS is misleading as a performance metric — different ISAs have different instruction counts for the same task. A faster MIPS machine might be slower for real programs if it needs more instructions. GATE asks for MIPS calculations but also tests this conceptual limitation.

CPU performance calculation:

Program runs on 2 GHz CPU. IC = 10⁹ instructions. CPI = 2. CPU Time = IC × CPI × T_clock = 10⁹ × 2 × (1/2×10⁹) = 10⁹ × 2 / (2×10⁹) = 1 second MIPS = 2×10⁹ / (2 × 10⁶) = 1000 MIPS If we upgrade to 4 GHz same CPU (same CPI=2): CPU Time = 10⁹ × 2 / (4×10⁹) = 0.5 seconds (2× speedup) If instead we optimize compiler to reduce CPI to 1.5 (same 2 GHz): CPU Time = 10⁹ × 1.5 / (2×10⁹) = 0.75 seconds (1.33× speedup)

Weighted Average CPI

Mixed-Instruction CPI

If a program uses different instruction types with different CPIs and frequencies:
CPI_avg = Σ (fraction_i × CPI_i)
where fraction_i = fraction of instructions of type i.

Program: 40% ALU (CPI=1), 30% Load (CPI=2), 20% Store (CPI=2), 10% Branch (CPI=3):

CPI_avg = 0.40×1 + 0.30×2 + 0.20×2 + 0.10×3 = 0.40 + 0.60 + 0.40 + 0.30 = 1.70 On 1 GHz processor running 10⁶ instructions: CPU Time = 10⁶ × 1.70 / 10⁹ = 1.7 ms

4.2   Amdahl's Law GATE Tested

Amdahl's Law

If a fraction f of a program can be sped up by a factor S, the overall speedup is:
Speedup = 1 / ((1−f) + f/S)
As S → ∞: Maximum speedup = 1/(1−f).
The non-parallelizable portion (1−f) limits overall improvement — diminishing returns as parallelism increases.

80% of a program can be parallelized. Maximum speedup with infinite cores?

f = 0.80, S = ∞ Max speedup = 1 / (1 - 0.80) = 1 / 0.20 = 5× With 4 cores (S=4): Speedup = 1 / ((1-0.80) + 0.80/4) = 1 / (0.20 + 0.20) = 1/0.40 = 2.5× The sequential 20% creates a hard ceiling of 5× no matter how many cores. This is why single-thread performance still matters even for parallel workloads.
Gustafson's Law — The Counter-argument

Amdahl's law fixes the problem size — the sequential fraction stays constant. Gustafson argued that with more processors, people typically scale up the problem size, making the parallel fraction grow and the sequential fraction shrink proportionally. Gustafson's Law: Speedup = S + (1−S)×f (linear with processors for scalable workloads). GATE sticks to Amdahl's — but knowing the distinction shows depth.

Chapter 05
Pipelining — Stages, Hazards & Forwarding

5.1   Pipeline Concept & Performance Most Tested

Pipelining overlaps the execution of multiple instructions — while one instruction is being executed, the next is being decoded, and the one after is being fetched. Like an assembly line: multiple items in different stages simultaneously.

Classic 5-Stage RISC Pipeline

IF — Instruction Fetch: PC → instruction memory, load instruction into IR, PC ← PC+4.
ID — Instruction Decode / Register Read: decode opcode, read source registers from register file.
EX — Execute: ALU performs operation (arithmetic, address calculation, branch condition).
MA — Memory Access: read or write data memory (load/store instructions only).
WB — Write Back: write ALU result or loaded data to destination register.

InstructionCC1CC2CC3CC4CC5CC6CC7
I1IFIDEXMAWB
I2IFIDEXMAWB
I3IFIDEXMAWB
Pipeline Performance Formulas

Without pipeline: Time = n × k × t
(n = instructions, k = stages, t = stage time)

With pipeline (ideal, no hazards):
Time = (k + n − 1) × t
(k cycles to fill pipeline, then one new instruction completes per cycle)

Speedup (ideal): n×k / (k+n−1)
As n → ∞: Speedup → k (stage count) — throughput improvement, not latency.

Throughput = 1/t (one instruction per clock cycle in steady state)
Latency (per instruction) = k×t (no improvement — same stages to traverse)

5-stage pipeline, stage time = 2ns, execute 100 instructions:

Without pipeline: 100 × 5 × 2 = 1000 ns With pipeline: (5 + 100 - 1) × 2 = 104 × 2 = 208 ns Speedup: 1000/208 ≈ 4.81× (approaches 5× for large n) Single instruction latency: 5 × 2 = 10 ns (same as without pipeline) Pipeline improves THROUGHPUT, not latency of individual instructions.
Pipeline Improves Throughput, Not Latency

A k-stage pipeline takes k cycles to complete ONE instruction — exactly the same as without a pipeline. The benefit is THROUGHPUT: after filling, one instruction completes every cycle instead of every k cycles. "Pipeline makes instructions faster" — false. "Pipeline makes the processor handle more instructions per second" — true. GATE tests this distinction directly.

5.2   Pipeline Hazards — Three Types GATE Critical

Structural Hazard

Two instructions need the same hardware resource at the same time. Example: a single memory used for both instruction fetch and data access — a load in MA and a fetch in IF conflict. Solution: separate instruction and data caches (Harvard architecture), or pipeline stall.

Data Hazard

An instruction needs data that a previous instruction hasn't produced yet. Three subtypes: RAW (Read After Write — the common one), WAR (Write After Read), WAW (Write After Write). RAW is the most critical.

Control Hazard

Branch outcome not known until EX stage — the pipeline has already fetched and started decoding instructions after the branch that may be wrong. Solution: branch prediction, branch delay slots, flushing.

Data Hazard — RAW in Detail

RAW hazard: I2 reads R1 before I1 writes R1:

I1: ADD R1, R2, R3 (writes R1 in WB stage — cycle 5) I2: SUB R4, R1, R5 (reads R1 in ID stage — cycle 3) I3: AND R6, R1, R7 (reads R1 in ID stage — cycle 4) Without forwarding: I2 reads R1 in CC3 (ID), but I1 writes R1 in CC5 (WB). I2 reads OLD value of R1. INCORRECT. Without forwarding, need 2 stall cycles between I1 and I2: I1: IF ID EX MA WB stall stall (bubbles/NOPs inserted) I2: IF ID EX MA WB

Forwarding (Data Bypassing)

Forwarding

Forward the ALU result directly from the EX/MA/WB pipeline register to the EX stage input — without waiting for WB to write and ID to read the register file. This eliminates most RAW stalls without requiring instruction stalls.

With forwarding:

I1: ADD R1, R2, R3 IF ID EX MA WB ↑ │ EX→EX forward: R1 result available after I1's EX │ Forward to I2's EX input I2: SUB R4, R1, R5 IF ID EX MA WB ↑ receives forwarded R1 ← no stall needed For LOAD followed immediately by use: I1: LOAD R1, 200(R2) ← R1 available after MA stage I2: ADD R4, R1, R5 ← needs R1 at start of EX stage Pipeline: I1: IF ID EX MA WB I2: IF ID ** EX MA WB ← ONE stall cycle needed (load-use hazard) Even with forwarding, load-use hazard requires 1 stall. MA result available → can forward to next EX, but 1 cycle gap required.
Load-Use Hazard — 1 Stall Even with Forwarding

A LOAD instruction followed immediately by an instruction that uses the loaded value always requires exactly 1 stall cycle — even with full forwarding. The loaded data is available after the MA stage (end of cycle 4), but the next instruction needs it at the start of its EX stage (cycle 4 for no stall). The 1-cycle gap cannot be eliminated without reordering. GATE tests this distinction: ALU→ALU hazards can be fully forwarded; LOAD→USE always needs 1 stall.

5.3   Hazard Counting — GATE NAT Method Most Tested

Count stall cycles for this sequence (5-stage pipeline, full forwarding):

I1: ADD R1, R2, R3 I2: LOAD R4, 0(R1) ← uses R1 (output of I1). ALU→ALU: forwarded, no stall. I3: SUB R5, R4, R6 ← uses R4 (output of I2 LOAD). Load-use hazard: 1 stall. I4: AND R7, R5, R8 ← uses R5 (output of I3 SUB). ALU→ALU: forwarded, no stall. I5: BEQ R7, R0, label ← uses R7 (output of I4 AND). ALU→ALU: forwarded, no stall. Branch itself: 1 stall (branch resolves in EX, fetched 2 wrong, but with predict-not-taken penalty = stalls if branch taken) Total data hazard stalls: 1 (between I2 and I3) Branch stall: depends on prediction strategy and taken/not-taken Execution with 1 data stall, no branch stall: Cycles = (5 + 5 - 1) + 1 = 10 cycles for 5 instructions (ideal: 9)
Systematic Hazard Counting

For each instruction pair (producer, consumer): (1) Identify if consumer reads a register written by producer. (2) If ALU→ALU: with forwarding, 0 stalls. (3) If LOAD→use: 1 stall regardless. (4) Count stalls from all pairs. Total cycles = (k + n − 1) + total_stalls. GATE NAT questions give an instruction sequence and ask for total execution cycles — apply this method systematically.

Chapter 06
Cache Memory — Mapping, Replacement & Performance

6.1   Cache Fundamentals & Address Decomposition GATE Critical

Cache

Cache is a small, fast memory between the CPU and main memory. It exploits temporal locality (recently accessed data will be accessed again soon) and spatial locality (nearby data will be accessed soon) to reduce average memory access time.

Cache Address Decomposition

A memory address is split into three fields depending on cache organization:
Tag | Set Index | Block Offset
Block offset bits = log₂(block size in bytes)
Set index bits = log₂(number of sets)
Tag bits = address bits − set index bits − block offset bits

Cache: 32-entry direct-mapped, block size = 16 bytes, 32-bit addresses.

Block offset bits: log₂(16) = 4 bits (byte within a 16-byte block) Set index bits: log₂(32) = 5 bits (which of the 32 cache lines) Tag bits: 32 − 5 − 4 = 23 bits Address decomposition: [Tag: 23 bits | Index: 5 bits | Offset: 4 bits] Cache size (data only): 32 lines × 16 bytes = 512 bytes Cache size (with tags): 32 × (1 valid bit + 23 tag bits + 16×8 data bits) = 32 × 152 bits

6.2   Cache Mapping Techniques Most Tested

k-way Set Associativen/k sets of k linesBlock → any of k lines in its setReducedModerate

Direct-mapped cache: 8 lines (0–7), block size = 1 word. Memory blocks 0–31.

Memory block i maps to cache line: i mod 8 Block 0 → line 0 Block 8 → line 0 (conflict with block 0!) Block 1 → line 1 Block 9 → line 1 ... Block 7 → line 7 Block 15 → line 7 Access pattern: block 0, block 8, block 0, block 8, ... Every access is a miss (thrashing — two blocks competing for same cache line). This worst-case pattern doesn't exist with 2-way set associative.

2-way set associative: 8 lines in 4 sets of 2 ways each.

Set index = address mod 4 (2 bits) Block 0 → set 0 (can go to way 0 or way 1) Block 4 → set 0 (same set as block 0) Block 8 → set 0 (same set, conflict with 0 and 4) With 2-way: blocks 0 and 4 can coexist (one per way). The thrashing pattern (block 0, 8 alternating) still conflicts (both map to set 0). Block 0 and block 4 can now coexist — improvement over direct-mapped.

6.3   Cache Performance — AMAT Formula Most Tested

AMAT — Average Memory Access Time

AMAT = Hit time + Miss rate × Miss penalty
Where:
Hit time: Time to access the cache (assuming hit) — typically 1–4 cycles.
Miss rate: Fraction of accesses that miss in cache.
Miss penalty: Additional time to fetch from next level — typically 50–200 cycles for main memory.

L1 cache: hit time = 2 cycles, miss rate = 5%, miss penalty = 100 cycles.

AMAT = 2 + 0.05 × 100 = 2 + 5 = 7 cycles Without cache (100 cycle memory): AMAT = 100 cycles. Cache gives 100/7 ≈ 14× improvement. With L2 cache (L1 miss goes to L2): AMAT = L1 hit time + L1 miss rate × (L2 hit time + L2 miss rate × Main memory time) = 2 + 0.05 × (10 + 0.20 × 100) = 2 + 0.05 × (10 + 20) = 2 + 0.05 × 30 = 2 + 1.5 = 3.5 cycles

SRAM vs DRAM — Know the Differences Cold

SRAM: fast (1–4 ns), expensive, doesn't need refresh, uses more area (6 transistors/bit). Used in caches. DRAM: slower (60–100 ns), cheap, needs periodic refresh (capacitor leaks charge), denser (1 transistor/bit). Used in main memory. GATE tests: "Which technology is used in cache?" → SRAM. "Which needs refresh?" → DRAM.

DMA Operation

CPU sets up DMA: source address, destination address, byte count, direction (read/write). DMA controller takes over the system bus and performs the transfer word-by-word or burst-by-burst. During burst DMA, CPU cannot access the bus (cycle stealing or burst mode). On completion, DMA interrupts the CPU. Used for disk transfers, network packets, audio buffers.

GATE Traps & Master Reference

8.1   All the Traps in One Place

#TrapCorrect Understanding
1Pipeline improves latencyPipeline improves THROUGHPUT (instructions per second), not latency of a single instruction. A 5-stage pipeline takes 5 cycles per instruction — same latency as non-pipelined. Speedup applies to streams of instructions.
2All data hazards require stallsALU→ALU RAW hazards are fully resolved by forwarding (0 stalls). LOAD→USE hazards require exactly 1 stall even with forwarding. WAR and WAW are rare and can also be resolved by forwarding/register renaming.
3More pipeline stages = always betterMore stages increase throughput but also increase: (1) branch misprediction penalty (more stages to flush), (2) hazard stall cycles (data must travel farther), (3) pipeline register overhead. Optimal stage count balances these tradeoffs.
4Cache size = number of lines × block sizeThis gives DATA size only. Total cache size includes tag bits and valid/dirty bits: Total bits = lines × (tag bits + valid bit + dirty bit + block_size × 8). GATE sometimes asks "total bits" — include metadata.
5Direct mapped: index bits = log₂(cache_size)Index bits = log₂(number of lines/sets), NOT log₂(cache_size). For direct-mapped: lines = cache_size / block_size. Set associative: sets = lines / associativity. Always compute number of sets first.
6Higher associativity always winsHigher associativity reduces conflict misses but increases hit time (more tags to compare) and cost. Fully associative has the lowest miss rate but the highest hardware complexity. Real caches use 2-way or 4-way as sweet spots.
7AMAT = hit rate × hit time + miss rate × miss penaltyAMAT = hit_time + miss_rate × miss_penalty. Hit time is ALWAYS paid (whether hit or miss). Miss penalty is ADDITIONAL time on a miss. Do NOT multiply hit time by hit rate.
8RISC has fewer instructions therefore runs fasterRISC has fewer instruction TYPES but often executes MORE instructions than CISC for the same task (no complex multi-step instructions). RISC wins because its simple instructions execute in 1 cycle and pipeline efficiently — not because it executes fewer instructions.
9DMA always uses interruptsDMA completion generates an interrupt. But during the transfer itself, DMA steals bus cycles from the CPU (cycle stealing) or takes the bus for burst periods. The CPU is NOT interrupted during transfer — only after completion.
10CPI = 1 always for pipelined processorsIdeal pipelined CPI = 1. With hazards: CPI > 1. Effective CPI = 1 + stalls_per_instruction. "A 5-stage pipeline with 10% load instructions and 1 load-use stall each: CPI = 1 + 0.10×1 = 1.10."
11Addressing modes: direct and immediate are the sameImmediate: operand value IS in the instruction (no memory access). Direct: operand ADDRESS is in the instruction (1 memory access to fetch operand). Completely different. "ADD R1, #5" (immediate, no memory). "ADD R1, 1000" (direct, 1 memory access to address 1000).
12SRAM is used for main memorySRAM is used for CACHE (fast, expensive). DRAM is used for main memory (slower, cheaper, needs refresh). Flash (a type of EEPROM) is used for SSDs and USB drives (non-volatile). These three are tested together very frequently.

8.3   GATE Question Patterns

  • CPI calculation NAT: "40% load (CPI=2), 30% ALU (CPI=1), 30% branch (CPI=3). Find average CPI." 0.4×2+0.3×1+0.3×3=0.8+0.3+0.9=2.0. Then CPU Time = IC×2/f. 2 marks.
  • Amdahl's Law NAT: "Program: 70% parallelizable. Speedup with 4 processors?" 1/(0.3+0.7/4)=1/(0.3+0.175)=1/0.475≈2.1×. 2 marks.
  • Pipeline cycles NAT: "5-stage pipeline, 100 instructions, 10 load-use hazards (1 stall each). Total cycles?" (5+100−1)+10=114. 2 marks.
  • Cache address decomposition NAT: "16KB direct-mapped cache, 32-byte blocks, 32-bit addresses. Number of tag bits?" Sets=16384/32=512. Index=log₂(512)=9. Offset=log₂(32)=5. Tag=32−9−5=18. 2 marks.
  • AMAT NAT: "L1 hit time=2, miss rate=10%, L2 hit time=8, L2 miss rate=20%, RAM=100 cycles." AMAT=2+0.1×(8+0.2×100)=2+0.1×28=2+2.8=4.8 cycles. 2 marks.
  • Addressing mode MCQ: "Instruction: LOAD R1, 4(R2). R2=1000. Effective address?" EA=1000+4=1004. "How many memory accesses?" 1 (read from address 1004). 1 mark.
  • Forwarding MCQ: "With full forwarding, which hazard still requires a stall?" Load-use hazard — 1 stall always required. 1 mark.
  • Cache miss types MCQ: "First access to a data element always causes which type of miss?" Compulsory (cold) miss. 1 mark.
  • RISC vs CISC MCQ: "Which ISA type uses load/store architecture?" RISC. "Which uses hardwired control?" RISC. "Which has fixed instruction length?" RISC. 1-mark each.
