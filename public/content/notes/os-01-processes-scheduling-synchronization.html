Chapter 01
Processes & Threads

1.1   Process — Definition & State Machine Core

A process is a program in execution — an active entity. A program is passive (a file on disk); a process is that program loaded into memory with a program counter, stack, heap, and data segment actively being executed. Multiple processes can run the same program simultaneously — they are distinct instances.

Process Control Block (PCB)

Each process is represented by a PCB (also called Task Control Block) containing: Process ID (PID), process state, program counter, CPU registers, memory management info (page tables, base/limit), I/O status info, and accounting info. The OS uses the PCB to save and restore process context during context switches.

Process State Diagram ┌──────────────────────────────────────────────────────┐ │ ADMITTED │ new ──→│── → ──── ready ◄────── running ──→ terminated │ │ │ scheduler ↑ ↓ │ │ │ dispatch │ I/O or event wait │ │ │ └── ← ── waiting │ │ └──────────────────── (I/O complete) │ └──────────────────────────────────────────────────────┘ States: New: process being created Ready: in memory, waiting for CPU Running: instructions being executed Waiting: waiting for I/O or event Terminated: finished execution
GATE Trap — Ready ≠ Running

A process in the ready state has all resources it needs EXCEPT the CPU. It is queued, waiting to be scheduled. A process in the running state is currently using the CPU. Only one process per CPU core can be running at any moment. The transition from ready→running is called dispatch (done by the scheduler). Running→ready happens on preemption or time quantum expiry.

1.2   Threads — Lightweight Processes GATE Tested

Thread

A thread is a lightweight unit of execution within a process. Multiple threads in the same process share: code segment, data segment, open files, and signal handlers. Each thread has its own: thread ID, program counter, register set, and stack.

AttributeShared across threadsPer-thread (private)
Code (text segment)Shared—
Global variables (data segment)Shared—
Heap memoryShared—
Open file descriptorsShared—
Stack—Private
Program counter—Private

User-Level Threads

Managed by a user-space library. Kernel unaware — sees one process. Fast context switch. If one thread blocks (I/O), all threads in the process block.

Kernel-Level Threads

Managed by the OS kernel directly. Each thread scheduled independently. One blocking thread doesn't block others. More overhead — kernel involvement for every context switch.

Context Switch Cost

Switching between threads within the same process is cheaper than switching between processes — there's no need to switch the memory address space (page tables, TLB flush). Process context switch = thread context switch + address space switch. This is the primary reason threads exist as a concurrency primitive.

Chapter 02
CPU Scheduling — All Algorithms

2.1   Scheduling Criteria & Terminology Core

TermDefinition
Arrival Time (AT)When the process enters the ready queue
Burst Time (BT)CPU time needed to complete the process
Completion Time (CT)When the process finishes execution
Turnaround Time (TAT)CT − AT   (total time from arrival to completion)
Waiting Time (WT)TAT − BT   (time spent waiting in ready queue)
Response Time (RT)Time from arrival to first CPU response (first scheduling)
Preemptive Scheduling

CPU can be taken away from a running process. A higher-priority or shorter process can preempt. Better response time, but higher overhead from context switches.

Non-Preemptive Scheduling

Once a process gets the CPU, it keeps it until it finishes or voluntarily releases (I/O wait). Simpler implementation. Lower overhead. May cause poor response for short jobs.

2.2   FCFS — First Come First Served GATE Tested

Non-preemptive. Processes served in order of arrival. Simple to implement. Suffers from the convoy effect — a long process can hold up many short ones.

2.3   SJF — Shortest Job First & SRTF GATE Critical

SJF selects the process with the smallest burst time from the ready queue. Non-preemptive version = SJF. Preemptive version = SRTF (Shortest Remaining Time First) — preempts if a new arrival has shorter remaining time.

SJF is Optimal for Average Waiting Time

SJF (non-preemptive) gives the minimum average waiting time among all non-preemptive algorithms for a given set of processes. SRTF gives the minimum average waiting time among ALL scheduling algorithms (preemptive or not). This is a proved theorem, not just an observation.

SJF (non-preemptive): P1(AT=0,BT=6), P2(AT=2,BT=8), P3(AT=4,BT=7), P4(AT=6,BT=3)

At t=0: Only P1 available → run P1 (BT=6) At t=6: Ready queue = {P2(8), P3(7), P4(3)} → pick P4 (shortest) At t=9: Ready queue = {P2(8), P3(7)} → pick P3 (shorter) At t=16: Only P2 remains → run P2 Gantt: [P1: 0-6][P4: 6-9][P3: 9-16][P2: 16-24] CT: P1=6, P2=24, P3=16, P4=9 TAT: P1=6, P2=22, P3=12, P4=3. Avg TAT = 43/4 = 10.75 WT: P1=0, P2=14, P3=5, P4=0. Avg WT = 19/4 = 4.75

SRTF (preemptive SJF): P1(AT=0,BT=8), P2(AT=1,BT=4), P3(AT=2,BT=9), P4(AT=3,BT=5)

SJF Requires Knowledge of Future — Impractical

SJF requires knowing the burst time in advance — which the OS cannot know precisely for general processes. In practice, burst time is estimated using exponential averaging: τₙ₊₁ = α·tₙ + (1−α)·τₙ where tₙ is the actual last burst and τₙ is the previous estimate. α controls how much weight recent history gets.

2.4   Priority Scheduling & Round Robin GATE Tested

Priority Scheduling

Each process has a priority. CPU goes to the highest-priority ready process. Can be preemptive (new high-priority process preempts) or non-preemptive. Problem: starvation — low-priority processes may wait indefinitely. Solution: aging — gradually increase priority of waiting processes.

Priority Convention — Check the Problem

Some problems define priority 1 = highest priority. Others define higher number = higher priority. GATE specifies the convention explicitly — always read it. Getting this backwards flips the entire answer.

Round Robin (RR)

Round Robin

Each process gets a fixed time quantum q. After q time units, the process is preempted and added to the back of the ready queue. Best for time-sharing systems. Fair — no starvation. Response time bounded by (n−1)×q where n is the number of processes.

RR with q=4: P1(AT=0,BT=24), P2(AT=0,BT=3), P3(AT=0,BT=3)

Queue order at t=0: P1, P2, P3 t=0-4: P1 runs 4 units (remaining=20). Preempted. Queue: P2,P3,P1 t=4-7: P2 runs 3 units (finishes). CT[P2]=7. Queue: P3,P1 t=7-10: P3 runs 3 units (finishes). CT[P3]=10. Queue: P1 t=10-14: P1 runs 4 units (remaining=16). Preempted. Queue: P1 ... (P1 gets quanta at 14-18, 18-22, 22-26, 26-30) t=26-30: P1 finishes. CT[P1]=30. TAT: P1=30, P2=7, P3=10. Avg TAT = 47/3 ≈ 15.67 WT: P1=6, P2=4, P3=7. Avg WT = 17/3 ≈ 5.67
Effect of Time Quantum on Round Robin

Very large quantum → behaves like FCFS. Very small quantum → low response time but high context switch overhead. Optimal quantum: large enough to minimize overhead, small enough to give good response. Rule of thumb: 80% of CPU bursts should be shorter than the quantum.

Multilevel Queue & Multilevel Feedback Queue

Multilevel Queue: Ready queue partitioned into multiple queues (foreground/interactive vs background/batch). Each queue has its own scheduling algorithm. Fixed assignment — processes don't move between queues.

Multilevel Feedback Queue (MLFQ): Processes can move between queues based on behavior. CPU-bound processes drop to lower-priority queues. I/O-bound processes stay in higher-priority queues. Most sophisticated and most flexible — approximates optimal scheduling. Used in modern Unix/Linux systems.

Chapter 04
Critical Section Problem — Requirements & Solutions

4.1   The Critical Section Problem GATE Critical

Critical Section

A critical section is a code segment that accesses shared resources (shared variables, files, hardware) and must not be executed by more than one process simultaneously. The problem: design a protocol so processes can enter their critical sections safely.

Three Requirements for a Correct CS Solution

1. Mutual Exclusion: At most one process can be in its critical section at any time.
2. Progress: If no process is in its CS and some processes want to enter, only those processes waiting to enter participate in the decision, and this selection cannot be postponed indefinitely.
3. Bounded Waiting: There must exist a bound on how many times other processes can enter their CS after a process has made a request to enter — before that request is granted.

Progress ≠ No Starvation (Bounded Waiting)

Progress and bounded waiting are different. Progress says the system makes forward progress — it doesn't deadlock (indefinite postponement at the group level is prevented). Bounded waiting says no individual process starves — it will eventually enter its CS. A solution can satisfy progress but violate bounded waiting (unfair to one process).

Peterson's Solution — Software Solution for 2 Processes

// Shared variables boolean flag[2] = {false, false} int turn // Process i (other process is j = 1-i) Entry Section: flag[i] = true // "I want to enter" turn = j // "But you go first if you want" while (flag[j] && turn == j) ; // Wait if j wants in AND it's j's turn // Critical Section Exit Section: flag[i] = false // "I'm done"

Peterson's solution satisfies all three requirements for 2 processes. However, it relies on memory accesses being atomic and sequential — modern CPUs with reordering may violate it without memory barriers.

4.2   Hardware Synchronization — TestAndSet & CompareAndSwap GATE Tested

TestAndSet — Atomic Hardware Instruction

TestAndSet(lock) atomically reads the value of lock and sets it to true, returning the old value. If the old value was false (lock was free), the caller acquired the lock. Implemented as a single indivisible CPU instruction.

// Atomic hardware instruction boolean TestAndSet(boolean *lock): old = *lock *lock = true return old // Using TestAndSet for mutual exclusion shared boolean lock = false Process i: while (TestAndSet(&lock)) ; // Spin until lock acquired // Critical Section lock = false // Release

This satisfies mutual exclusion (only one process can find lock=false) and progress (the while loop will terminate if the lock is free). However, it does NOT satisfy bounded waiting — a process can be perpetually bypassed by others who repeatedly acquire the lock.

CompareAndSwap (CAS)

CAS(ptr, expected, new_val) atomically: if *ptr == expected, set *ptr = new_val and return true. Otherwise return false. More powerful than TestAndSet — the foundation of lock-free data structures.

Chapter 05
Semaphores & Monitors

5.1   Semaphores — Definition & Types Most Tested

Semaphore

A semaphore S is an integer variable accessed only through two atomic operations:
wait(S) (also P, down): while S ≤ 0: block; S--
signal(S) (also V, up): S++; wake one waiting process if any

Binary Semaphore (Mutex)

S ∈ {0, 1}. Used for mutual exclusion. Initialize S=1. wait() before CS, signal() after CS. Exactly one process in CS at a time.

Counting Semaphore

S can be any non-negative integer. Represents a count of available resources. Initialize S = number of resource units. Used for resource management and synchronization.

Mutual exclusion using semaphore (initialized to 1):

Semaphore mutex = 1 Process P₁: Process P₂: wait(mutex) wait(mutex) // critical section // critical section signal(mutex) signal(mutex)

Synchronization: P₂ must execute AFTER P₁ completes statement S₁.

Semaphore sync = 0 // Initialize to 0, not 1! Process P₁: Process P₂: S₁ wait(sync) // Block until P₁ signals signal(sync) S₂ // Now safe to execute

Key: for synchronization (ordering), initialize semaphore to 0. For mutual exclusion, initialize to 1 (or the number of resources).

Busy Waiting vs Blocking

The simple wait(S) shown above uses busy waiting (spin lock) — the process loops and wastes CPU. OS implementations use blocking: when S ≤ 0, the process is put to sleep on a waiting queue and woken up by signal(). Busy waiting is acceptable in multi-core systems for very short critical sections (spinlocks in OS kernels) but unacceptable for user processes.

5.2   Monitors — High-Level Synchronization GATE Tested

Monitor

A monitor is a high-level synchronization construct — a module (class) with: shared variables, procedures that operate on them, and an implicit mutex lock. Only one process can execute inside the monitor at any time. The monitor enforces mutual exclusion automatically without explicit wait/signal calls on mutexes.

Monitors include condition variables with two operations:

  • condition.wait(): atomically releases the monitor lock and suspends the calling process. The process will be resumed by a signal.
  • condition.signal(): wakes one process waiting on the condition variable. If none waiting, the signal is lost (unlike semaphore where signal increments the counter).
Semaphore vs Monitor Condition Variable

A semaphore's signal() is never lost — it increments S even if no process is waiting. A monitor's condition.signal() IS lost if no process is waiting on that condition. This means the signaling code order matters for monitors but not semaphores. This is the classic exam distinction.

Chapter 06
Classical Synchronization Problems

6.1   Producer-Consumer (Bounded Buffer) GATE Tested

Producers add items to a shared buffer of size N. Consumers remove items. Constraints: producer must not add to a full buffer; consumer must not remove from an empty buffer; only one thread accesses the buffer at a time.

Semaphore mutex = 1 // Mutual exclusion on buffer Semaphore empty = N // Counts empty slots (init = N) Semaphore full = 0 // Counts filled slots (init = 0) Producer: Consumer: while(true): while(true): produce item wait(full) wait(empty) wait(mutex) wait(mutex) remove item from buffer add item to buffer signal(mutex) signal(mutex) signal(empty) signal(full) consume item
Order of wait() Matters — Deadlock Risk

In the Producer: wait(empty) MUST come before wait(mutex). If reversed — wait(mutex) then wait(empty) — a producer holding mutex could block on empty (buffer full), while a consumer needing mutex to remove an item is also blocked. Deadlock. The rule: always acquire the counting semaphore before the mutex semaphore.

6.2   Readers-Writers Problem GATE Tested

Multiple readers can read simultaneously. A writer needs exclusive access — no reader or other writer during writing.

Semaphore rw_mutex = 1 // Mutual exclusion for writers Semaphore mutex = 1 // Mutual exclusion on read_count int read_count = 0 // Number of active readers Reader: Writer: wait(mutex) wait(rw_mutex) read_count++ // Exclusive write access if read_count == 1: write data wait(rw_mutex) signal(rw_mutex) signal(mutex) read data wait(mutex) read_count-- if read_count == 0: signal(rw_mutex) signal(mutex)

This is the "first readers-writers" problem — readers have priority. If readers continuously arrive, writers can starve. The "second readers-writers" problem gives writers priority (once a writer is waiting, no new readers admitted).

6.3   Dining Philosophers Problem GATE Tested

5 philosophers sit at a round table. Between each pair: one fork (5 forks total). A philosopher needs BOTH adjacent forks to eat. The problem: design a protocol that avoids deadlock and starvation.

Naïve solution (deadlocks):

Semaphore fork[5] = {1,1,1,1,1} Philosopher i: wait(fork[i]) // Pick up left fork wait(fork[(i+1)%5]) // Pick up right fork eat() signal(fork[i]) signal(fork[(i+1)%5])

All 5 philosophers pick up their left fork simultaneously → circular wait → deadlock.

Solutions:

  • Asymmetric solution: One philosopher (say P4) picks up right fork first. Breaks circular wait.
  • Allow at most 4 philosophers: Use a semaphore initialized to 4. At most 4 can try simultaneously — at least one can always get both forks.
  • Both forks or neither: Use a mutex to pick up both forks atomically. Release both if can't get second.
  • Monitor-based: A philosopher only picks up forks if both are available (checked under monitor lock).

Chapter 07
Deadlocks — Detection, Prevention & Avoidance

7.1   Conditions for Deadlock — Coffman's Four GATE Critical

Coffman's Four Necessary Conditions

A deadlock can occur if and only if ALL four conditions hold simultaneously:
1. Mutual Exclusion: At least one resource is held in a non-sharable mode.
2. Hold and Wait: A process holds at least one resource and is waiting for additional resources held by other processes.
3. No Preemption: Resources cannot be forcibly taken away — only voluntarily released by the holding process.
4. Circular Wait: A circular chain of processes each waiting for a resource held by the next: P1 waits for P2's resource, P2 waits for P3's, ..., Pn waits for P1's.

Prevention — Negate One Condition

To prevent deadlocks, ensure at least one of the four conditions can never hold: (1) Allow sharing (weaken mutual exclusion — not always possible). (2) Require all resources at once (eliminate hold and wait — wastes resources). (3) Allow preemption (forcibly take resources — may cause inconsistency). (4) Order all resources and require acquisition in order (eliminate circular wait — practical and commonly used).

7.2   Resource Allocation Graph GATE Tested

Resource Allocation Graph (RAG)

A directed graph with two node types: processes (circles) and resources (rectangles with dots for instances). Request edge Pᵢ → Rⱼ: process Pᵢ is waiting for resource Rⱼ. Assignment edge Rⱼ → Pᵢ: an instance of Rⱼ is assigned to Pᵢ.

Graph ConditionDeadlock conclusion
No cycle in RAGNo deadlock
Cycle in RAG, each resource has exactly 1 instanceDeadlock definitely exists
Cycle in RAG, some resources have multiple instancesDeadlock MIGHT exist (need further analysis)
Cycle ≠ Deadlock (Multi-Instance Resources)

If a resource type has multiple instances, a cycle in the RAG does not guarantee deadlock. Example: R has 2 instances, P1 holds one and waits for the second, P2 holds one. There's a cycle (P1→R→P2→R→P1 in a sense) but P2 may finish and release its instance, allowing P1 to proceed. GATE directly tests this distinction.

7.3   Banker's Algorithm — Deadlock Avoidance Most Tested

Safe State

A system state is safe if there exists a safe sequence — an ordering P₁, P₂, ..., Pₙ of ALL processes such that for each Pᵢ, the resources Pᵢ needs can be satisfied by currently available resources PLUS resources held by all Pⱼ with j < i. If no safe sequence exists → unsafe state → deadlock possible.

Banker's Algorithm Data Structures

Available[j]: number of available instances of resource type j.
Max[i][j]: maximum demand of process i for resource j.
Allocation[i][j]: currently allocated to process i of resource j.
Need[i][j]: remaining need = Max[i][j] − Allocation[i][j].

Safety Algorithm (find safe sequence):

Work = Available (initially) Finish[i] = false for all i Repeat until no progress: Find i such that: Finish[i] == false AND Need[i] ≤ Work If found: Work = Work + Allocation[i] // i finishes, releases resources Finish[i] = true If all Finish[i] == true → SAFE STATE (sequence = order found) Else → UNSAFE STATE

Safe ≠ No Deadlock Currently, Unsafe ≠ Deadlock Certainly

Safe state means there EXISTS a sequence that avoids deadlock — not that the system will necessarily follow it. Unsafe state means deadlock is POSSIBLE if processes make their maximum requests simultaneously — not that deadlock has occurred or is inevitable. Banker's algorithm is conservative — it may reject requests that would be safe in practice.

7.4   Deadlock Detection & Recovery GATE Tested

Instead of preventing or avoiding deadlock, detect it after it occurs and recover. Detection uses a variant of the Banker's safety algorithm — run it periodically or when processes are waiting long.

StrategyMethodCostEffect
PreventionNegate one of Coffman's four conditionsMedium — restricts resource useDeadlock never occurs
AvoidanceBanker's algorithm — grant only safe requestsMedium — requires advance max infoDeadlock never occurs
Detection + RecoveryDetect cycles or run safety algorithm; kill or preempt processesHigh — deadlock occurs, then fixedDeadlock resolved after occurrence
Ignorance (Ostrich)Pretend deadlock can't happenNoneDeadlock may cause system restart — used in some real OS

Recovery Strategies

  • Process termination: Abort one or more deadlocked processes. Abort all (expensive) or abort one at a time until deadlock broken (cheaper but requires repeated detection).
  • Resource preemption: Preempt resources from some processes and give to others. Requires rollback capability — the preempted process must restart from a safe checkpoint.

8.3   GATE Question Patterns

  • Scheduling trace NAT: "Given 4 processes with arrival and burst times, find average waiting time under FCFS/SJF/RR(q=2)." Draw the Gantt chart, compute CT, TAT, WT for each, average them. Most common 2-mark question.
  • Comparison MCQ: "Which algorithm minimizes average waiting time?" SRTF for any (preemptive optimal). SJF for non-preemptive. 1 mark.
  • Deadlock conditions: "A system has 3 processes and 2 resource types. Is deadlock possible?" Apply Coffman's conditions — check if circular wait can occur. 1-2 marks.
  • Banker's Algorithm: "Given allocation/max tables, is the state safe? Find safe sequence." Apply safety algorithm step by step. 2-mark computation question.
  • RAG analysis: "Given a resource allocation graph, is there a deadlock?" Find cycles, check instance counts. 1-2 marks.
  • Semaphore initialization: "What initial value should semaphore S have?" For mutex=1, for synchronization=0, for n-resource pool=n. 1 mark.
  • CS requirements: "Does this solution satisfy mutual exclusion/progress/bounded waiting?" Analyze Peterson's or a given solution against the three properties. 2 marks.
  • Producer-Consumer order: "What happens if wait(mutex) is done before wait(empty) in the Producer?" Deadlock when buffer is full — producer holds mutex, consumer can't acquire it. 1-2 marks.
