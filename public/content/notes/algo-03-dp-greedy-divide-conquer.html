Chapter 01
Dynamic Programming — Foundations

1.1   The Core Idea — When and Why DP Works Core

Dynamic programming (DP) applies to optimization and counting problems that have two key properties. Without both, DP either doesn't apply or doesn't help.

Optimal Substructure

An optimal solution to the problem contains within it optimal solutions to subproblems. That is, you can build the global optimum by combining locally optimal subproblem solutions. This is what allows a DP recurrence to be correct.

Overlapping Subproblems

The recursive solution solves the same subproblems repeatedly. If subproblems were all distinct, simple divide-and-conquer would suffice. DP gains its power by storing and reusing subproblem solutions instead of recomputing them.

The naive recursive Fibonacci — T(n) = T(n−1)+T(n−2)+O(1) — solves F(n−2) repeatedly across different branches of the recursion tree. The number of calls is exponential (Θ(φⁿ)). DP stores each F(k) once, reducing time to Θ(n). This is the canonical example of both properties.

Top-Down (Memoization)

Write the recursive solution. Cache results in a table. On each call, check the cache first — if found, return immediately. Natural to write, uses only the subproblems that are actually needed.

Bottom-Up (Tabulation)

Fill the DP table iteratively from smallest subproblems to largest. No recursion overhead. Better cache performance. Requires knowing the subproblem dependency order.

1.2   The Four-Step DP Design Process GATE Framework

  1. 1
    Define the subproblem. What does dp[i] or dp[i][j] represent? Make it precise. "dp[i] = the length of the longest increasing subsequence ending at index i."
  2. 2
    Write the recurrence. How does dp[i] relate to smaller subproblems? Enumerate the choices at each step and express dp[i] in terms of previously computed values.
  3. 3
    Identify base cases. What are the smallest subproblems with known answers? These seed the table.
  4. 4
    Determine the fill order and extract the answer. Ensure when computing dp[i], all dp[j] it depends on are already computed. Identify which entry in the table is the final answer.

Classic DP Problems — LCS, LIS & 0/1 Knapsack

2.1   Longest Common Subsequence (LCS) GATE Critical

Problem Statement

Given two sequences X = x₁x₂...xₘ and Y = y₁y₂...yₙ, find the length of their longest common subsequence. A subsequence is obtained by deleting some (or zero) characters without changing order — it need not be contiguous.

Recurrence

Let dp[i][j] = length of LCS of X[1..i] and Y[1..j].
Base: dp[0][j] = 0, dp[i][0] = 0 (empty prefix has LCS 0).
If X[i] == Y[j]: dp[i][j] = dp[i−1][j−1] + 1
Else: dp[i][j] = max(dp[i−1][j], dp[i][j−1])

LCS of X = "ABCBDAB" and Y = "BDCABA"

LCS length = 4. One LCS = "BCBA". Time: Θ(mn). Space: Θ(mn), reducible to Θ(n) by keeping only two rows.

LCS → Edit Distance Connection

Edit distance (minimum insertions + deletions to convert X to Y) = (m − LCS) + (n − LCS) = m + n − 2·LCS. If substitutions are also allowed (cost 1), use a different DP. GATE sometimes asks for edit distance — reduce it to LCS.

2.2   Longest Increasing Subsequence (LIS) GATE Tested

Problem & Recurrence

Given sequence A[1..n], find the length of the longest strictly increasing subsequence.
Let dp[i] = length of LIS ending at index i.
dp[i] = 1 + max{dp[j] | j < i and A[j] < A[i]}, with dp[i]=1 if no such j exists.
Answer = max(dp[1], dp[2], ..., dp[n]).

A = [10, 9, 2, 5, 3, 7, 101, 18]

dp[1]=1 (10) dp[2]=1 (9 — nothing before 9 is smaller) dp[3]=1 (2 — nothing before 2 is smaller) dp[4]=2 (5 — 2<5, dp[3]+1=2) dp[5]=2 (3 — 2<3, dp[3]+1=2) dp[6]=3 (7 — 2<3<7... best: dp[5]+1=3 or dp[4]+1=3) dp[7]=4 (101 — 2<5<7<101, dp[6]+1=4) dp[8]=4 (18 — 2<5<7<18, dp[6]+1=4) Answer = max(dp) = 4. LIS = [2, 3, 7, 18] or [2, 5, 7, 18]

LIS in O(n log n) — Patience Sorting Idea

Maintain an array tails where tails[k] = smallest tail element of all increasing subsequences of length k+1 found so far. For each new element, binary search in tails for the first element ≥ current, replace it (or extend). Length of tails at the end = LIS length. GATE rarely tests the O(n log n) implementation but does test the O(n log n) complexity claim.

2.3   0/1 Knapsack Problem GATE Critical

Problem Statement

Given n items, each with weight wᵢ and value vᵢ, and a knapsack of capacity W. Select a subset of items to maximize total value such that total weight ≤ W. Each item is taken at most once (0/1 — binary choice).

Recurrence

dp[i][w] = maximum value using first i items with weight capacity exactly w.
Base: dp[0][w] = 0 (no items), dp[i][0] = 0 (zero capacity).
If wᵢ > w: dp[i][w] = dp[i−1][w] (can't include item i)
Else: dp[i][w] = max(dp[i−1][w], dp[i−1][w−wᵢ] + vᵢ)

Items: (w=2,v=6), (w=2,v=10), (w=3,v=12). Capacity W=5.

Itemw=0w=1w=2w=3w=4w=5
None000000
Item 1 (2,6)006666
Item 2 (2,10)0010101616
Item 3 (3,12)0010121622

Maximum value = 22 (take items 2 and 3: 10+12=22, weight 2+3=5 ≤ 5). Time: Θ(nW). Space: Θ(nW), reducible to Θ(W).

0/1 Knapsack is NP-Hard — But Pseudo-Polynomial

Θ(nW) looks polynomial but is actually pseudo-polynomial — it's polynomial in W (the numeric value) but exponential in the number of bits needed to represent W. If W = 2^n, the DP table has 2^n columns — exponential in input size. This is why 0/1 Knapsack is NP-hard despite having a DP solution. The Fractional Knapsack (greedy) is polynomial.

Chapter 03
Advanced DP — MCM, Coin Change & Edit Distance

3.1   Matrix Chain Multiplication (MCM) GATE Tested

Problem Statement

Given n matrices A₁, A₂, ..., Aₙ with dimensions p₀×p₁, p₁×p₂, ..., pₙ₋₁×pₙ, find the optimal parenthesization to minimize the number of scalar multiplications.

Multiplying a (p×q) matrix by a (q×r) matrix costs p·q·r scalar multiplications. The order of multiplication affects total cost drastically — associativity means we can choose any parenthesization.

Recurrence

m[i][j] = minimum cost to multiply matrices Aᵢ through Aⱼ.
Base: m[i][i] = 0 (single matrix, no multiplication needed).
m[i][j] = min over k from i to j−1 of: m[i][k] + m[k+1][j] + pᵢ₋₁ · pₖ · pⱼ
(split at position k: multiply Aᵢ..Aₖ and Aₖ₊₁..Aⱼ then combine)

A₁(10×30), A₂(30×5), A₃(5×60). Find optimal cost.

Dimensions: p = [10, 30, 5, 60] m[1][2] = p₀·p₁·p₂ = 10·30·5 = 1500 (A₁ × A₂) m[2][3] = p₁·p₂·p₃ = 30·5·60 = 9000 (A₂ × A₃) m[1][3] = min of: k=1: m[1][1] + m[2][3] + p₀·p₁·p₃ = 0 + 9000 + 10·30·60 = 9000 + 18000 = 27000 k=2: m[1][2] + m[3][3] + p₀·p₂·p₃ = 1500 + 0 + 10·5·60 = 1500 + 3000 = 4500 Optimal: 4500 — parenthesize as (A₁A₂)A₃ Compute A₁A₂ first (cost 1500), then multiply result (10×5) by A₃ (cost 10·5·60=3000)

Time: O(n³). Space: O(n²). Fill the table diagonally — length-2 subproblems first, then length-3, etc.

MCM Fill Order — Diagonal by Chain Length

The key to MCM bottom-up: fill m[i][j] for all chains of length l = j−i+1, starting from l=2. For l=2: fill m[1][2], m[2][3], ..., m[n−1][n]. For l=3: fill m[1][3], m[2][4], ..., m[n−2][n]. Continue until l=n. This ensures m[i][k] and m[k+1][j] are always computed before m[i][j].

3.2   Coin Change Problems GATE Tested

Minimum Coins (Unbounded)

Problem & Recurrence

Given coin denominations c₁,...,cₖ and target amount n. Find minimum coins to make n (each coin usable unlimited times).
dp[i] = minimum coins to make amount i.
dp[0] = 0. dp[i] = 1 + min{dp[i−cⱼ] | cⱼ ≤ i} for i ≥ 1.

Coins = {1,3,4}, target = 6:

dp[0]=0 dp[1]=1 (1) dp[2]=2 (1+1) dp[3]=1 (3) dp[4]=1 (4) dp[5]=2 (4+1) dp[6]=2 (3+3) ← not 4+1+1=3

Greedy (largest first) would give: 4+1+1 = 3 coins. DP gives 3+3 = 2 coins. Greedy fails on this input — this is why Coin Change requires DP for general denominations.

Number of Ways (Coin Change II)

Counting Combinations

Count distinct ways to make amount n using coins (order doesn't matter).
dp[i] = number of ways to make amount i.
For each coin c: for i from c to n: dp[i] += dp[i−c]
(iterate coins in outer loop to avoid counting permutations separately)

3.3   Edit Distance (Levenshtein Distance) GATE Tested

Problem & Recurrence

Minimum operations (insert, delete, substitute — each cost 1) to transform string X into Y.
dp[i][j] = edit distance between X[1..i] and Y[1..j].
Base: dp[0][j] = j (insert j characters), dp[i][0] = i (delete i characters).
If X[i]==Y[j]: dp[i][j] = dp[i−1][j−1]
Else: dp[i][j] = 1 + min(dp[i−1][j−1], dp[i−1][j], dp[i][j−1])
                       (substitute,    delete,   insert)

Edit distance between "kitten" and "sitting":

kitten → sitten (substitute k→s, cost 1) sitten → sittin (substitute e→i, cost 1) sittin → sitting (insert g at end, cost 1) Total: 3 operations. dp[6][7] = 3.

Time: O(mn). Space: O(mn) reducible to O(min(m,n)).

Chapter 04
Greedy Algorithms — Foundations & Proof of Correctness

4.1   The Greedy Paradigm Core

A greedy algorithm makes the locally optimal choice at each step, hoping the sequence of local optima leads to a global optimum. Unlike DP which considers all subproblem solutions, greedy commits to one choice and never reconsiders.

Greedy Choice Property

There always exists an optimal solution that includes the greedy choice. Making the greedy choice doesn't foreclose the path to an optimal solution.

Optimal Substructure

After making the greedy choice, the remaining subproblem's optimal solution combined with the greedy choice gives an optimal solution to the original.

Proving Greedy Correctness — Two Methods

Greedy stays ahead: Show that after each step, the greedy solution is at least as good as any other solution at that point. By induction, it's optimal overall.

Exchange argument: Take any optimal solution. Show that exchanging the first point of difference (where it disagrees with greedy) with the greedy choice gives an equally good or better solution. Hence greedy produces an optimal solution.

Greedy ≠ Always Optimal — The Critical Distinction

Greedy works for Fractional Knapsack but NOT 0/1 Knapsack. Works for Activity Selection but NOT Weighted Job Scheduling. The difference: problems where greedy is optimal have the greedy choice property provably. For others, greedy gives suboptimal results. GATE frequently tests: "does greedy give the optimal solution for problem X?" — always verify with a proof or counterexample.

Classic Greedy Problems — Activity Selection, Huffman & Knapsack

5.1   Activity Selection Problem GATE Tested

Problem

Given n activities, each with start time sᵢ and finish time fᵢ. Select the maximum number of non-overlapping activities. Activities i and j are compatible if their intervals don't overlap: fᵢ ≤ sⱼ or fⱼ ≤ sᵢ.

Greedy Algorithm

Sort activities by finish time. Always select the activity with the earliest finish time that is compatible with previously selected activities.

Activities: (1,4),(3,5),(0,6),(5,7),(3,9),(5,9),(6,10),(8,11),(8,12),(2,14),(12,16)

Sort by finish time: (1,4),(3,5),(0,6),(5,7),(3,9),(5,9),(6,10),(8,11),(8,12),(2,14),(12,16) Select (1,4) — first activity Skip (3,5) — 3 < 4, overlaps Skip (0,6) — 0 < 4, overlaps Select (5,7) — 5 ≥ 4, compatible ✓ Skip (3,9) — 3 < 7 Skip (5,9) — 5 < 7 Select (8,11) — 8 ≥ 7, compatible ✓ Skip (8,12) — 8 < 11 Select (12,16)— 12 ≥ 11, compatible ✓ Result: 4 activities selected. Optimal.
Why Earliest Finish Time is Optimal

Exchange argument: suppose an optimal solution selects activity A instead of B (the earliest-finish greedy choice) at some step. Since B finishes no later than A, replacing A with B in the optimal solution doesn't reduce the number of subsequent compatible activities. So the greedy choice gives at least as many activities as any optimal solution.

5.2   Huffman Coding GATE Tested

Problem

Given n characters with frequencies f₁, f₂, ..., fₙ. Assign binary codes to minimize the total encoding length (weighted path length from root to each character leaf in the code tree).

Huffman Algorithm

Build a min-heap of all character frequencies. Repeatedly extract the two minimum-frequency nodes, create a new internal node with frequency = sum of the two, reinsert. Repeat until one node remains (the root). Time: O(n log n).

Characters: a(5), b(9), c(12), d(13), e(16), f(45). Build Huffman tree.

Initial heap: [5(a), 9(b), 12(c), 13(d), 16(e), 45(f)] Step 1: Extract 5,9 → new node 14. Heap: [12,13,14,16,45] Step 2: Extract 12,13 → new node 25. Heap: [14,16,25,45] Step 3: Extract 14,16 → new node 30. Heap: [25,30,45] Step 4: Extract 25,30 → new node 55. Heap: [45,55] Step 5: Extract 45,55 → root 100. Codes (left=0, right=1): f:0 (1 bit) — most frequent, shortest code c:100, d:101 (3 bits each) a:1100, b:1101 (4 bits each) e:111 (3 bits — assigned to e) Total bits: 45×1 + 16×3 + 12×3 + 13×3 + 5×4 + 9×4 = 45 + 48 + 36 + 39 + 20 + 36 = 224 bits
Huffman Code Properties

Huffman codes are prefix-free (no code is a prefix of another — enables unambiguous decoding). Higher frequency → shorter code. Lower frequency → longer code. The Huffman tree is a full binary tree (every internal node has exactly 2 children). GATE often asks about the code length for a specific character, or the total encoding cost.

5.3   Fractional Knapsack & Job Scheduling GATE Tested

Fractional Knapsack

Algorithm

Items can be divided. Compute value-to-weight ratio vᵢ/wᵢ for each item. Sort by ratio (descending). Take items greedily from highest ratio to lowest, taking full items until capacity is exceeded, then take fraction of last item.

Items: (w=10,v=60), (w=20,v=100), (w=30,v=120). W=50.

Ratios: 6, 5, 4. Sort: item1(6), item2(5), item3(4).

Take all of item1 (10kg, 60 value). Take all of item2 (20kg, 100 value). Remaining: 20kg. Take 20/30 of item3 (80 value). Total: 60+100+80 = 240.

Job Scheduling to Minimize Lateness

Each job j has processing time tⱼ and deadline dⱼ. Schedule on one machine to minimize maximum lateness (Lmax = max of (completion time − deadline) over all jobs).

Greedy: Sort by deadline (earliest deadline first — EDF). This is optimal. Proof: any adjacent pair where a later-deadline job precedes an earlier-deadline job can be swapped without increasing Lmax.

Chapter 06
Divide & Conquer

6.1   The Paradigm & Classic Algorithms GATE Tested

Divide & Conquer Pattern

Divide: Split the problem into subproblems of roughly equal size.
Conquer: Solve each subproblem recursively (base case when small enough).
Combine: Merge subproblem solutions into the overall solution.

The key analysis tool: write the recurrence T(n) = aT(n/b) + f(n) and apply the Master Theorem (covered in File 1).

Merge Sort — O(n log n)

Divide array in half, sort each half recursively, merge in O(n). T(n) = 2T(n/2)+O(n) → Θ(n log n). Covered in detail in File 1.

Strassen's Matrix Multiplication — O(n^2.81)

Strassen's Algorithm

Standard n×n matrix multiplication: 8 recursive multiplications of n/2×n/2 matrices + O(n²) additions → T(n)=8T(n/2)+O(n²) → O(n³).
Strassen: 7 recursive multiplications using clever combinations → T(n)=7T(n/2)+O(n²) → O(n^log₂7) ≈ O(n^2.807).

The constant-factor improvement in recursive calls (7 vs 8) propagates multiplicatively across all recursion levels, giving a genuinely faster asymptotic algorithm.

Closest Pair of Points — O(n log n)

Algorithm Sketch

Divide points by median x-coordinate into left (L) and right (R) halves. Find closest pair in each half recursively (distances δL and δR). Let δ = min(δL, δR). Check the strip of width 2δ around the dividing line — at most 8 points in any δ×2δ rectangle in the strip, so each point is compared to at most 7 others. Combine in O(n).

T(n) = 2T(n/2) + O(n) → Θ(n log n) [dominated by sort + strip scan] Naïve: O(n²) (check all pairs). D&C gives Θ(n log n) — a genuine improvement.

Integer Multiplication — Karatsuba O(n^1.585)

Multiplying two n-digit numbers naïvely: O(n²). Karatsuba's algorithm splits each number into two halves and performs 3 recursive multiplications instead of 4: T(n) = 3T(n/2)+O(n) → O(n^log₂3) ≈ O(n^1.585).

The Simplest Test

Draw the recursion tree. If the same subproblem appears multiple times in the tree → overlapping subproblems → use DP. If every leaf in the recursion tree is a distinct subproblem → divide and conquer is fine, DP gives no benefit. Fibonacci: exponentially many repeated nodes → DP. NP-Completeness — P, NP & Reductions

7.1   P, NP, NP-Hard & NP-Complete GATE Critical

P — Polynomial Time

The class of decision problems solvable by a deterministic algorithm in polynomial time O(nᵏ) for some constant k. Examples: sorting, searching, shortest paths, MST, bipartiteness testing.

NP — Nondeterministic Polynomial Time

The class of decision problems where a "yes" answer can be verified in polynomial time given a certificate (witness). Equivalently, solvable by a nondeterministic polynomial-time algorithm. P ⊆ NP — every polynomial-time algorithm is also a polynomial-time verifier.

NP-Hard

A problem H is NP-Hard if every problem in NP can be polynomial-time reduced to H. H is at least as hard as the hardest problem in NP. H need not be in NP itself (e.g., Halting problem is NP-Hard but not in NP).

NP-Complete

A problem C is NP-Complete if (1) C ∈ NP (solutions are polynomial-time verifiable), AND (2) C is NP-Hard (every NP problem reduces to C). NP-Complete = NP ∩ NP-Hard. These are the "hardest" problems in NP.

P vs NP — The Million Dollar Question

It is unknown whether P = NP. If P = NP, then every problem whose solution can be quickly verified can also be quickly solved — cryptography, optimization, AI would all be revolutionized. The prevailing belief (but not proven) is P ≠ NP. GATE accepts "P ≠ NP is unproven" as the current state of knowledge.

7.2   Proving NP-Completeness by Reduction GATE Tested

To prove problem B is NP-Complete:

  1. 1
    Show B ∈ NP: given a solution, verify it in polynomial time.
  2. 2
    Show B is NP-Hard: choose a known NP-Complete problem A. Show A ≤_p B — polynomial-time reduction from A to B.

Reduction chain GATE uses:

SAT → 3-SAT → Independent Set → Vertex Cover → Clique ↓ 3-SAT → Hamiltonian Cycle → TSP ↓ 3-SAT → Subset Sum → 0/1 Knapsack SAT is the "first" NP-Complete problem (Cook-Levin theorem, 1971). All others proved NP-Complete by reduction FROM a known NP-Complete problem.
NP-Complete Problems Have Polynomial Verifiers

For SAT: given an assignment of variables, evaluate the formula in polynomial time — easy. For Hamiltonian Cycle: given a cycle, check it visits every vertex once — polynomial. For 0/1 Knapsack: given a subset, sum weights and values — polynomial. The verification is always polynomial; it's the FINDING that's hard.

8.3   GATE Question Patterns

  • LCS length NAT: "Find LCS of AGGTAB and GXTXAYB." Fill the dp table. Answer: 4 (GTAB). 2-mark NAT.
  • Knapsack value: "Given items and capacity, what is the maximum value?" Trace the DP table. Very common 2-mark question — verify with the recurrence, don't guess.
  • MCM cost: "Given matrix dimensions, what is the minimum number of multiplications?" Apply MCM DP. Answer depends on which split minimizes cost.
  • Greedy vs DP distinction: "Which problems can be solved optimally by a greedy algorithm?" Activity selection YES. 0/1 Knapsack NO. Fractional Knapsack YES. Coin Change (general) NO. Huffman YES. 1-mark MCQ.
  • Huffman code length: "Given character frequencies, what is the code length for character X?" Build the Huffman tree step by step, trace the path to X.
  • NP-Complete classification: "Which of the following is NP-Complete?" SAT, TSP (decision), 3-Coloring, Hamiltonian Cycle — all NP-Complete. Shortest Path, MST, Sorting — in P. 1-mark MCQ.
  • NP reduction direction: "If 3-SAT reduces to Problem X in polynomial time, what can you conclude?" X is NP-Hard. If also X ∈ NP, then X is NP-Complete.
  • P vs NP: "If a polynomial algorithm is found for SAT, what follows?" P = NP (since SAT is NP-Complete, polynomial algorithm for SAT → every NP problem solvable in polynomial time). 1-mark reasoning.
