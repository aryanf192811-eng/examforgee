Chapter 01
Turing Machine — Model & Formal Definition

1.1   What Is a Turing Machine? Core

A Turing Machine (TM) is the most powerful model of computation known. It was conceived by Alan Turing in 1936 as an abstract model of a human computer performing mechanical calculation. Every algorithm that can be computed by any digital computer can also be computed by a Turing machine — and vice versa. This is not just a claim; it is the Church-Turing thesis, the foundational assertion of theoretical computer science.

The key difference from a finite automaton: a TM has an infinite tape as its memory. It can read from and write to the tape, and move the read/write head in either direction. This gives it unbounded memory — unlike FAs (no memory beyond states) and PDAs (stack memory, but restricted access).

Turing Machine Architecture ┌─────────────────────────────────────────┐ Infinite Tape: │ B │ a │ a │ b │ b │ B │ B │ B │ B │ ...│ └─────────────────────────────────────────┘ ↑ Read/Write Head (moves Left or Right) │ ┌──────────┴────────────┐ │ Finite Control │ │ (current state, │ │ transition fn δ) │ └───────────────────────┘ B = blank symbol (fills all unwritten tape cells) Head starts at leftmost symbol of input.
TM — 7-Tuple Definition

A TM is M = (Q, Σ, Γ, δ, q₀, q_accept, q_reject) where:
Q — finite set of states
Σ — input alphabet (does not contain blank B)
Γ — tape alphabet (Σ ⊆ Γ, B ∈ Γ)
δ: Q × Γ → Q × Γ × {L, R} — transition function
   (current state, tape symbol) → (new state, write symbol, direction)
q₀ ∈ Q — start state
q_accept ∈ Q — accept state (halts and accepts)
q_reject ∈ Q — reject state (halts and rejects), q_accept ≠ q_reject

Three Outcomes for a TM on Input w

Accept: TM halts in q_accept. String w is in L(M).
Reject: TM halts in q_reject. String w is NOT in L(M).
Loop (diverge): TM runs forever without halting. This is the fundamental problem — we cannot always detect looping from outside.

1.2   TM Computation — Configuration & Moves Core

Configuration

A configuration of a TM captures its complete state at an instant: (current state, tape contents, head position). Written as: uqv where q is current state, u is tape content left of head, v is tape content from head position rightward (v's first symbol is what head is reading).

TM for L = {w#w | w ∈ {0,1}*} — zigzag matching:

Strategy: Cross off matching symbols on both sides of # using marks. 1. Read leftmost unmarked symbol from left side (say '0'), mark it (write 'x'). 2. Scan right across left side and # to find first unmarked symbol on right side. 3. If it matches, mark it. If not, reject. 4. Return to leftmost unmarked symbol on left side. 5. Repeat until all symbols matched. Accept if # separates two identical halves. This requires O(n²) steps for length-n strings. A TM can do this; a PDA cannot (PDAs can't compare both halves arbitrarily).

The language accepted by M is L(M) = {w | M accepts w}. The collection of languages recognized by TMs is exactly the class of Recursively Enumerable (RE) languages.

TM Variants & The Church-Turing Thesis

2.1   TM Variants — All Equivalent in Power GATE Tested

Many extensions of the basic TM model have been proposed. Every one of them turns out to be equivalent in computational power to the standard single-tape TM. This robustness is part of what makes the Church-Turing thesis so compelling.

Nondeterministic TM ≠ More Powerful

Unlike PDAs where NPDA > DPDA, a nondeterministic TM (NTM) recognizes exactly the same class of languages as a deterministic TM. An NTM accepting a string has at least one computation branch that accepts — a deterministic TM can simulate this by doing a BFS over all branches. The simulation is exponentially slower, but terminates. This is the P vs NP question in disguise at the level of decidability.

2.2   The Church-Turing Thesis Core

Church-Turing Thesis

Every effectively computable function (every function that can be computed by any algorithm, by any physical machine, by any reasonable computational model) can be computed by a Turing machine.

This is a thesis, not a theorem — it cannot be formally proven because "effectively computable" is an informal notion. However, it is universally accepted because:

  • Every known model of computation (λ-calculus, recursive functions, RAM machines, modern computers) has been proven equivalent to TMs.
  • No one has ever found a physically realizable computation that a TM cannot perform.
  • The robustness of TMs to extensions (multitape, nondeterminism, etc.) is strong evidence of their universality.
Practical Implication

When we say a problem is "computable," we mean a TM can solve it. When we say it is "uncomputable," we mean no TM can solve it — which also means no real computer can solve it (for all inputs). The TM is the mathematical proxy for all computers.

2.3   The Universal Turing Machine (UTM) GATE Tested

Universal TM

A Universal TM (UTM) U is a single TM that can simulate any other TM M on any input w. U takes as input the encoding of M (written ⟨M⟩) and the input w, and simulates M's computation on w: U accepts iff M accepts w, U rejects iff M rejects w, U loops iff M loops.

The UTM is the theoretical foundation of the stored-program computer. A modern CPU is essentially a UTM — it takes a program (encoding of M) and data (w) and executes it. The concept of software running on hardware comes directly from Turing's 1936 paper.

Encoding a TM — ⟨M⟩

Any TM M can be encoded as a binary string ⟨M⟩. The encoding captures Q, Σ, Γ, δ, q₀, q_accept, q_reject in a systematic format. This encoding is crucial — it allows TMs to take other TMs as input, enabling the diagonal argument and undecidability proofs.

Chapter 03
Decidable Languages

3.1   Decidability — Precise Definitions Core

Recognizable (Recursively Enumerable)

A language L is Turing-recognizable (or recursively enumerable, RE) if there exists a TM M such that M accepts every string in L, and M either rejects or loops on every string not in L.

Decidable (Recursive)

A language L is decidable (or recursive) if there exists a TM M — called a decider — that always halts (never loops) AND accepts every string in L and rejects every string not in L.

The difference is critical: a recognizer may loop forever on strings not in L. A decider always answers yes or no in finite time. Every decidable language is recognizable, but not every recognizable language is decidable.

Decider

Always halts. On L: accept. On L̄: reject. No looping on any input. The "nice" case — like a function that always returns a value.

Recognizer

On L: always accepts (and halts). On L̄: rejects OR loops forever. We can't always distinguish "will reject eventually" from "looping forever."

Not Recognizable

No TM exists that accepts all strings in L. Some strings in L will be missed — either rejected or looped on. Language is "beyond" TM reach.

3.2   Decidable Problems — Key Examples GATE Tested

The Pattern — Simpler Models Are Decidable, TMs Are Not

Every acceptance/emptiness/equivalence problem for DFAs and NFAs is decidable — these models always halt, so simulation always terminates. CFG acceptance and emptiness are decidable, but CFG equivalence is not. TM acceptance is undecidable — the model is powerful enough to loop, and we cannot detect that looping in general. This layered pattern is testable: the more powerful the model, the fewer decision problems are solvable.

Chapter 04
Undecidability — The Diagonalization Argument

4.1   Why Undecidable Problems Must Exist Core

Before proving any specific language undecidable, we can show that most languages must be undecidable using a simple counting argument.

The Counting Argument

The set of all TMs is countably infinite — each TM can be encoded as a binary string, and the set of all binary strings is countable (can be listed: ε, 0, 1, 00, 01, 10, 11, ...).

The set of all languages over Σ = {0,1} is uncountably infinite — by Cantor's diagonal argument, the power set of a countably infinite set (Σ*) is uncountable.

Therefore: there are uncountably many languages but only countably many TMs. Most languages have no TM. Most languages are not even recognizable.

This argument doesn't tell us which specific languages are undecidable — just that they must exist in overwhelming quantity. The diagonalization argument constructs a specific undecidable language.

4.2   Diagonalization — The Diagonal Language L_d GATE Tested

Cantor's diagonalization constructs something that differs from every item on a supposedly complete list. Turing applied this to TMs to construct a language no TM can decide.

The Diagonal Language

List all TMs M₁, M₂, M₃, ... and all binary strings w₁, w₂, w₃, ... Form an infinite matrix where entry (i,j) = 1 if Mᵢ accepts wⱼ, else 0.

Define L_d = {wᵢ | Mᵢ does NOT accept wᵢ} — the set of strings whose TM "rejects itself."

L_d cannot be recognized by any TM. Proof: suppose Mₖ recognizes L_d. Ask: does Mₖ accept wₖ?
If Mₖ accepts wₖ: then wₖ ∉ L_d (definition). But L_d = L(Mₖ) means wₖ ∈ L_d. Contradiction.
If Mₖ does not accept wₖ: then wₖ ∈ L_d. But L_d = L(Mₖ) means Mₖ should accept wₖ. Contradiction.
Either way: contradiction. L_d is not recognizable by any TM.

L_d vs A_TM

L_d (the diagonal language) is NOT recognizable at all — no TM even partially recognizes it. A_TM (the halting/acceptance problem) IS recognizable (a UTM recognizes it) but not decidable. These are different levels of undecidability. GATE tests whether you can distinguish "not recognizable" from "recognizable but not decidable."

Chapter 05
The Halting Problem — A_TM is Undecidable

5.1   A_TM — The Acceptance Problem Most Tested

A_TM — The Acceptance Problem

A_TM = {⟨M, w⟩ | M is a TM and M accepts input w}

A_TM is recognizable (a UTM U can simulate M on w — if M accepts, U accepts) but NOT decidable. No TM can always correctly answer "does M accept w?" in finite time.

Proof that A_TM is Undecidable — By Contradiction

  1. 1
    Assume for contradiction that A_TM is decidable. Then there exists a TM H (a decider) such that: H accepts ⟨M,w⟩ if M accepts w, and H rejects ⟨M,w⟩ if M does not accept w. H always halts.
  2. 2
    Construct a new TM D using H: D takes input ⟨M⟩ (encoding of a TM M). D runs H on ⟨M, ⟨M⟩⟩ (asks: does M accept its own encoding?). D accepts if H rejects, and D rejects if H accepts. D is a "self-referential inverter."
  3. 3
    Ask: what does D do on input ⟨D⟩ (D applied to its own encoding)?
    If D accepts ⟨D⟩: → H rejects ⟨D,⟨D⟩⟩ (by D's construction) → M=D does NOT accept ⟨D⟩ (by H's correctness) → D does NOT accept ⟨D⟩. CONTRADICTION. If D rejects ⟨D⟩: → H accepts ⟨D,⟨D⟩⟩ → M=D DOES accept ⟨D⟩ → D accepts ⟨D⟩. CONTRADICTION.
  4. 4
    Both cases lead to contradiction. Our assumption (H exists) is false. A_TM is undecidable. ∎
The Structure of This Proof — Internalize It

This proof appears in many forms in GATE. The key structure is always: (1) Assume a decider H for the problem exists. (2) Construct a self-referential machine D using H. (3) Ask what D does on ⟨D⟩. (4) Derive a contradiction. The specific problem changes, but the template remains identical. Every undecidability proof via direct diagonalization follows this blueprint.

5.2   HALT_TM — The Halting Problem GATE Classic

HALT_TM

HALT_TM = {⟨M, w⟩ | M is a TM and M halts on input w} (either accepts or rejects — just doesn't loop)

HALT_TM is undecidable. Proof is by reduction from A_TM: if we could decide HALT_TM, we could decide A_TM (which is impossible). The reduction:

Reduction: A_TM ≤_m HALT_TM Given ⟨M, w⟩ (instance of A_TM), construct ⟨M', w⟩ where: M' simulates M on w. If M accepts w → M' accepts (halts). If M rejects w → M' accepts (halts). ← convert reject to accept If M loops → M' loops. Now: HALT_TM decider on ⟨M', w⟩ tells us if M' halts on w. M' halts on w iff M halts on w (loop case preserved). If M halts → run M on w and see if it accepts → decides A_TM. But A_TM is undecidable → HALT_TM must be undecidable too. ∎
HALT_TM is Recognizable

HALT_TM is RE (recognizable): simulate M on w using a UTM. If M halts (accept or reject), accept ⟨M,w⟩. If M loops, the UTM loops too. The recognizer accepts all strings in HALT_TM — it just may not halt on strings not in HALT_TM. HALT_TM is recognizable but NOT decidable.

Chapter 06
Reductions & Undecidable Problems

6.1   Many-One Reductions Most Tested

Many-One Reduction (Mapping Reduction)

A is many-one reducible to B (written A ≤_m B) if there exists a computable function f such that for every string w: w ∈ A ↔ f(w) ∈ B. The function f "translates" instances of A into instances of B.

Reductions are the workhorse of undecidability proofs. The key inference rules:

Upward Transfer (Decidability)

If A ≤_m B and B is decidable, then A is decidable.
A is "no harder than" B.

Downward Transfer (Undecidability)

If A ≤_m B and A is undecidable, then B is undecidable.
B is "at least as hard as" A.

Direction of Inference — The Most Common Mistake

If A ≤_m B: A reduces TO B. We can solve A using B.
To prove B undecidable: show A ≤_m B where A is already known undecidable (e.g., A_TM).
To prove B decidable: show B ≤_m A where A is already known decidable.
Never conclude "B is undecidable because B ≤_m A_TM" — this is backwards. The reduction must go FROM the undecidable problem TO B.

Proving E_TM Undecidable — A Full Reduction

E_TM = {⟨M⟩ | L(M) = ∅} is undecidable. Proof by reduction from A_TM:

Reduction: A_TM ≤_m E_TM (via complement) Given ⟨M, w⟩ (instance of A_TM), construct TM M_w: M_w on input x: Ignore x. Simulate M on w. If M accepts w → M_w accepts x. If M rejects w → M_w rejects x. Key property: If M accepts w: M_w accepts ALL inputs → L(M_w) = Σ* ≠ ∅ If M does not accept w: M_w accepts NO inputs → L(M_w) = ∅ So: M accepts w ↔ L(M_w) ≠ ∅ ↔ ⟨M_w⟩ ∉ E_TM This means: A_TM ≤_m Ē_TM (complement of E_TM). Since A_TM is undecidable, Ē_TM is undecidable. Since Ē_TM is undecidable and decidability is closed under complement, E_TM is also undecidable. ∎

Post Correspondence Problem (PCP):

Given a finite list of domino pairs {[t₁/b₁], [t₂/b₂], ..., [tₖ/bₖ]}, does there exist a sequence of indices i₁, i₂, ..., iₙ (repetition allowed) such that t_{i₁}t_{i₂}...t_{iₙ} = b_{i₁}b_{i₂}...b_{iₙ}?

Example: dominos [a/ab], [b/a], [ab/b] Try i=1,2,1,3: Top: a + b + a + ab = abab Bot: ab + a + ab + b = ababab → Not equal Try i=1,1,3: Top: a + a + ab = aaab Bot: ab + ab + b = ababb → Not equal PCP asks: does ANY sequence work? This is undecidable in general.

Chapter 07
Rice's Theorem — The Universal Undecidability Result

7.1   Rice's Theorem — Statement & Proof GATE Critical

Rice's Theorem

Every non-trivial property of the language of a Turing machine is undecidable.

A property P of TM languages is:
Trivial if P holds for ALL TMs, or P holds for NO TMs (vacuously true/false).
Non-trivial if P holds for some TMs but not others.

Rice's Theorem: if P is non-trivial, then {⟨M⟩ | L(M) satisfies P} is undecidable.

Rice's theorem is a meta-theorem — it subsumes infinitely many individual undecidability results. Instead of proving each language property undecidable separately, Rice's theorem lets you invoke it directly.

Applying Rice's Theorem — 3-Step Check

  1. 1
    Is it a property of the language L(M)? The property must depend only on what language M recognizes, not on how M is structured (number of states, tape symbols, etc.).
  2. 2
    Is it non-trivial? Find one TM where the property holds, and one where it doesn't.
  3. 3
    If both conditions hold: invoke Rice's Theorem → the problem is undecidable. Done.

Apply Rice's Theorem to these problems:

ProblemProperty of L(M)?Non-trivial?Verdict
{⟨M⟩ | L(M) contains "01"}YesYes (some do, some don't)Undecidable ✓
{⟨M⟩ | L(M) = Σ*}YesYesUndecidable ✓
{⟨M⟩ | L(M) is regular}YesYesUndecidable ✓
{⟨M⟩ | L(M) is a CFL}YesYesUndecidable ✓
{⟨M⟩ | L(M) is infinite}YesYesUndecidable ✓

Rice's Theorem Scope — Language Properties Only

Rice's theorem applies ONLY to properties of the language L(M), not properties of the TM M itself. "M has fewer than 10 states" is a property of the machine — Rice's theorem doesn't apply (and this IS decidable). "L(M) is finite" is a property of the language — Rice's theorem applies (undecidable). The distinction is subtle but critical for GATE.

7.2   Why Rice's Theorem is True — Proof Sketch Advanced

The proof reduces A_TM to any non-trivial language property problem. Let P be a non-trivial property. Since P is non-trivial, there exist TMs M_yes (L(M_yes) satisfies P) and M_no (L(M_no) does not satisfy P).

Proof sketch (assume ∅ does not satisfy P — the other case is symmetric) Since P is non-trivial and ∅ doesn't satisfy P: ∃ M_yes such that L(M_yes) satisfies P. Given ⟨M, w⟩ (instance of A_TM), construct TM M': M' on input x: 1. Simulate M on w. 2. If M accepts w: simulate M_yes on x, accept if M_yes accepts. 3. If M never finishes step 1: M' never reaches step 2. Key: If M accepts w: L(M') = L(M_yes) → satisfies P → ⟨M'⟩ ∈ {⟨T⟩ | L(T) satisfies P} If M rejects/loops w: L(M') = ∅ → doesn't satisfy P → ⟨M'⟩ ∉ {⟨T⟩ | L(T) satisfies P} So: M accepts w ↔ ⟨M'⟩ ∈ L_P This is a reduction A_TM ≤_m L_P. Since A_TM is undecidable → L_P is undecidable. ∎

Chapter 08
RE, co-RE & The Decidability Hierarchy

8.1   RE, co-RE, and Their Relationship GATE Critical

Recursively Enumerable (RE)

L is RE if there exists a TM that accepts every string in L (and either rejects or loops on strings not in L). Equivalently, a TM can enumerate all strings in L (print them out one by one).

co-RE

L is co-RE if its complement L̄ is RE. That is, L is co-RE if there exists a TM that accepts every string NOT in L.

Decidable (Recursive)

L is decidable if and only if L is BOTH RE and co-RE. Equivalently, both L and L̄ are RE.

Deciding with Two Recognizers

If L is RE and co-RE, you can decide L by running two TMs in parallel: one that recognizes L, one that recognizes L̄. One of them must eventually accept — and when it does, you know the answer. This is why RE ∩ co-RE = Decidable: you don't need a single decider, you just need both sides to be recognizable.

8.2   Enumeration & RE Languages Core

Enumerator Theorem

A language L is RE if and only if there exists a TM (enumerator) that prints (enumerates) all strings in L one by one — possibly with repetitions — and eventually prints every string in L.

This gives the alternative name "recursively enumerable" — L can be listed by a TM. If L is finite, the enumerator eventually stops. If L is infinite, the enumerator runs forever but prints every string eventually.

Why RE Languages Are Called "Recognizable"

A TM recognizes L means: it says "yes" for every string in L. It might say "no" or might loop forever for strings not in L. An enumerator for L essentially runs all computations in parallel and announces results — this is the same as recognizing. The two definitions are provably equivalent.

9.1   All the Traps in One Place

#TrapCorrect Understanding
1NTM = DTM in powerUnlike PDAs, nondeterministic TMs and deterministic TMs recognize exactly the same languages (RE). An NTM can be simulated by a DTM — exponentially slower but equivalent in what it can compute.
2A_TM is recognizableA_TM is NOT decidable, but it IS recognizable — a UTM recognizes it. "Undecidable" does NOT mean "unrecognizable." These are different levels. Always specify which level.
3Rice's theorem scopeRice's applies ONLY to properties of L(M) — what language the TM recognizes. Properties of the TM itself (number of states, transitions) are NOT covered. "M has 5 states" is a machine property — decidable. "L(M) is finite" is a language property — undecidable by Rice's.
4Reduction directionTo prove B undecidable: show A ≤_m B where A is known undecidable. You reduce FROM the hard problem TO B. "A reduces to B" means: if B were decidable, then A would be decidable. Since A isn't, B isn't.
5E_TM vs A_TM — which is RE, which is co-REA_TM is RE (not co-RE). E_TM is co-RE (not RE). Their complements swap: Ā_TM is co-RE, Ē_TM is RE. EQ_TM is neither RE nor co-RE.
6Decidable = RE ∩ co-REA language is decidable if AND ONLY IF it is both RE and co-RE. To show undecidability, show either L is not RE or L is not co-RE (or that one of its completions falls into the wrong class).
7Halting problem is not "halt on empty input"HALT_TM = {⟨M,w⟩ | M halts on w} — it asks about any w. HALT_ε = {⟨M⟩ | M halts on ε} is a different (also undecidable) problem. Both are undecidable but they are distinct languages.
8Finite language ⊄ decidable implies regularEvery finite language IS regular (can be expressed as a finite union of strings). Every regular language is decidable. But not every decidable language is regular — decidable languages are a much broader class including many non-regular, non-CFL languages.
9CFG equivalence vs CFL equivalence"Is L(G₁) = L(G₂)?" for CFGs is undecidable. "Are these two DFAs equivalent?" is decidable. The decidability of equivalence depends critically on the model — CFG and above: undecidable. DFA/NFA: decidable.
10Rice's theorem covers ALL non-trivial propertiesThere is no clever algorithm for any non-trivial language property of TMs. People often think "but this particular property seems simple" — it doesn't matter. If it's non-trivial and about L(M), Rice's theorem says it's undecidable, period.

9.3   GATE Question Patterns

  • Classify a language: Given a language description, identify RE/co-RE/decidable/none. The hierarchy diagram is your guide. 1–2 mark MCQ.
  • Apply Rice's theorem: "Is the following problem decidable? {⟨M⟩ | L(M) contains at least 5 strings}." Check: language property? Non-trivial? → Undecidable by Rice's. 1-mark direct application.
  • Reduction validity: "Which reduction proves HALT_TM is undecidable?" — A_TM ≤_m HALT_TM. Direction must be correct. 1-mark trap.
  • TM vs NTM power: "An NTM recognizes more languages than a DTM — True or False?" False. They recognize the same class. 1-mark.
  • Recognizable but not decidable: "Give an example of a language that is RE but not decidable." A_TM, HALT_TM. Know at least 3 examples cold.
  • Two-recognizer decidability: "If both L and L̄ are recognizable, what can you conclude?" L is decidable. This is the RE ∩ co-RE = Decidable theorem applied directly.
  • Reduction construction: Given a new problem, construct a reduction from A_TM to prove it undecidable. This is a 2-mark question requiring the full reduction argument.
  • Properties of TM (not Rice's): "Is it decidable whether a TM has an even number of states?" Yes — count the states in the encoding. Not a language property, so Rice's doesn't apply.
