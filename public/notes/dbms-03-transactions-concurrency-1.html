Chapter 01
Transactions & ACID Properties

1.1   What Is a Transaction? Core

Definition

A transaction is a logical unit of work that accesses and possibly modifies database items. It is a sequence of database operations (reads and writes) that must be executed as a single, indivisible unit — either all operations complete successfully (commit) or none of them take effect (abort/rollback).

The canonical example: transferring ₹1000 from account A to account B. This involves two operations — debit A and credit B. If the system crashes after debiting A but before crediting B, money has vanished. A transaction ensures both happen or neither happens.

A transaction transitions through these states:

Transaction State Diagram Active → Partially Committed → Committed (success) ↓ ↓ Failed ← Failed ↓ Aborted (rolled back) Active: Executing operations Partially Committed: Last operation executed, awaiting final commit Committed: All changes permanently saved Failed: Normal execution cannot proceed Aborted: Rolled back, database restored to state before transaction

1.2   ACID Properties GATE Tested

Atomicity

"All or nothing." Either every operation in the transaction executes, or none do. Ensured by the recovery manager using undo logs.

Consistency

A transaction takes the database from one consistent state to another. Consistency is the programmer's responsibility — the DBMS enforces integrity constraints.

Isolation

Concurrent transactions execute as if they were serial — no partial results of one transaction are visible to another. Ensured by the concurrency control manager.

Durability

Once a transaction commits, its changes persist permanently — even if the system crashes immediately after. Ensured by the recovery manager using redo logs.

ACID → Recovery & Concurrency Connection

Atomicity + Durability are handled by the recovery system (undo/redo logs, checkpoints). Isolation is handled by concurrency control (2PL, timestamps). Consistency is a user/programmer contract enforced through integrity constraints. This mapping appears in GATE as "which ACID property is violated by schedule X?"

Chapter 02
Schedules & Serializability

2.1   Schedules — Definition & Types Core

Definition

A schedule is an ordering of operations from multiple concurrent transactions. It specifies the chronological order in which operations (read/write) from all transactions are interleaved. A schedule must preserve the relative order of operations within each individual transaction.

Notation: R₁(X) = Transaction 1 reads X. W₂(X) = Transaction 2 writes X. C₁ = Transaction 1 commits. A₂ = Transaction 2 aborts.

Serial Schedule

Transactions execute one after another with no interleaving. Always correct (consistent). But wastes CPU while one transaction waits for I/O.

Concurrent Schedule

Operations from multiple transactions are interleaved. Better throughput and resource utilization. May or may not be correct — correctness must be verified.

Serializable Schedule

A concurrent schedule that produces the same result as some serial schedule. This is the gold standard for correctness in concurrent execution.

Why Not Just Run Serially?

If T1 takes 10 seconds and T2 takes 10 seconds, serial execution takes 20 seconds. If T1 waits 8 seconds for disk I/O during which T2 can run, concurrent execution takes ~12 seconds. Concurrency gives throughput — but requires careful control to remain correct.

2.2   Concurrency Problems GATE Tested

1. Dirty Read (Uncommitted Dependency)

T2 reads a value written by T1 before T1 commits. If T1 then aborts, T2 has read a value that never officially existed.

TimeT1T2
1W1(X=50)
2R2(X) ← reads 50 (dirty!)
3ABORT
4Uses X=50 (which never existed)

2. Lost Update

Two transactions read the same value, both modify it, and one update overwrites the other.

TimeT1T2
1R1(X=100)
2R2(X=100)
3W1(X=150)
4W2(X=120) ← overwrites T1's update!

T1's update (100→150) is lost. Final value 120 ignores T1's work.

3. Unrepeatable Read (Non-repeatable Read)

T1 reads X, T2 modifies and commits X, T1 reads X again — gets a different value.

4. Phantom Read

T1 executes a query (e.g., count of records satisfying a condition), T2 inserts a new record satisfying that condition, T1 repeats the query — gets a different count. The "phantom" record appeared between the two reads.

Conflict Serializability & Precedence Graphs

3.1   Conflicting Operations Core

Definition — Conflict

Two operations conflict if and only if ALL three conditions hold: (1) they belong to different transactions, (2) they access the same data item, and (3) at least one is a Write.

Operation PairSame Transaction?Same Item?At least one Write?Conflict?
R₁(X), R₂(X)NoYesNo (both reads)No conflict
R₁(X), W₂(X)NoYesYesConflict ✓
W₁(X), R₂(X)NoYesYesConflict ✓
W₁(X), W₂(X)NoYesYes (both writes)Conflict ✓
W₁(X), W₁(Y)Yes (same T1)DifferentYesNo conflict
Conflict Equivalent Schedules

Two schedules are conflict equivalent if one can be transformed into the other by swapping the order of non-conflicting operations. Swapping conflicting operations changes the result and is not allowed.

Conflict Serializable

A schedule is conflict serializable if it is conflict equivalent to some serial schedule. This is the standard test for correctness used in GATE.

3.2   Precedence Graph Algorithm Most Tested

The precedence graph (also called the serialization graph) gives a mechanical test for conflict serializability. You don't need to try all serial schedules — just build the graph and check for cycles.

Algorithm — Step by Step

Step 1: Create one node per transaction (T1, T2, T3, ...).
Step 2: For each pair of conflicting operations, draw a directed edge from the transaction whose operation comes first in the schedule to the transaction whose operation comes second.
Step 3: Draw an edge Ti → Tj if any of these hold (and Tᵢ's operation appears before Tⱼ's):
  · Ri(X) then Wj(X)   [read-write conflict]
  · Wi(X) then Rj(X)   [write-read conflict]
  · Wi(X) then Wj(X)   [write-write conflict]
Step 4: If the graph has no cycle → schedule is conflict serializable. If it has a cycle → NOT conflict serializable.

Schedule S: R1(A), W2(A), R1(B), W2(B), C1, C2

TimeT1T2

3R1(B)
4W2(B)

6C2

Find conflicts:

  • R1(A) then W2(A): T1→T2 (write-read on A)
  • R1(B) then W2(B): T1→T2 (write-read on B)

Precedence graph: T1 → T2 (two edges, same direction). No cycle.

Result: Conflict Serializable ✓. Equivalent serial order: T1 then T2.

  • W2(B) then R1(B): T2→T1 (write-read on B)

Precedence graph: T1→T2 AND T2→T1. CYCLE detected.

Result: NOT Conflict Serializable ✗. Cannot be equivalent to any serial schedule.

Conflict Serializable ≠ View Serializable (but close)

Every conflict serializable schedule is view serializable, but not vice versa. View serializability is a broader class but NP-hard to test. GATE always tests conflict serializability using the precedence graph. Unless explicitly told otherwise, assume conflict serializability when the question asks about serializability.

Chapter 04
Recoverability Hierarchy

4.1   The Four Levels — From Weakest to Strongest GATE Critical

Serializability tells us about correctness. Recoverability tells us whether the database can be returned to a consistent state after a failure. These are orthogonal properties — a schedule can be serializable but not recoverable, or recoverable but not serializable.

Broadest — contains all others
All Schedules
Any interleaving, including non-recoverable ones
⊃
Level 1
Recoverable Schedules
Tj commits only after Ti commits, if Tj read X written by Ti
⊃
Level 2
Cascadeless (ACA) Schedules
Tj reads X only after Ti (who wrote X) commits. No dirty reads → no cascading rollbacks
⊃
Strictest
Strict Schedules
Tj neither reads nor writes X until Ti (who wrote X) commits or aborts
Strict ⊂ Cascadeless ⊂ Recoverable ⊂ All Schedules

Every strict schedule is cascadeless. Every cascadeless schedule is recoverable. Not vice versa. GATE asks: "Is this schedule recoverable? Cascadeless? Strict?" — check from the bottom up (strictest first). If it's strict, it's automatically cascadeless and recoverable.

4.2   Each Level — Precise Definitions & State Traces GATE Tested

Recoverable Schedule

Definition

A schedule is recoverable if: for every pair of transactions Tᵢ and Tⱼ, if Tⱼ reads a data item written by Tᵢ, then Tᵢ must commit before Tⱼ commits.

Non-recoverable (BAD):

TimeT1T2
1W1(X)
2R2(X) ← reads T1's write
3C2 ← T2 commits BEFORE T1!
4ABORT

T2 committed based on T1's write, but T1 aborted. T2 used a value that never existed. We CANNOT undo T2 because it already committed. The database is permanently inconsistent. This is a non-recoverable schedule.

Cascadeless Schedule (ACA — Avoids Cascading Aborts)

Definition

A schedule is cascadeless if every transaction reads only items written by already-committed transactions. No dirty reads are allowed.

Without this, aborting one transaction can force a chain of other transactions to abort — this is a cascading rollback. T1 writes X, T2 reads X, T3 reads X, T4 reads X — abort T1 requires aborting T2, T3, T4. This cascading effect is expensive and avoided by the cascadeless property.

Recoverable but NOT Cascadeless:

TimeT1T2
1W1(X)
2R2(X) ← dirty read (T1 not committed)
3C1
4C2

T2 commits AFTER T1 → recoverable ✓. But T2 read X before T1 committed → dirty read → NOT cascadeless ✗. If T1 had aborted at step 3, T2 would need to be rolled back too (cascading rollback).

Strict Schedule

Definition

A schedule is strict if no transaction reads OR writes a data item X until the last transaction that wrote X has committed or aborted. Both reads and writes are restricted, not just reads.

Strict schedules are the easiest to recover from — to undo a transaction, simply restore the before-image of all items it wrote. No need to worry about what other transactions did with those items.

Chapter 05
Concurrency Control — Two-Phase Locking

5.1   Lock Types & Compatibility Core

Shared Lock (S-Lock / Read Lock)

Acquired before a read. Multiple transactions can hold S-locks on the same item simultaneously. Any transaction with an S-lock can read, but no one can write.

Exclusive Lock (X-Lock / Write Lock)

Acquired before a write. Only one transaction can hold an X-lock at a time. No other transaction can read or write while X-lock is held.

5.2   Two-Phase Locking — Three Variants GATE Critical

2PL Rule

A transaction follows Two-Phase Locking if all lock acquisitions precede all lock releases. It has two phases: (1) Growing Phase — only acquire locks, never release. (2) Shrinking Phase — only release locks, never acquire new ones. The point of transition is called the lock point.

Key theorem: Any schedule produced by 2PL transactions is conflict serializable. The serial order is determined by the order of lock points.

Basic 2PL

Follow growing and shrinking phases. Locks can be released before commit.

Guarantees: Conflict serializability.

Does NOT prevent: Deadlocks, cascading rollbacks (dirty reads possible).

Strict 2PL

Hold all exclusive (write) locks until transaction commits or aborts.

Guarantees: Conflict serializability + no cascading rollbacks (strict schedule).

Does NOT prevent: Deadlocks.

Rigorous 2PL

Hold ALL locks (shared and exclusive) until commit or abort.

Guarantees: Conflict serializability + strictest schedules + no cascading rollbacks.

Does NOT prevent: Deadlocks (still possible).

ProtocolSerializable?No Cascading Rollbacks?No Deadlocks?Lock Release
Basic 2PLYesNoNoAnytime in shrinking phase
Strict 2PLYesYesNoX-locks held until commit/abort
Rigorous 2PLYesYesNoAll locks held until commit/abort
2PL Never Prevents Deadlocks

All 2PL variants can produce deadlocks. T1 holds lock on A, waits for B. T2 holds lock on B, waits for A. Neither can proceed. Deadlock detection (wait-for graph, cycle detection) or prevention (ordering, preemption) must be handled separately from 2PL.

Deadlock Detection — Wait-for Graph

Build a directed graph where Tᵢ → Tⱼ means Tᵢ is waiting for a lock held by Tⱼ. If the wait-for graph has a cycle, a deadlock exists. Break it by aborting one transaction in the cycle (the "victim").

Chapter 06
Timestamp Ordering & Thomas Write Rule

6.1   Timestamp-Based Concurrency Control Core

Timestamp ordering (TSO) is a lock-free concurrency protocol. Instead of locks, each transaction gets a unique timestamp when it starts: TS(T). The DBMS maintains for each data item X:

  • R-TS(X) — the timestamp of the most recent transaction that read X.
  • W-TS(X) — the timestamp of the most recent transaction that wrote X.
Basic Timestamp Ordering Rules

When T reads X:
If TS(T) < W-TS(X): T is trying to read a value written by a younger transaction. T is "too old" → ABORT and restart T.
Else: Allow read. Update R-TS(X) = max(R-TS(X), TS(T)).

When T writes X:
If TS(T) < R-TS(X): A younger transaction already read X expecting T hadn't written it yet → ABORT and restart T.
If TS(T) < W-TS(X): A younger transaction already wrote X → ABORT and restart T (basic protocol).
Else: Allow write. Update W-TS(X) = TS(T).

Advantages over 2PL

No deadlocks (aborts instead of waiting). No locks needed. High concurrency for read-heavy workloads.

Disadvantages

Starvation possible (a transaction keeps getting aborted and restarted). Long transactions get repeatedly killed by short ones.

6.2   Thomas Write Rule Advanced Distinction

Thomas Write Rule

A modification to the basic timestamp protocol: when T tries to write X and TS(T) < W-TS(X) (a younger transaction already wrote X), instead of aborting T — simply ignore (skip) T's write. The younger transaction's value is more recent and should be kept.

Why this works: If a younger transaction already wrote X and a later read will see that younger value anyway, the older transaction's write is obsolete — it will be immediately overwritten. Ignoring it is semantically equivalent to T writing it and then the younger transaction overwriting it. No harm is done.

Why it matters: The Thomas Write Rule allows schedules that are not conflict serializable but are still correct (view serializable). It increases concurrency by avoiding unnecessary aborts. This is the key distinction GATE tests: TSO with Thomas Write Rule generates a superset of the schedules generated by basic TSO.

Without Thomas Write Rule: T1 (TS=1) tries to write X, but T2 (TS=2) already wrote X (W-TS(X)=2). Since TS(T1)=1 < W-TS(X)=2 → ABORT T1.

With Thomas Write Rule: Same situation → ignore T1's write (it would be obsolete anyway). T1 continues executing without abort. More concurrency, same correctness.

Chapter 07
Log-Based Recovery — Undo, Redo & Checkpoints

7.1   The Write-Ahead Log (WAL) Core

The log is a sequence of records written to stable storage (disk) that records every action taken by every transaction. It is the DBMS's source of truth for recovery. The fundamental rule that governs it:

Write-Ahead Logging (WAL) Protocol

Before a data item X is modified on disk, the log record for that modification must be written to stable storage first. Before a transaction commits, all its log records (including the commit record) must be written to stable storage.

WAL ensures that on recovery, the log always contains enough information to undo incomplete transactions and redo committed ones — even if the system crashed mid-operation.

Log Record Types

<T, START>
Transaction T has begun execution
<T, X, V_old, V_new>
T wrote data item X, changing value from V_old to V_new (both needed for undo and redo)
<T, COMMIT>
T has completed successfully and its changes are permanent
<T, ABORT>
T has been rolled back and all its changes undone
<CHECKPOINT [T1,T2,...]>
Checkpoint taken while transactions T1, T2, ... were active

7.2   Undo and Redo Logic GATE Tested

UNDO — Atomicity

Applied to transactions that were active at crash time (started but did not commit). Use V_old values from log records to reverse all their writes. Restores database to state before these transactions began.

REDO — Durability

Applied to transactions that committed but whose changes may not have reached disk (they were in memory buffer). Use V_new values to reapply all their writes. Ensures committed work is not lost.

Recovery Algorithm — After a Crash

Step 1 — Analysis: Scan log from last checkpoint to crash point. Identify which transactions committed and which were active at crash.
Step 2 — REDO: Redo all operations of committed transactions in forward order (oldest first). Apply V_new values.
Step 3 — UNDO: Undo all operations of active (uncommitted) transactions in reverse order (newest first). Apply V_old values. Write <T, ABORT> for each undone transaction.

Log at crash time (system crashed after record 6):

<T1, START> <T2, START> <T1, X, 10, 50> ← T1 changed X from 10 to 50 <T2, Y, 20, 80> ← T2 changed Y from 20 to 80 <T1, COMMIT> ← T1 committed <T2, Z, 30, 90> ← T2 changed Z from 30 to 90 CRASH HERE — no commit record for T2

Recovery:

  • Committed: T1 (has COMMIT record) → REDO T1's writes: set X = 50 (V_new).
  • Active at crash: T2 (no COMMIT record) → UNDO T2's writes in reverse: set Z = 30 (V_old), then set Y = 20 (V_old). Write <T2, ABORT>.

Final state: X=50 (T1's committed change preserved), Y=20, Z=30 (T2's changes reversed).

7.3   Checkpoints — Reducing Recovery Time GATE Tested

Checkpoint

A checkpoint is a periodic synchronization point where: (1) all log records currently in memory are written to disk, (2) all modified database pages in memory (buffer) are written to disk, (3) a <CHECKPOINT> record is written to the log. After a checkpoint, recovery only needs to process the log from the checkpoint onwards — not from the beginning of time.

Without checkpoints, after a crash you'd need to replay the entire log history. With checkpoints every 5 minutes, you only need to recover the last 5 minutes of log records. This makes recovery dramatically faster at the cost of a small periodic overhead.

Checkpoint recovery boundary ───────────────[CHECKPOINT]──────────────────[CRASH] T1 started T2 started T1 committed T2 active (no commit) Recovery needs only the log from [CHECKPOINT] to [CRASH]: → REDO T1 (committed after checkpoint, may not be on disk) → UNDO T2 (active at crash, never committed) → Ignore everything before [CHECKPOINT] (already on disk)
Fuzzy Checkpoints

Traditional checkpoints pause all transactions during the checkpoint operation (expensive). Fuzzy checkpoints allow transactions to continue during the checkpoint — the <CHECKPOINT> record lists which transactions were active at checkpoint start. Recovery must handle transactions that may have begun before or during the checkpoint.

GATE Traps & Schedule Trace Reference

8.1   All the Traps in One Place

#TrapCorrect Understanding
1 Serializability ≠ Recoverability These are independent properties. A schedule can be serializable but not recoverable (T2 commits based on T1's write before T1 commits). A schedule can be recoverable but not serializable. Always check both.
2 Conflict Serializable ≠ View Serializable Conflict serializable ⊂ View serializable. Every CS schedule is VS, but not vice versa. GATE tests conflict serializability via precedence graphs.
3 Read-Read is not a conflict Two reads on the same item from different transactions do NOT conflict. Only pairs involving at least one write on the same item from different transactions are conflicts.
4 2PL guarantees serializability, not deadlock-freedom All 2PL variants guarantee conflict serializability. None prevent deadlocks. Deadlock handling is separate (detection + victim selection, or prevention via ordering).
5 Basic 2PL allows dirty reads Basic 2PL releases locks during shrinking phase — before commit. Another transaction may read those released items (dirty read). Only Strict and Rigorous 2PL prevent this by holding write locks until commit.
6 Strict ≠ Strict 2PL "Strict schedule" and "Strict 2PL protocol" are different concepts. Strict schedule = no transaction reads/writes X until previous writer commits. Strict 2PL = hold X-locks until commit. Strict 2PL produces strict schedules, but this relationship must be stated, not assumed.
7 Thomas Write Rule allows non-CS schedules TSO with Thomas Write Rule produces view serializable schedules that may NOT be conflict serializable. This is its distinguishing feature. The precedence graph test won't work for schedules using the Thomas Write Rule — they may fail the graph test but still be correct.
8 UNDO uses V_old, REDO uses V_new The log record <T, X, V_old, V_new> contains both values. UNDO: restore X = V_old (reverse the change). REDO: restore X = V_new (reapply the change). Getting this backwards is a common exam mistake.
9 Checkpoint eliminates log before it, not after After a checkpoint, transactions that committed BEFORE the checkpoint don't need to be redone (their changes are on disk). Transactions active DURING or AFTER the checkpoint still need analysis. The checkpoint record lists active transactions at that point.
10 Cascadeless ≠ No dirty reads in all cases Cascadeless (ACA) means you only read committed values. This eliminates dirty reads entirely. But it's stricter than just "recoverable" — recoverable only requires commit ordering, not restricting reads to committed data.

8.2   Schedule Classification — Quick Decision Algorithm

Step-by-step classification for any schedule

Step 1 — Recoverable? For every pair where Tj reads what Ti wrote: does Ti commit before Tj commits? If any Tj commits before Ti → NOT recoverable. Stop.

Step 2 — Cascadeless? For every read operation Rj(X): has the last writer of X already committed before this read? If any dirty read exists → NOT cascadeless (but may be recoverable).

Step 3 — Strict? For every operation on X by Tj: has every prior writer of X (Ti) already committed or aborted? If not → NOT strict.

Step 4 — Conflict Serializable? Build precedence graph. Cycle → NOT CS. No cycle → CS. Find equivalent serial order by topological sort.

Classify: R1(A), W2(A), C2, R1(B), W1(B), C1

  • Recoverable? T1 reads A (written by nobody yet — original value). T2 writes A and commits at time 3. T1 never reads T2's write. T1 commits at time 6. No dependency issue. ✓ Recoverable.
  • Cascadeless? R1(A) at time 1: no transaction has written A yet (original). R1(B) at time 4: nobody wrote B before this read. No dirty reads. ✓ Cascadeless.
  • Strict? T2 writes A at time 2 and commits at time 3. T1 doesn't read or write A after T2's write (T1 read A at time 1, before T2 wrote it). ✓ Strict.
  • Conflict Serializable? Conflicts: R1(A) then W2(A) → T1→T2. No other conflicts on same items between different transactions. Graph: T1→T2. No cycle. ✓ CS. Equivalent serial order: T1, T2.

Final classification: Strict, Cascadeless, Recoverable, AND Conflict Serializable. ✓
