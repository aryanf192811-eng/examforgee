Chapter 01
Why DBMS Exists — The Problem Statement

1.1   The Core Problem

Before diving into any definition, you need to understand why this whole subject exists. You have real-world information — students, courses, employees, transactions — and you need to store it digitally in a way that avoids three fundamental problems.

Imagine storing all your data in a single flat file or a poorly designed spreadsheet. What goes wrong?

Insert Anomaly

You cannot record a new department unless at least one student is enrolled in it. The fact "CSE department exists" is trapped inside the student record.

Delete Anomaly

Deleting the last student from CSE accidentally erases the fact that Dr. Rao heads CSE. One delete destroys two independent pieces of information.

Update Anomaly

Dr. Rao's name changes. You must now update every single row that belongs to CSE. Miss one row and your database becomes inconsistent — two different names for the same person.

All three anomalies arise from the same root cause: storing two independent facts in the same row. The relational model, functional dependencies, and normal forms are a systematic framework to detect and eliminate this problem.

The Golden Rule

Every concept in normalization exists to enforce one principle: each fact should be stored exactly once, in exactly one place. Keep returning to this when you feel lost.

1.2   Three Properties a Good Database Must Have

Consistency

No contradictions anywhere in the data. The same fact cannot have two different values in two places.

Non-Redundancy

No unnecessary repetition of facts. A fact stored in n places must be updated in n places — that's n chances for inconsistency.

Queryability

You can ask any question about the data and get a correct, consistent answer through a well-defined algebra.

Chapter 02
The Relational Model

2.1   What Is a Relation? Core

Definition

A relation is a named, two-dimensional structure consisting of a set of attributes (columns) and a set of tuples (rows), where each tuple assigns exactly one atomic value from each attribute's domain.

The word "set" here is precise and important. A set has no duplicates and no ordering. This means two things about a relation that most beginners miss:

  • There are no duplicate rows in a true relation. Two identical tuples cannot coexist.
  • The order of rows does not matter. There is no "first row" or "last row" in the mathematical sense.
  • The order of columns does not matter either. Each value is accessed by attribute name, not by position.
GATE Trap

A question may present a table with duplicate rows and ask "is this a valid relation?" The answer is no — a relation is a set of tuples and sets cannot have duplicates. SQL tables (called bags or multisets) do allow duplicates, but the mathematical relational model does not.

Running Example Used Throughout These Notes

RollNoNameDeptDeptHead
101AryanCSEDr. Rao
102MeeraECEDr. Sen
103KaranCSEDr. Rao

Relation: Student  |  4 attributes  |  3 tuples

2.2   Attribute, Tuple, Domain

Attribute

An attribute is a named column. In our example: RollNo, Name, Dept, DeptHead. The number of attributes in a relation is called its degree. Our Student relation has degree 4.

Domain

Definition

The domain of an attribute is the complete set of valid values that attribute is permitted to hold. It defines the type and constraint on that column.

Dept has domain {CSE, ECE, ME, CE, EE, ...} — a finite enumerated set.

RollNo has domain positive integers.

CGPA has domain [0.0, 10.0] — a bounded real range.

Tuple

A tuple is one row — a single assignment of values across all attributes. For a relation of degree n, each tuple is an ordered n-tuple. The number of tuples in a relation at a given time is its cardinality.

Degree

Number of columns (attributes). Rarely changes — it's part of the schema.

Cardinality

Number of rows (tuples). Changes constantly as data is inserted and deleted.

NULL — A Special Non-Value

Critical Distinction

NULL is not a value. NULL means "unknown" or "not applicable." It is emphatically not the same as 0, not the same as an empty string, and not the same as false. This distinction affects key constraints (primary keys cannot be NULL), join results, and aggregate functions. GATE has tested this directly.

2.3   Schema vs. Instance Common Trap

Schema

The schema of a relation is its structure: the relation name, the set of attribute names, and the domain of each attribute. The schema is the blueprint. It rarely changes.

Instance

An instance of a relation is the actual set of tuples stored at a particular moment in time. The instance is the current data. It changes every time you insert, update, or delete a row.

Schema: Student(RollNo: int, Name: varchar, Dept: varchar, DeptHead: varchar) — this never changes unless you ALTER the table.

Instance: the three rows shown in the table above — this is a snapshot of today. Tomorrow it might have 500 rows.

The Most Important Implication for GATE

When a GATE question asks you to find candidate keys or determine functional dependencies, it is asking about all possible instances of that relation — not just the specific instance shown. A key must work for every possible snapshot the relation could ever have, not just the one you're looking at right now.

"If you determine keys by looking at current data, you will get the wrong answer on GATE every single time."

2.4   Prime vs. Non-Prime Attributes GATE Critical

Prime Attribute

An attribute that is part of at least one candidate key is called a prime attribute.

Non-Prime Attribute

An attribute that does not appear in any candidate key is called a non-prime attribute (also called a non-key attribute).

This distinction is the entire foundation of 2NF and 3NF. Get it wrong and normalization becomes arbitrary guesswork.

Relation: Enrollment(RollNo, CourseID, CourseName, Grade)

Only candidate key: {RollNo, CourseID}

Prime attributes: RollNo, CourseID — both are part of the candidate key.

Non-prime attributes: CourseName, Grade — neither appears in any candidate key.

What to do first in any GATE normalization problem

Always find all candidate keys first. Then classify every attribute as prime or non-prime. Only then proceed to check normal forms. Skipping this step is the single most common reason for wrong answers.

Chapter 03
Keys — Types and Properties

3.1   The Four Key Types

A key is an attribute or set of attributes that uniquely identifies tuples within a relation. The word "uniquely" means: no two distinct tuples in the relation can have the same value for that attribute set. There are four related types, each more specific than the last.

Superkey

Definition

A superkey is any set of one or more attributes that uniquely identifies every tuple in every possible instance of a relation.

Superkeys can be large and contain "extra" attributes beyond what is strictly necessary. For the Student relation, all of the following are superkeys:

  • {RollNo} — because roll numbers are unique by design.
  • {RollNo, Name} — still unique; adding Name doesn't break uniqueness.
  • {RollNo, Name, Dept} — still unique, even more attributes added.
  • {RollNo, Name, Dept, DeptHead} — all attributes; always a superkey.

Candidate Key

Definition

A candidate key is a minimal superkey. It is a superkey from which no attribute can be removed while still maintaining the uniqueness property.

Minimality is the only thing that distinguishes a candidate key from a superkey. {RollNo, Name} is a superkey but not a candidate key because removing Name still leaves {RollNo} which is unique. {RollNo} itself cannot be reduced further — it is a candidate key.

A relation can have multiple candidate keys. If a relation stores both RollNo and Email and both are guaranteed unique, then {RollNo} and {Email} are both candidate keys.

Primary Key

Definition

The primary key is one candidate key chosen by the database designer as the official, enforced identifier for the relation. It is unique and cannot be NULL in any tuple.

Every relation has exactly one primary key. The remaining candidate keys (not chosen as PK) are called alternate keys. The choice of which candidate key to use as the PK is a design decision, not a mathematical one.

Foreign Key

Definition

A foreign key is an attribute (or set of attributes) in relation R whose values must either match a primary key value in some relation S, or be NULL. This constraint is called referential integrity.

If Enrollment has a column RollNo that references Student(RollNo), then you cannot insert an enrollment for RollNo 999 unless there exists a student with RollNo 999. Attempting to do so violates referential integrity.

Foreign keys can be NULL — a NULL FK simply means "this relationship does not exist for this row."

Chapter 04
Functional Dependencies

4.1   Definition and Intuition GATE Critical

Definition

A functional dependency X → Y (read: "X functionally determines Y") holds on a relation R if, in every valid instance of R, whenever two tuples agree on the values of attributes X, they must also agree on the values of attributes Y.

The intuition is simple: knowing the value of X tells you exactly what value Y must be. X → Y means Y is a function of X — there is one Y for every X.

Think of it like a lookup table. Dept → DeptHead means: if you tell me the department name, I can tell you the department head with certainty. Each department has exactly one head. The converse is not necessarily true — knowing the department head doesn't tell you which department they run (they could run multiple, in theory).

FDs that hold on the Student relation:

  • RollNo → Name — every roll number maps to exactly one student name.
  • RollNo → Dept — a student belongs to exactly one department.
  • Dept → DeptHead — each department has exactly one head.
  • RollNo → DeptHead — holds transitively through Dept (see Chapter 5).

FDs that do NOT hold:

  • Name → RollNo — two students can share the same name.
  • DeptHead → Dept — may hold currently, but not guaranteed by design.
  • Name → Dept — no design rule enforces this.

4.2   FDs Are Semantic, Not Observational Critical Trap

This is the most misunderstood point about functional dependencies. An FD is a constraint on the meaning of the data — it comes from the real-world rules your system models. It is not something you determine by looking at the current data.

Consider: currently, every student in CSE has Dr. Rao as their department head. Does this mean Name → Dept holds? Absolutely not. Two students could happen to have the same name. The current data doesn't violate it, but future data might. An FD holds only if you know — through the problem description or real-world logic — that it must always be true.

Rule

Never infer an FD by pattern-matching on a specific instance. FDs come from the semantics of the domain. In GATE problems, the FD set is always given explicitly, or derivable from the described domain rules.

Trivial vs. Non-Trivial FDs

Definition

An FD X → Y is trivial if Y ⊆ X — that is, Y is already part of X. Example: {RollNo, Name} → Name. This always holds and carries no information. An FD is non-trivial if Y ⊄ X.

In normalization, we only care about non-trivial FDs. Trivial dependencies are universal and meaningless for analysis.

4.3   Types of Functional Dependencies in Normalization

The following three types are the direct inputs to deciding which normal form a relation satisfies:

Full Functional Dependency

Definition

X → Y is a full functional dependency if Y is functionally dependent on X, but not on any proper subset of X. Removing any attribute from X breaks the dependency.

In Enrollment(RollNo, CourseID, Grade), the dependency {RollNo, CourseID} → Grade is full. If you remove RollNo, CourseID → Grade doesn't hold. If you remove CourseID, RollNo → Grade doesn't hold either. Both attributes are necessary.

Partial Functional Dependency

Definition

X → Y is a partial functional dependency if Y is functionally dependent on a proper subset of X. Some attribute in X is unnecessary for the dependency.

In Enrollment(RollNo, CourseID, CourseName, Grade), the dependency {RollNo, CourseID} → CourseName is partial because CourseID → CourseName alone already holds. RollNo is unnecessary — CourseName depends only on CourseID, which is a proper subset of the candidate key.

Partial dependencies on non-prime attributes are exactly what 2NF forbids.

Important Boundary Condition

Partial dependencies only make sense when the candidate key is composite (has two or more attributes). If the only candidate key is a single attribute, partial dependency is impossible by definition — you can't have a proper subset of a single-element set. This means any relation with a single-attribute CK is automatically in 2NF.

Transitive Functional Dependency

Definition

A non-prime attribute Z is transitively dependent on the primary key X if there exists a non-prime attribute Y such that X → Y and Y → Z, and Y does not functionally determine X (i.e., Y → X does not hold).

In Student(RollNo, Name, Dept, DeptHead):

RollNo → Dept (direct), and Dept → DeptHead (direct)

Therefore: RollNo → DeptHead (transitive)

Here, DeptHead is non-prime, and it reaches the key transitively through Dept, which is also non-prime. This transitive path through non-prime attributes is exactly what 3NF forbids.

Chapter 05
Armstrong's Axioms & Attribute Closure

5.1   Armstrong's Axioms Sound & Complete

Given a set of functional dependencies F, you need a systematic way to derive all dependencies that logically follow — this larger set is called F⁺, the closure of F. Armstrong's axioms are three rules that are sound (they never produce false FDs) and complete (together they can derive every valid FD in F⁺).

AxiomFormal StatementExampleIntuition
Reflexivity If Y ⊆ X, then X → Y {A,B} → A Knowing more always lets you know less. Trivial FD.
Augmentation If X → Y, then XZ → YZ A→B implies AC→BC Adding the same attribute to both sides of an FD preserves it.
Transitivity If X→Y and Y→Z, then X→Z A→B, B→C implies A→C If X determines Y, and Y determines Z, X can chain through to Z.

Three additional rules are derived from the axioms (not independent, but useful shortcuts):

Union

If X→Y and X→Z, then X→YZ.
Combine multiple RHS into one.

Decomposition

If X→YZ, then X→Y and X→Z.
Split a combined RHS.

Pseudotransitivity

If X→Y and WY→Z, then WX→Z.
Substitute X for Y in a larger FD.

5.2   Attribute Closure Algorithm Most Tested

Definition

The attribute closure of a set X under a set of FDs F, written X⁺, is the set of all attributes that X can functionally determine using the FDs in F. It is the largest set Y such that X → Y can be derived using Armstrong's axioms.

The closure algorithm is iterative and straightforward:

  1. 1
    Initialize: X⁺ = X (X always determines itself, by reflexivity).
  2. 2
    Scan all FDs in F. For each FD A → B: if A ⊆ X⁺ (all attributes of A are already in X⁺), then add B to X⁺.
  3. 3
    Repeat step 2 until X⁺ does not change in a full scan (fixpoint reached).
  4. 4
    The final X⁺ is the answer.

Relation: R(A, B, C, D, E)   FDs: AB→C,  C→D,  D→E,  E→B

Find {A,B}⁺:

Start: {AB}⁺ = { A, B } Round 1 scan: AB→C: AB ⊆ {A,B} ✓ → add C → {A, B, C} C→D: C ⊆ {A,B,C} ✓ → add D → {A, B, C, D} D→E: D ⊆ {A,B,C,D} ✓ → add E → {A, B, C, D, E} E→B: E ⊆ {A,B,C,D,E} ✓ → B already present, no change Round 2 scan: No new additions. STOP. {AB}⁺ = {A, B, C, D, E} = all attributes ✓ → AB is a superkey. → Is AB minimal? {A}⁺ = {A} ≠ all. {B}⁺ = {B} ≠ all. → Neither A nor B alone is a superkey. So AB is MINIMAL. → AB is a candidate key. ✓
Why Closure Matters — It Solves Everything

Is X a superkey? Compute X⁺. If it equals all attributes, yes.
Does FD X→Y hold in F⁺? Compute X⁺. If Y ⊆ X⁺, yes.
Find all candidate keys. Compute closures of subsets, test minimality.
Verify lossless decomposition. Use closure to check the intersection condition.

5.3   Canonical Cover 3NF Synthesis

Definition

A canonical cover Fc of a set of FDs F is a minimal equivalent set — it produces the same closure (Fc⁺ = F⁺) but contains no redundant FDs and no extraneous attributes. Every FD in Fc has a single attribute on its RHS.

An attribute A is extraneous in X → Y if removing A from X (or Y) still produces an equivalent set of FDs.

Steps to compute canonical cover:

  1. 1
    Decompose RHS: Split every FD so each has exactly one attribute on the right side. AB→CD becomes AB→C and AB→D.
  2. 2
    Remove extraneous LHS attributes: For each FD XA→Y, check if X→Y already holds (compute X⁺ without A). If yes, A is extraneous — remove it.
  3. 3
    Remove redundant FDs: For each FD X→Y, check if Y is derivable from the remaining FDs in Fc without using this FD. If yes, it is redundant — remove it.
Key Property

The canonical cover is not unique. Multiple valid canonical covers may exist for the same F. If a GATE question asks "which of the following is a canonical cover," any option that produces the same F⁺ and satisfies minimality conditions is correct.

Chapter 06
Normal Forms — 1NF through BCNF

6.0   The Hierarchy and Its Purpose

Normal forms are a sequence of increasingly strict constraints on a relation's design. Each normal form eliminates a specific type of redundancy or anomaly. The containment relationship is:

Hierarchy

BCNF  ⊂  3NF  ⊂  2NF  ⊂  1NF

Every relation in BCNF is necessarily in 3NF, 2NF, and 1NF. A relation in 3NF is in 2NF and 1NF, but not necessarily in BCNF. The converse of each containment does not hold.

When you're asked "what is the highest normal form this relation satisfies?" — test from the top down. Start with BCNF. If it fails, try 3NF. And so on.

1NF

First Normal Form

Every attribute must contain only atomic (indivisible) values.

A relation is in 1NF if every cell contains a single, atomic value. No attribute may hold a set, list, array, or composite value. This is the baseline — without 1NF, the relational model cannot function at all.

What violates 1NF?

  • Multi-valued attribute: A column Courses containing {DBMS, OS, CN} in a single cell.
  • Composite attribute: A column Address containing (Street, City, State) as a nested structure.
  • Repeating groups: Columns Course1, Course2, Course3 to represent multiple courses.

Violates 1NF:

RollNoCourses (VIOLATION)
101{DBMS, OS, CN}

Fixed (1NF): One row per course value.

RollNoCourse
101DBMS
101OS
101CN
GATE Note

Every relation in the formal relational model is assumed to be in 1NF by definition. GATE uses 1NF questions to test understanding of atomicity and to set up context for 2NF/3NF questions on the expanded table.

2NF

Second Normal Form

1NF + No non-prime attribute is partially dependent on any candidate key.

A relation is in 2NF if it is in 1NF and every non-prime attribute is fully functionally dependent on every candidate key. Equivalently: no non-prime attribute should depend on only part of a candidate key.

When can a partial dependency exist?

Only when a candidate key is composite (has two or more attributes). If an attribute depends on just one part of that composite key — but not the whole key — that is a partial dependency. A relation with a single-attribute candidate key cannot have partial dependencies and is therefore automatically in 2NF.

Relation: Enrollment(RollNo, CourseID, CourseName, Grade)

Candidate key: {RollNo, CourseID}

FDs:

{RollNo, CourseID} → Grade ← Full dependency on entire CK ✓ CourseID → CourseName ← Partial dep: non-prime CourseName depends only on CourseID (part of CK) ✗

The partial dependency CourseID → CourseName violates 2NF because CourseName is non-prime.

Fix — decompose into:

  • Enrollment(RollNo, CourseID, Grade) — the grade genuinely depends on both.
  • Course(CourseID, CourseName) — course name is a property of the course, not the enrollment.
Common Mistake

Students often check partial dependencies for prime attributes. 2NF only concerns non-prime attributes. A prime attribute can "partially depend" on a key — that does not violate 2NF. Only non-prime attributes participating in a partial dependency cause a 2NF violation.

3NF

Third Normal Form

2NF + No non-prime attribute is transitively dependent on any candidate key.

A relation is in 3NF if, for every non-trivial functional dependency X → Y in F⁺, at least one of the following conditions holds:

  • X is a superkey of the relation, OR
  • Y is a prime attribute (part of some candidate key).

The second condition — "Y is a prime attribute" — is sometimes called the "escape clause." It is the key difference between 3NF and BCNF. 3NF permits an FD where the LHS is not a superkey, as long as the RHS is a prime attribute.

Relation: Student(RollNo, Name, Dept, DeptHead)

CK: {RollNo}   |   Prime: RollNo   |   Non-prime: Name, Dept, DeptHead

Problem FD: Dept → DeptHead

Check: Is Dept a superkey? No — multiple students share a dept. Check: Is DeptHead a prime attribute? No — it's not in any CK. → Both conditions fail. 3NF is VIOLATED.

The transitive chain: RollNo → Dept → DeptHead

DeptHead is a fact about the department, not about the student. It's been placed in the wrong relation.

Fix:

  • Student(RollNo, Name, Dept)
  • Department(Dept, DeptHead)
Informal Test for 3NF

For every non-trivial FD X → Y, ask: "Is X something that uniquely identifies a row?" If yes, X is a superkey and you're fine. If no, ask: "Is Y part of a key?" If yes, fine. If no to both — you have a 3NF violation.

BCNF

Boyce-Codd Normal Form

For every non-trivial FD X → Y, X must be a superkey. No exceptions.

BCNF is a stricter version of 3NF. It removes the "escape clause" entirely. In BCNF, the only permissible non-trivial FDs are those where the LHS is a superkey. There is no exception for prime attributes on the RHS.

Every relation in BCNF is in 3NF, but not every relation in 3NF is in BCNF. The difference arises when a relation has multiple candidate keys that overlap (share attributes).

The classic BCNF-violating, 3NF-satisfying relation:

Teaching(Student, Course, Teacher)

Semantics: A teacher teaches exactly one course. A student-course pair has exactly one assigned teacher.

FDs: {Student, Course} → Teacher Teacher → Course Candidate Keys: {Student, Course}⁺ = {Student, Course, Teacher} ✓ {Student, Teacher}⁺ = {Student, Teacher, Course} ✓ → Two CKs: {Student, Course} and {Student, Teacher} All three attributes are PRIME (each appears in some CK). Check 3NF for Teacher → Course: Is Teacher a superkey? No. Is Course a prime attribute? YES (it's in CK {Student, Course}). → 3NF: SATISFIED (escape clause used). Check BCNF for Teacher → Course: Is Teacher a superkey? No. → BCNF: VIOLATED. No escape clause in BCNF.

This is the canonical example that every GATE student must memorize. A relation can be in 3NF but not in BCNF when all attributes are prime and a non-superkey FD exists.

The Critical Trade-off

Decomposing Teaching into BCNF-compliant relations may lose the functional dependency {Student, Course} → Teacher — it cannot be checked without a join. BCNF removes all redundancy but may sacrifice dependency preservation. This trade-off is a direct GATE question pattern.

Chapter 07
Decomposition — Lossless & Dependency Preserving

7.1   What Is Decomposition?

Normalization works by decomposing a relation R into two or more smaller relations R₁, R₂, ... such that R can be reconstructed from them (lossless) and the FDs are still enforceable (dependency-preserving). A decomposition that violates either property is useless or harmful.

Lossless Join Decomposition

Definition

A decomposition of R into R₁ and R₂ is lossless if R₁ ⋈ R₂ = R for every valid instance of R. No spurious tuples are generated when you join the decomposed relations back together.

Test for lossless decomposition (binary):

A decomposition R → {R₁, R₂} is lossless if and only if at least one of the following FDs holds:

(R₁ ∩ R₂) → (R₁ − R₂) — the common attributes determine R₁'s unique attributes OR (R₁ ∩ R₂) → (R₂ − R₁) — the common attributes determine R₂'s unique attributes

In other words, the attributes shared between R₁ and R₂ must form a superkey for at least one of them. If the intersection can "identify" rows in one of the relations, then joining them back will never produce spurious tuples.

Lossless ≠ No Data Loss

"Lossless" does not mean no data is lost. It means no information is lost — you can perfectly reconstruct the original relation. A "lossy" decomposition generates spurious (fake) tuples on natural join, meaning the join result has more rows than the original and contains wrong data.

Dependency-Preserving Decomposition

Definition

A decomposition is dependency-preserving if every functional dependency in F can be checked using only the data within a single decomposed relation — without needing to join relations back together.

Formally: let Fᵢ be the projection of F onto Rᵢ (only FDs whose attributes all appear in Rᵢ). The decomposition preserves dependencies if (F₁ ∪ F₂ ∪ ...)⁺ = F⁺.

3NF Synthesis Algorithm

Always produces a decomposition that is both lossless AND dependency-preserving. This is its primary advantage over BCNF.

BCNF Decomposition Algorithm

Always produces a lossless decomposition, but may not preserve all dependencies. Sometimes no BCNF decomposition exists that is also dependency-preserving.

Chapter 08
GATE Traps & Quick Revision

8.1   The Ten Traps That Kill GATE Marks

Trap 1

Inferring keys from current data. You see that every row has a unique Name in the current instance, and conclude Name is a candidate key. Wrong — future instances may have duplicate names. Keys are determined by the semantics of the problem, not the current data.

Trap 2

Forgetting the single-attribute shortcut. If every candidate key is a single attribute, the relation is automatically in 2NF. Many GATE questions have single-attribute CKs — skip 2NF analysis entirely and go straight to 3NF.

Trap 3

Applying 2NF rules to prime attributes. 2NF only concerns non-prime attributes. A prime attribute that partially depends on a key does not violate 2NF. Many students flag this incorrectly.

Trap 4

Confusing 3NF and BCNF on all-prime relations. When all attributes are prime, 3NF is automatically satisfied (the escape clause covers everything). But BCNF may still be violated if a non-superkey FD exists. The Teaching relation is the canonical trap here.

Trap 5

Assuming BCNF is always achievable with dependency preservation. It is not. Some relations cannot be decomposed to BCNF while preserving all FDs. When asked whether a BCNF decomposition exists that is also dependency-preserving, the answer may be no.

Trap 6

Treating canonical cover as unique. It is not. Multiple valid canonical covers may exist. Don't rule out an option just because it looks different from the one you derived.

Trap 7

NULL in FD computations. Attributes with NULL values behave differently in joins and aggregates. A key attribute cannot be NULL. Don't confuse "attribute can take value 0" with "attribute can be NULL."

Trap 8

Not finding ALL candidate keys before normalizing. If you miss a candidate key, you'll misclassify prime attributes, and your entire normalization analysis will be wrong. Always be exhaustive when finding CKs.

Trap 9

Confusing lossless with dependency-preserving. They are independent properties. A decomposition can be lossless but not dependency-preserving (BCNF case), or dependency-preserving but lossy (rare but possible). Both properties must be checked independently.

Trap 10

Forgetting that degree determines if a relation is in 1NF. A relation is not "more in 1NF" because it has more rows. 1NF is about atomicity of attribute values, not the number of tuples.

8.3   GATE PYQ Question Patterns

These are the question patterns that repeat in GATE DBMS year after year:

  • Find all candidate keys — given a relation schema and a set of FDs, compute closures and enumerate all CKs. Medium frequency, often 2 marks.
  • Determine highest NF — given a relation and FDs, state whether it's in 1NF/2NF/3NF/BCNF and justify. Very high frequency.
  • Is a decomposition lossless? — apply the intersection test. Test appears regularly, often combined with normalization.
  • Is a decomposition dependency-preserving? — check if the union of projected FD sets equals original F⁺. Often paired with BCNF decomposition.
  • Find canonical cover — compute Fc from a given F. Appears in questions about 3NF synthesis. Medium frequency.
  • 3NF vs BCNF trade-off — "Can this relation be decomposed to BCNF while preserving all dependencies?" The answer is sometimes explicitly no. Know the Teaching relation cold.
  • Count candidate keys — given n attributes and certain FDs, how many candidate keys exist? Combinatorial reasoning over closures.
  • Minimum number of relations in decomposition — given a relation in violation of some NF, what is the minimum decomposition that achieves the target NF?
Your GATE strategy for normalization questions

Step 1: Find all candidate keys (compute closures).
Step 2: Classify all attributes as prime or non-prime.
Step 3: Check each NF from the bottom up.
Step 4: For decomposition questions, verify lossless + dependency-preserving separately.
