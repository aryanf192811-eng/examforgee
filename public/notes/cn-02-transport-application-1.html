UDP — Connectionless Transport

1.1   UDP — Definition, Header & Use Cases Core

UDP — User Datagram Protocol

UDP provides a minimal, best-effort, connectionless transport service. No handshake, no state maintained between sender and receiver, no guaranteed delivery, no ordering, no congestion control. Just send datagrams — fast and simple.

Maximum UDP payload = 65535 − 8 = 65527 bytes. Checksum covers header + data + pseudo-header (src IP, dst IP, protocol, UDP length).

Why Applications Choose UDP

Applications that need speed over reliability (DNS — a lost query is just re-sent quickly), or that have their own reliability mechanism (HTTP/3 over QUIC), or that cannot tolerate retransmission delay (real-time audio/video — old data is useless), or that use multicast/broadcast (DHCP discovers the network by broadcasting). GATE frequently asks: "Which protocol uses UDP?" — know: DNS, DHCP, SNMP, RIP, TFTP, NTP, VoIP use UDP.

2.1   TCP Segment Structure GATE Tested

Sequence and Acknowledgment Numbers

TCP sequence numbers count bytes, not segments. If segment has SEQ=1000 and carries 500 bytes, the next segment starts at SEQ=1500. The ACK number = SEQ of the next expected byte. If receiver has received bytes 0–999, it sends ACK=1000 ("I want byte 1000 next").

2.2   Three-Way Handshake & Connection Teardown GATE Critical

Connection Establishment — 3-Way Handshake

Client Server │ │ │──── SYN (SEQ=x) ──────────────────→│ Step 1: Client sends SYN │ │ (x = Initial Sequence Number, ISN) │←─── SYN+ACK (SEQ=y, ACK=x+1) ─────│ Step 2: Server acknowledges client ISN │ │ and sends its own ISN y │──── ACK (SEQ=x+1, ACK=y+1) ───────→│ Step 3: Client acknowledges server ISN │ │ │══════ Connection Established ══════│ Why 3 steps (not 2)? 2-way handshake cannot guarantee both sides have acknowledged each other's ISNs. The third ACK confirms the server's SYN was received — making it truly bidirectional. SYN consumes 1 sequence number despite carrying no data. FIN also consumes 1 sequence number.

Connection Teardown — 4-Way Handshake (FIN)

Client Server │──── FIN (SEQ=u) ──────────────────→│ Client done sending │←─── ACK (ACK=u+1) ─────────────────│ Server acknowledges │ (Server may still send) │ │←─── FIN (SEQ=v) ───────────────────│ Server done sending │──── ACK (ACK=v+1) ────────────────→│ Client acknowledges │ │ Client waits 2×MSL (Maximum Segment Lifetime) in TIME_WAIT before fully closing. Ensures delayed packets don't interfere with a new connection on the same port pair.
TIME_WAIT State — Why It Exists

After the client sends its final ACK, it enters TIME_WAIT for 2×MSL (typically 60s–240s). This ensures: (1) the final ACK reaches the server — if it's lost, the server retransmits FIN and the client can re-send ACK; (2) old duplicate segments from this connection can't appear in a new connection on the same port pair and confuse the new TCP session.

Chapter 03
TCP Flow Control — Receiver Window & Sliding Window

3.1   TCP Flow Control — Receiver Window GATE Tested

Flow Control — What and Why

Flow control prevents a fast sender from overwhelming a slow receiver's buffer. TCP uses a receive window (rwnd) advertised in every ACK — it tells the sender how much buffer space is available at the receiver. The sender can have at most rwnd bytes of unacknowledged data in flight at any time.

Sender's Constraint

LastByteSent − LastByteAcked ≤ rwnd
The sender keeps track of the last byte sent and the last byte acknowledged. The difference (unacknowledged bytes in flight) must not exceed the receiver's window.

TCP flow control scenario:

Receiver buffer = 4096 bytes (rwnd initially = 4096) Step 1: Sender sends 1000 bytes. In-flight = 1000. Receiver has 3096 bytes free. ACK arrives: ACK=1001, rwnd=3096. Step 2: Sender sends 2000 more bytes. In-flight = 2000. Receiver has 1096 free. ACK arrives for 1000 bytes (bytes 1-1000 consumed by app). Now buffer has: 1096 + 1000 = 2096 bytes free. ACK=3001, rwnd=2096. If rwnd=0: Sender must STOP sending data. Sender sends 1-byte "window probe" periodically to check if window opened. When receiver opens window, it sends ACK with new rwnd > 0.
Silly Window Syndrome

If the receiver advertises a tiny window (e.g., rwnd=1) and the sender immediately fills it, many tiny segments waste bandwidth. Solutions: Nagle's algorithm (sender accumulates data until a full segment or ACK arrives before sending) and Clark's solution (receiver waits until it has enough buffer to advertise a useful window). GATE tests: "What is Nagle's algorithm?" — don't send tiny segments if there are unacknowledged bytes in flight.

RTT Estimation — Karn's Algorithm

Estimated RTT: ERTT = (1−α)×ERTT + α×SampleRTT (exponential weighted moving average, typically α=0.125).
DevRTT = (1−β)×DevRTT + β×|SampleRTT − ERTT| (deviation, β=0.25).
RTO = ERTT + 4×DevRTT.
Karn's algorithm: Don't update ERTT using retransmitted segments — you can't tell if an ACK is for the original or the retransmission.

Chapter 04
TCP Congestion Control — Slow Start, AIMD & CUBIC

4.1   Congestion Control — Core Concepts GATE Critical

Congestion vs Flow Control

Flow control: Prevents sender from overwhelming the receiver's buffer. Controlled by rwnd (receiver-set).
Congestion control: Prevents sender from overwhelming the network. Controlled by cwnd (congestion window — sender-maintained). The actual sending rate is limited by min(rwnd, cwnd).

TCP infers congestion from two signals: timeout (severe — packet lost, network badly congested) and 3 duplicate ACKs (mild — one packet lost but network still delivering others).

4.2   Slow Start, Congestion Avoidance & Fast Retransmit Most Tested

TCP Reno Congestion Control Phases

Slow Start (SS): cwnd starts at 1 MSS. Doubles every RTT (cwnd += 1 for each ACK received). Exponential growth. Continues until cwnd ≥ ssthresh or congestion detected.

Congestion Avoidance (CA): cwnd increases by 1 MSS per RTT (linear growth). cwnd += MSS×MSS/cwnd for each ACK (≈ +1 MSS per RTT).

Fast Retransmit: On 3 duplicate ACKs, immediately retransmit the missing segment (don't wait for timeout).

Fast Recovery: After fast retransmit, set ssthresh = cwnd/2, cwnd = ssthresh + 3 (inflate window), enter CA — NOT slow start. This is the key difference between TCP Reno and TCP Tahoe.

TCP Reno congestion control trace:

Initial: cwnd=1 MSS, ssthresh=8 MSS RTT 1: Slow Start. cwnd=1. Send 1 segment. ACK received. cwnd=2. RTT 2: cwnd=2. Send 2. ACK×2. cwnd=4. RTT 3: cwnd=4. cwnd=8. (Hit ssthresh=8) → switch to CA. RTT 4: CA. cwnd=9. RTT 5: cwnd=10. RTT 6: cwnd=11. RTT 7: cwnd=12. LOSS DETECTED (timeout): ssthresh = cwnd/2 = 6 cwnd = 1 → back to Slow Start (TCP Reno/Tahoe both do this on timeout) RTT 8: SS. cwnd=2. RTT 9: cwnd=4. RTT 10: cwnd=6 (=ssthresh) → switch to CA. RTT 11: cwnd=7. RTT 12: cwnd=8. LOSS (3 dup ACKs, TCP Reno): ssthresh = cwnd/2 = 4 cwnd = ssthresh + 3 = 7 → Fast Recovery (NOT SS) ← Reno advantage Tahoe (older): always sets cwnd=1 on any loss → always goes to SS. Reno: timeout → cwnd=1 (SS). 3 dup ACKs → cwnd=ssthresh+3 (Fast Recovery).

AIMD — Additive Increase Multiplicative Decrease

TCP's congestion avoidance follows the AIMD principle: add 1 MSS per RTT when no congestion (additive increase), multiply by 0.5 when congestion detected (multiplicative decrease). AIMD is provably optimal for fairness — multiple TCP flows sharing a bottleneck link will converge to equal shares. This is the mathematical backbone of TCP's fairness property.

Classic GATE NAT: TCP window size after k RTTs in slow start.

cwnd after k RTTs in Slow Start (starting from cwnd=1, no ssthresh hit): cwnd = 2ᵏ MSS After k RTTs from ssthresh crossing (in CA): cwnd = ssthresh + k MSS "TCP connection uses MSS=1KB. ssthresh=8KB. After how many RTTs does cwnd reach 16KB starting from cwnd=1KB?" Slow Start: cwnd = 1,2,4,8 → 3 RTTs to reach ssthresh=8KB. CA from RTT 4: cwnd = 9,10,11,...,16 → 8 more RTTs. Total: 3 + 8 = 11 RTTs.

DNS — Domain Name System

5.1   DNS — Hierarchy, Resolution & Record Types GATE Tested

DNS

DNS is a distributed, hierarchical database that maps domain names (human-readable) to IP addresses (machine-readable). It is the internet's phone book. DNS uses UDP port 53 for queries (fast), TCP port 53 for zone transfers (large data, reliability needed).

DNS Hierarchy Root DNS Servers (13 clusters, e.g., a.root-servers.net) / | \ .com .org .edu / amazon.com google.com ... / www.amazon.com mail.amazon.com ... Resolution order: 1. Check browser cache 2. Check OS cache (/etc/hosts, OS DNS cache) 3. Query local DNS resolver (usually ISP's or 8.8.8.8) 4. Local resolver queries root → TLD → authoritative server

Iterative vs Recursive Resolution

TypeWho does the work?How it works
RecursiveDNS resolver does all workClient asks resolver "what is IP of X?" Resolver asks root → TLD → authoritative, collects answer, returns final IP to client.
IterativeClient does the workClient asks root: gets referral to TLD. Client asks TLD: gets referral to authoritative. Client asks authoritative: gets IP. Client assembles answer.

In practice: client→local resolver is recursive; local resolver→root/TLD/authoritative is iterative.

DNS Caching & TTL

Every DNS response includes a TTL (Time to Live) specifying how long the record can be cached. A short TTL (e.g., 60s) means every request hits the DNS server — flexible for changing IPs but high load. A long TTL (e.g., 86400s = 1 day) reduces DNS load but makes IP changes slow to propagate. GATE: "What field controls how long a DNS record is cached?" → TTL.

6.1   HTTP — Versions & Persistent Connections GATE Tested

HTTP — HyperText Transfer Protocol

HTTP is a stateless, request-response application layer protocol. Client sends request; server sends response. Uses TCP (port 80 for HTTP, 443 for HTTPS). HTTP itself is stateless — cookies are used to maintain state across requests.

RTT analysis — fetching a web page with 10 objects using HTTP/1.0 vs HTTP/1.1:

Assume each TCP setup = 1 RTT, each HTTP request = 1 RTT. Objects: 1 base HTML + 10 embedded images. HTTP/1.0 (non-persistent): For each of 11 objects: 1 RTT (TCP SYN+SYN-ACK) + 1 RTT (request+response) Total = 11 × 2 RTT = 22 RTT (Can be parallelized with multiple connections) HTTP/1.1 (persistent, no pipelining): 1 RTT for TCP handshake (reused for all) 11 RTT for 11 sequential requests Total = 1 + 11 = 12 RTT HTTP/1.1 (persistent, pipelining): 1 RTT for TCP handshake 1 RTT for base HTML 1 RTT for all 10 images (pipelined) Total = 1 + 1 + 1 = 3 RTT (ignoring transmission time)

6.2   HTTPS & TLS GATE Tested

TLS Handshake (simplified)

1. ClientHello: Client sends supported TLS versions, cipher suites, random nonce.
2. ServerHello: Server selects cipher suite, sends its certificate (public key).
3. Key Exchange: Client generates pre-master secret, encrypts with server's public key, sends.
4. Session Keys: Both derive symmetric session keys from pre-master secret + nonces.
5. Finished: Both send encrypted "Finished" messages. Secure channel established.

TLS provides: Confidentiality (symmetric encryption with session keys), Integrity (MAC — Message Authentication Code), Authentication (server certificate verified against trusted CA). TLS runs above TCP (port 443) but below HTTP.

SMTP Only Pushes — It Cannot Pull

SMTP is a push protocol — the sender initiates the connection and pushes mail to the recipient's server. It cannot be used by a client to retrieve mail (that's POP3/IMAP). When GATE asks "which protocol does a mail client use to READ email?" — the answer is POP3 or IMAP, never SMTP.

7.2   FTP — Two Connections, Active vs Passive GATE Tested

FTP — File Transfer Protocol

FTP uses two separate TCP connections: a control connection (port 21, persistent, sends commands) and a data connection (port 20 for active mode, dynamic for passive mode, established per file transfer). FTP is stateful — the server maintains state about the user's current directory and transfer mode.

7.3   DHCP — Dynamic Host Configuration Protocol GATE Tested

DHCP

DHCP automatically assigns IP addresses, subnet mask, default gateway, and DNS server addresses to hosts when they join a network. Uses UDP (port 67 server, 68 client). DHCP messages are broadcast — DHCP server must be on the same broadcast domain or accessible via DHCP relay agent.

DHCP DORA Process Client DHCP Server │ │ │── DHCP Discover (broadcast) ─────────────→│ "Anyone have an IP for me?" │ │ │←─ DHCP Offer (broadcast/unicast) ─────────│ "I can give you 192.168.1.10" │ │ │── DHCP Request (broadcast) ──────────────→│ "I'll take 192.168.1.10, please" │ │ │←─ DHCP ACK (broadcast/unicast) ───────────│ "Confirmed. Lease = 24 hours." │ │ Assigned: IP, subnet mask, gateway, DNS, lease time DORA = Discover, Offer, Request, ACK
DHCP Relay Agent

DHCP clients send broadcasts, but routers don't forward broadcasts. A DHCP relay agent (running on the router) receives the local broadcast, unicasts it to the DHCP server, and relays the response back. This allows one DHCP server to serve multiple subnets. GATE tests: "If client and DHCP server are on different subnets, what is needed?" → DHCP relay agent.

8.1   All the Traps in One Place

#TrapCorrect Understanding
1TCP SYN consumes a sequence numberSYN (and FIN) each consume one sequence number even though they carry no data. After SYN with SEQ=x, next byte is x+1. This is why the handshake ACKs are x+1 and y+1, not x and y.
2ACK number = last byte receivedACK number = last byte received + 1 = NEXT byte expected. If receiver got bytes 0-999, it sends ACK=1000 (not 999). TCP ACKs are cumulative and specify the next expected byte.
3Flow control = congestion controlFlow control is receiver-to-sender: "don't overflow my buffer" (rwnd). Congestion control is network-to-sender: "don't overflow the network" (cwnd). Both limit sending rate; min(rwnd, cwnd) is the effective window.
4Slow Start is slow"Slow Start" is slow only in name — it actually grows EXPONENTIALLY (doubling each RTT). It's called "slow" because it starts at 1 MSS instead of the full bandwidth. Congestion Avoidance is the linear (slow) phase.
5TCP Reno always resets to SS on any lossTCP Reno ONLY resets to Slow Start (cwnd=1) on TIMEOUT. On 3 duplicate ACKs, Reno uses Fast Recovery: ssthresh=cwnd/2, cwnd=ssthresh+3, then CA. Tahoe always resets to SS regardless of loss type.
6DNS uses TCP onlyDNS primarily uses UDP port 53 (fast queries). TCP port 53 is used for zone transfers between DNS servers (large data requiring reliability). Most GATE questions about DNS port → UDP 53.
7HTTP is statefulHTTP itself is STATELESS — each request is independent, server has no memory of prior requests. Cookies, sessions, and tokens are used at the application level to implement statefulness on top of stateless HTTP.
8SMTP retrieves emailSMTP only SENDS mail (push). Email retrieval uses POP3 (download and delete) or IMAP (server-side management, sync). GATE: "Which protocol does a client use to download email from a server?" → POP3 or IMAP.
9FTP uses one connectionFTP uses TWO TCP connections: one control connection (port 21, persistent) and one data connection (port 20 active / dynamic passive) per file transfer. HTTP/SMTP/POP3 each use one connection.
10DHCP uses TCPDHCP uses UDP (ports 67/68) because it broadcasts — and broadcasting is inherently unreliable. TCP requires a pre-established connection, which is impossible when the client doesn't yet have an IP address.
11cwnd in slow start grows by 1 each RTTcwnd doubles each RTT in Slow Start (exponential). cwnd grows by 1 MSS per RTT in Congestion Avoidance (linear). After k RTTs in Slow Start from cwnd=1: cwnd = 2ᵏ MSS.
12TIME_WAIT is on the passive closerTIME_WAIT is entered by the ACTIVE closer — the side that sent the first FIN. It waits 2×MSL to ensure all old segments expire. The passive closer (server side if client closes) goes through CLOSE_WAIT → LAST_ACK → CLOSED.

Chapter 09
Network Security — Cryptography, Signatures & Firewalls

9.1   Foundations of Cryptography Core

Cryptography is the science of securing communication in the presence of adversaries. It provides four security properties: Confidentiality (only intended recipient can read), Integrity (message was not altered), Authentication (sender is who they claim to be), and Non-repudiation (sender cannot deny sending). Different mechanisms address different subsets of these properties.

Symmetric Key Cryptography (Secret Key / Private Key)

The same secret key K is used for both encryption and decryption. Sender computes C = E(K, M). Receiver computes M = D(K, C). Both parties must share K beforehand through a secure channel.

Key algorithms:
DES (Data Encryption Standard) — 56-bit key, 64-bit block. Considered broken today (brute-forceable in hours).
3DES (Triple DES) — Applies DES three times with different keys. 112–168 bit effective security. Slower.
AES (Advanced Encryption Standard) — 128/192/256-bit keys, 128-bit block. Current standard. Fast in hardware and software.
RC4 — Stream cipher (byte by byte), used in WEP/WPA (now deprecated).

Asymmetric Key Cryptography (Public Key Cryptography)

Each entity has a key pair: a public key (shared openly) and a private key (kept secret). What one key encrypts, only the other can decrypt. This eliminates the key distribution problem of symmetric cryptography.

Key property: Given a public key, it is computationally infeasible to derive the private key (one-way function with trapdoor).
Key algorithm: RSA, Diffie-Hellman, ElGamal, ECC (Elliptic Curve Cryptography).

Hybrid Cryptography — Best of Both Worlds

Real systems (TLS, PGP, SSH) use hybrid encryption: asymmetric cryptography to securely exchange a randomly generated symmetric session key, then symmetric cryptography to encrypt the actual data. Asymmetric = key transport. Symmetric = data transport. This gives security of public key + speed of symmetric. TLS does exactly this — RSA or ECDH for key exchange, AES for data.

9.2   The RSA Algorithm — Full Derivation GATE Critical

RSA — Mathematical Foundation

RSA security rests on the integer factorization problem: multiplying two large primes is easy, but factoring the product is computationally infeasible for large enough primes. Given n = p×q (public), deriving p and q individually is intractable.

RSA Key Generation — 5 Steps

Step 1: Choose two large distinct primes p and q.
Step 2: Compute n = p × q. (n is the modulus — public.)
Step 3: Compute φ(n) = (p−1)(q−1). (Euler's totient — private.)
Step 4: Choose e such that 1 < e < φ(n) and gcd(e, φ(n)) = 1. (e is the public exponent.)
Step 5: Find d = e⁻¹ mod φ(n) — the modular multiplicative inverse of e. (d is the private exponent.)

Public Key: (e, n)
Private Key: (d, n)

RSA Encryption & Decryption

Encryption: C = Mᵉ mod n    (encrypt with receiver's public key)
Decryption: M = Cᵈ mod n    (decrypt with receiver's private key)

Complete RSA worked example (small primes for illustration):

Step 1: Choose p=61, q=53. Step 2: n = 61 × 53 = 3233. Step 3: φ(n) = (61−1)(53−1) = 60 × 52 = 3120. Step 4: Choose e=17. Check: gcd(17, 3120) = 1. ✓ (17 is prime, doesn't divide 3120) Step 5: Find d such that 17d ≡ 1 (mod 3120). Using Extended Euclidean Algorithm: 3120 = 183×17 + 9 17 = 1×9 + 8 9 = 1×8 + 1 8 = 8×1 + 0 Back-substitute: 1 = 9 − 1×8 = 9 − 1×(17 − 9) = 2×9 − 17 = 2×(3120 − 183×17) − 17 = 2×3120 − 367×17 d = −367 mod 3120 = 3120 − 367 = 2753 Public Key: (e=17, n=3233) Private Key: (d=2753, n=3233) Encrypt M=65 (must have M < n): C = 65¹⁷ mod 3233 = 2790 (using repeated squaring) Decrypt C=2790: M = 2790²⁷⁵³ mod 3233 = 65 ✓

Repeated Squaring — computing Mᵉ mod n efficiently:

To compute 65¹⁷ mod 3233: 17 in binary = 10001 65¹ mod 3233 = 65 65² mod 3233 = 4225 mod 3233 = 992 65⁴ mod 3233 = 992² mod 3233 = 984064 mod 3233 = 1097 65⁸ mod 3233 = 1097² mod 3233 = 1203409 mod 3233 = 1874 65¹⁶ mod 3233 = 1874² mod 3233 = 3511876 mod 3233 = 2827 65¹⁷ = 65¹⁶ × 65¹ = 2827 × 65 mod 3233 = 183755 mod 3233 = 2790 ✓ This is O(log e) multiplications — essential for large exponents.
GATE Trap — d is inverse mod φ(n), NOT mod n

The private exponent d satisfies: e × d ≡ 1 (mod φ(n)). The modulus for the inverse computation is φ(n) = (p−1)(q−1), not n. Using n instead of φ(n) gives a completely wrong d. This is the single most common calculation error in RSA NAT questions.

Diffie-Hellman Key Exchange

Diffie-Hellman — Secure Key Agreement Without Sharing a Secret

DH allows two parties to agree on a shared secret over a public channel without ever transmitting the secret itself. Security rests on the discrete logarithm problem.

Setup (public): Large prime g and generator p.
Alice: Chooses private aₐ, computes Aₚ = gᵃ mod p. Sends Aₚ.
Bob: Chooses private bᵦ, computes Bₚ = gᵇ mod p. Sends Bₚ.
Shared secret: Alice computes Bₚᵃ mod p. Bob computes Aₚᵇ mod p. Both = gᵃᵇ mod p.
An eavesdropper sees g, p, gᵃ, gᵇ but cannot compute gᵃᵇ without solving discrete log.

DH is Vulnerable to Man-in-the-Middle — Authentication Needed

DH provides key agreement but NOT authentication. An attacker Mallory can intercept: perform DH with Alice (posing as Bob) and DH with Bob (posing as Alice) separately. Both Alice and Bob think they share a key with each other, but Mallory controls both sessions. DH must be combined with digital certificates (public key infrastructure) to prevent MITM attacks. TLS does this — DH for key exchange + X.509 certificate for authentication.

9.3   Hash Functions & Digital Signatures GATE Tested

Cryptographic Hash Functions

Hash Function Properties

A cryptographic hash function H maps arbitrary-length input to a fixed-length digest (hash). Required properties:
1. Deterministic: Same input always gives same output.
2. Efficient: Fast to compute H(M) for any M.
3. One-way (Pre-image resistance): Given h, computationally infeasible to find M such that H(M) = h.
4. Second pre-image resistance: Given M₁, infeasible to find M₂ ≠ M₁ with H(M₂) = H(M₁).
5. Collision resistance: Infeasible to find any M₁ ≠ M₂ with H(M₁) = H(M₂).

AlgorithmOutput SizeStatusUse
MD5128 bitsBroken — collision attacks practicalLegacy, checksums only
SHA-1160 bitsBroken — collision attacks demonstrated (2017)Deprecated
SHA-256256 bitsSecure — current standardTLS, Bitcoin, code signing
SHA-3224/256/384/512 bitsSecure — different internal design from SHA-2Modern systems
Birthday Attack — Why Hash Size Matters

If a hash has n bits, a brute-force collision attack requires approximately 2^(n/2) operations (birthday paradox — not 2ⁿ). For MD5 (128 bits): 2⁶⁴ operations — feasible on modern hardware. For SHA-256 (256 bits): 2¹²⁸ operations — computationally infeasible. This is why hash output must be at least 256 bits for modern security.

Message Authentication Code (MAC)

MAC — Shared-Key Integrity

MAC = H(K || M) — a hash computed with a shared secret key K. Provides integrity and authentication (only parties knowing K can generate a valid MAC). Does NOT provide non-repudiation (either party could have generated it). Used in: TLS record layer (HMAC-SHA256), WPA2 (TKIP-MIC).

Digital Signatures — Authentication + Integrity + Non-repudiation

Digital Signature — How It Works

Signing (sender uses their PRIVATE key):
1. Compute hash of message: h = H(M)
2. Encrypt hash with private key: S = E(K_priv, h) — this is the signature
3. Send (M, S) together

Verification (receiver uses sender's PUBLIC key):
1. Compute hash of received message: h' = H(M)
2. Decrypt signature: h = D(K_pub, S)
3. Compare: if h == h', signature is valid

Provides: Authentication (only private key holder could have signed), Integrity (any change to M changes h'), Non-repudiation (sender cannot deny — only they have the private key).

Digital Signature — Complete Flow Alice wants to send M to Bob, proving it's from her and unmodified. ALICE (signing): BOB (verifying): M = "Send $100 to Carol" Receive (M, S) h = SHA256(M) h' = SHA256(M) S = RSA_encrypt(K_Alice_priv, h) h = RSA_decrypt(K_Alice_pub, S) Send (M, S) ──────────────────→ if h == h': VALID ✓ else: TAMPERED ✗ Anyone with Alice's public key can verify. Only Alice (with her private key) can sign.
Encrypt vs Sign — Direction of Key Use

Encrypt for confidentiality: Use receiver's PUBLIC key to encrypt → only receiver's private key can decrypt.
Sign for authentication: Use sender's PRIVATE key to sign → anyone with sender's public key can verify.
These are opposite operations. GATE directly tests: "Alice wants to send a message that only Bob can read. Which key does she use?" → Bob's public key. "Alice wants to prove she sent the message. What does she use?" → Her own private key. Getting these backwards is the classic exam error.

9.4   Public Key Infrastructure (PKI) & Certificates GATE Tested

The Key Distribution Problem in Public Key Cryptography

Public key cryptography eliminates the secret key distribution problem — but creates a new one: how do you know that a public key actually belongs to who it claims to belong to? An attacker (Mallory) could publish a key claiming to be Alice. Anyone using Mallory's key thinking it's Alice's sends secrets to Mallory.

Digital Certificate (X.509)

A digital certificate binds a public key to an identity (name, organization, domain). It contains: subject's public key, subject's identity (CN, O, etc.), validity period, issuer's name, and the CA's digital signature over all of this. The certificate is trusted because it is signed by a trusted Certificate Authority (CA).

Certificate Authority (CA) — The Trust Anchor

A CA is a trusted third party that verifies identities and issues digital certificates. Root CAs (DigiCert, Let's Encrypt, GlobalSign) are pre-installed in browsers and OS. The chain of trust: Root CA → Intermediate CA → End-entity certificate. Browsers verify the chain up to a trusted root CA.

Certificate Verification Chain Browser trusts: Root CA (pre-installed) ↓ (Root CA signs) Intermediate CA certificate ↓ (Intermediate CA signs) Website certificate (contains site's public key) ↓ (Site uses private key for TLS) Encrypted connection to user Certificate contains: Subject: CN=www.example.com Public Key: (e=65537, n=very_large_number) Issuer: DigiCert Global Root CA Valid: 2024-01-01 to 2025-01-01 Signature: CA's digital signature (verifiable with CA's public key)

9.5   Firewalls — Types & Operation GATE Tested

Firewall

A firewall is a network security system that monitors and controls incoming and outgoing network traffic based on predefined security rules. It creates a barrier between a trusted internal network and an untrusted external network (internet).

Firewall TypeOSI LayerExaminesStatePros/Cons
Packet FilterNetwork (L3) / Transport (L4)IP src/dst, ports, protocolStatelessFast, simple. Cannot detect spoofed packets or track connections.
Stateful InspectionTransport (L4)Packet headers + connection stateStatefulTracks TCP connections. Detects invalid packets not part of established sessions. Modern standard.
Application Layer ProxyApplication (L7)Full application contentStatefulUnderstands HTTP, FTP, DNS content. Most secure. Slowest — full content inspection.
Next-Generation Firewall (NGFW)L3–L7Everything above + DPI, IDS, app-idStatefulDeep packet inspection, application identification, TLS inspection. Current enterprise standard.

Demilitarized Zone (DMZ)

A DMZ is a network segment between the external internet and the internal trusted network. Servers that must be publicly accessible (web servers, mail servers, DNS) are placed in the DMZ. Two firewalls isolate it: external firewall between internet and DMZ, internal firewall between DMZ and internal LAN. Even if a DMZ server is compromised, the internal network is still protected by the inner firewall.

DMZ Architecture Internet │ ▼ [External Firewall] ← allows HTTP/HTTPS/SMTP inbound │ blocks everything else ├──→ [DMZ: Web Server, Mail Server, DNS] │ ▼ [Internal Firewall] ← allows only specific internal traffic │ ▼ [Internal Network: Databases, Workstations, File Servers]

IPsec Modes — Transport vs Tunnel

IPsec Transport Mode: Encrypts/authenticates only the payload of the IP packet. Original IP header is preserved. Used for end-to-end communication between two hosts.

Original packet: [ IP Header | TCP Header | Data ] Transport mode: [ IP Header | IPsec Header | Encrypted(TCP Header + Data) ] ↑ unchanged ↑ new ↑ protected

IPsec Tunnel Mode: Encrypts/authenticates the entire original IP packet, then adds a new outer IP header. Used in VPNs — gateway-to-gateway tunnels. The original source/destination IPs are hidden.

Original packet: [ IP Header (A→B) | TCP Header | Data ] Tunnel mode: [ New IP Header (GW1→GW2) | IPsec Header | Encrypted(IP Header(A→B) + TCP + Data) ] ↑ new outer ↑ new ↑ entire original packet encrypted
