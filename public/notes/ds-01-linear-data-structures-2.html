Chapter 01
Arrays — Memory, 2D Formulae & Sparse Representation

1.1   What Is an Array? Core

An array is the most fundamental data structure — a contiguous block of memory holding elements of the same data type. The word "contiguous" is the entire story of why arrays behave the way they do. Because elements are adjacent in memory, the location of any element can be computed directly from the base address and the index. This is what gives arrays their O(1) random access.

Every other property of arrays — their strength (fast access), their weakness (fixed size, expensive insertion), their address formulae — flows directly from this one fact: elements are stored one after another in memory with no gaps.

1D Array: int A[5], base address = 1000, each int = 4 bytes Index: [ 0 ] [ 1 ] [ 2 ] [ 3 ] [ 4 ] Value: [ 10 ] [ 20 ] [ 30 ] [ 40 ] [ 50 ] Address: 1000 1004 1008 1012 1016
Address Formula — 1D Array

Address of A[i] = Base + (i − Lower_Bound) × Size_of_element

If the array is 0-indexed: Address(A[i]) = Base + i × w where w is the element size in bytes. If it is 1-indexed: Address(A[i]) = Base + (i−1) × w.

Given: int A[10], Base = 2000, size of int = 2 bytes. Find address of A[6].

Solution: Address = 2000 + 6 × 2 = 2012. (0-indexed)

If 1-indexed: Address = 2000 + (6−1) × 2 = 2000 + 10 = 2010.

Always check: is the array 0-indexed or 1-indexed? GATE specifies this — read carefully.

1.2   2D Arrays — Row Major vs Column Major Frequently Tested

A 2D array A[m][n] has m rows and n columns. In memory, it must be linearized — stored as a sequence of 1D elements. There are two conventions for how to do this:

Row Major Order

Entire first row stored, then entire second row, and so on. Used by C, C++, Java (for arrays). Elements in the same row are adjacent in memory.

Column Major Order

Entire first column stored, then entire second column, and so on. Used by Fortran, MATLAB. Elements in the same column are adjacent in memory.

Address Formula — 2D Array, 0-indexed

Row Major:   Address(A[i][j]) = Base + (i × n + j) × w
Column Major: Address(A[i][j]) = Base + (j × m + i) × w
where n = number of columns, m = number of rows, w = element size

Address Formula — 2D Array, 1-indexed [L1..U1][L2..U2]

Row Major:   Base + [(i−L1)(U2−L2+1) + (j−L2)] × w
Column Major: Base + [(j−L2)(U1−L1+1) + (i−L1)] × w

Given: A[1..4][1..5], Row Major, Base = 100, w = 2 bytes. Find address of A[3][4].

Number of columns = 5−1+1 = 5. Number of rows = 4.

Address = 100 + [(3−1)×5 + (4−1)] × 2 = 100 + [10+3] × 2 = 100 + 26 = 126

GATE Trap — Row vs Column Major

GATE questions will give you a storage order and ask for the address, or give you an address and ask which element it belongs to. Memorize both formulas. The most common mistake is using m (rows) where n (columns) belongs in the row-major formula. Write it out — never do it in your head.

Why Insertion is Expensive

To insert at position i in an array of n elements, all elements from position i to n−1 must shift one position to the right to make room. In the worst case (insert at beginning), every element moves. This is the fundamental limitation arrays share versus linked lists.

1.4   What Is a Sparse Matrix? Core

A sparse matrix is a 2D matrix in which the majority of elements are zero. There is no strict threshold, but in practice a matrix is considered sparse when the number of non-zero elements is significantly smaller than m × n — often less than 10% of the total entries.

Storing a sparse matrix as a full 2D array is enormously wasteful. A 1000×1000 matrix with only 500 non-zero entries would occupy 1,000,000 memory locations just to store 500 meaningful values. The solution is to store only the non-zero elements along with their positions.

Example Sparse Matrix (4×5) — only 4 non-zero values out of 20 0 0 3 0 0 0 0 0 0 0 0 2 0 0 0 0 0 0 7 0 ↑ Storing this as a full array wastes 16 slots for zeros.

1.5   Triplet (Coordinate) Representation GATE Tested

The simplest compact representation. Store only the non-zero elements as a list of (row, column, value) triplets — plus the matrix dimensions (total rows, total columns, total non-zero count) in a header row.

Sparse matrix above → Triplet representation:

RowColValueNotes
454Header: 4 rows, 5 cols, 4 non-zeros
023A[0][2] = 3
212A[2][1] = 2
337A[3][3] = 7

Space used: 3 × (non-zero count + 1) integers = 3 × 4 = 12 values instead of 4 × 5 = 20. As the matrix gets sparser, the saving grows dramatically.

Advantages

Simple to implement. Space = O(k) where k = non-zero count. Easy to add or remove non-zero elements.

Disadvantages

Random access to A[i][j] requires searching the triplet list — O(k). Not cache-friendly for row-wise traversal. Triplets must be kept sorted for efficient operations.

1.6   Compressed Sparse Row (CSR) Advanced

CSR is the industry-standard format for sparse matrices in high-performance computing, machine learning libraries (SciPy, TensorFlow), and graph algorithms. It optimizes for row-wise traversal — iterating over all non-zero elements in a given row becomes O(non-zeros in that row), not O(total non-zeros).

CSR — Three Arrays

values[] — All non-zero values, stored row by row, left to right.
col_idx[] — Column index for each corresponding value in values[].
row_ptr[] — For each row i, row_ptr[i] = index in values[] where row i begins. row_ptr[num_rows] = total non-zeros.

Build CSR for the sparse matrix above:

Matrix: Row 0: A[0][2]=3 Row 1: (empty) Row 2: A[2][1]=2 Row 3: A[3][3]=7 values = [ 3, 2, 7 ] ← all non-zeros, row by row col_idx = [ 2, 1, 3 ] ← their column positions row_ptr = [ 0, 1, 1, 2, 3 ] ← row 0 starts at index 0 row 1 starts at index 1 (no entries → same as row 2) row 2 starts at index 1 row 3 starts at index 2 sentinel: total non-zeros = 3 To find all non-zeros in row i: → iterate values[row_ptr[i]] to values[row_ptr[i+1] − 1]

Row 2's non-zeros: values[row_ptr[2]] to values[row_ptr[3]−1] = values[1] to values[1] = {2}. Correct.

Why CSR Matters Beyond GATE

PageRank, graph neural networks, finite element methods, recommendation systems — all operate on matrices that are 99%+ sparse. CSR (and its column-wise cousin CSC) is why these computations are feasible at scale. Knowing the data structure is knowing why Google's algorithm can run on a billion-node graph.

Chapter 02
Linked Lists — All Variants

2.1   What Is a Linked List? Core

A linked list is a linear data structure where elements are stored in non-contiguous memory locations. Each element (called a node) stores two things: the data itself, and a pointer to the next node. The pointer is what links the nodes together into a sequence, even though they may be scattered across memory.

This is the opposite of an array. An array's power comes from contiguity. A linked list's power comes from the absence of contiguity — nodes can be allocated and freed independently, making insertion and deletion cheap.

Singly Linked List: 10 → 20 → 30 → NULL HEAD ↓ [10 | *]──→[20 | *]──→[30 | NULL] ↑ ↑ ↑ data next data next data next
Node Structure

Each node contains: (1) Data field — the actual value stored. (2) Pointer field — the memory address of the next node. In a singly linked list, each node has one pointer. In a doubly linked list, each node has two pointers (prev and next).

The HEAD pointer holds the address of the first node. It is the entry point to the entire list. If HEAD is NULL, the list is empty. The last node's next pointer is NULL — this is how you detect the end of the list.

2.2   Types of Linked Lists

Singly Linked List (SLL)

Each node has one pointer — pointing to the next node. Traversal is one-directional: you can only move forward. To find the previous node, you must start from HEAD and traverse. This is the simplest form and most memory-efficient.

Doubly Linked List (DLL)

Each node has two pointers — one pointing to the next node, one pointing to the previous node. Traversal is bidirectional. Deletion of a node is easier than SLL (no need to find the previous node separately) but each node uses more memory.

Doubly Linked List NULL←[10|*|*]↔[20|*|*]↔[30|*|*]→NULL prev next prev next prev next

Circular Linked List

The last node's next pointer points back to the HEAD (first node) instead of NULL. This creates a circular structure with no "end." Useful for round-robin scheduling, circular buffers. Can be singly or doubly circular.

Circular Singly Linked List ┌─────────────────────────┐ ↓ | [10|*]──→[20|*]──→[30|*]──→┘
Detecting End of Circular List

In a circular linked list, you cannot check for NULL to detect the end — because the last pointer doesn't point to NULL. Instead, stop traversal when you return to the starting node (HEAD). GATE has directly tested this: "how do you know when traversal is complete?"

2.3   Operations on Linked Lists GATE Tested

Insertion

Insertion requires updating pointer(s) only — no shifting. The key is doing the pointer updates in the correct order.

Insert node NEW after node P in a singly linked list:

Step 1: NEW.next = P.next // NEW points to what P used to point to Step 2: P.next = NEW // P now points to NEW

Order matters critically. If you do Step 2 first, you lose the reference to the rest of the list — it becomes unreachable and effectively deleted.

Insert at beginning (new HEAD):

Step 1: NEW.next = HEAD // NEW points to current first node Step 2: HEAD = NEW // HEAD now points to NEW

Deletion

Delete node Q where P is Q's predecessor:

P.next = Q.next // Bypass Q entirely free(Q) // Release Q's memory

In a singly linked list, to delete a node you need its predecessor's pointer. This is why SLL deletion requires either knowing the previous node, or traversing from HEAD to find it — O(n) in the worst case.

DLL Advantage in Deletion

In a doubly linked list, every node has a prev pointer. To delete node Q, you do: Q.prev.next = Q.next and Q.next.prev = Q.prev. No need to find the predecessor separately. This makes deletion O(1) if you have the node's address — a key advantage over SLL.

The O(1) vs O(n) Insertion Distinction

Linked list insertion is O(1) only if you already have a reference to the insertion point. If you must first search for the position (e.g., "insert after the node with value 30"), the search itself is O(n). GATE questions will distinguish between "given a pointer to node P" (O(1) insertion) and "insert at position k" (O(n) traversal + O(1) insert).

2.4   Classic Linked List Problems GATE Algorithms

Detecting a Cycle — Floyd's Algorithm

Floyd's Cycle Detection (also called the tortoise and hare algorithm) uses two pointers: slow moves one step at a time, fast moves two steps at a time. If a cycle exists, fast will eventually lap slow and they will meet inside the cycle. If no cycle, fast reaches NULL.

slow = HEAD, fast = HEAD while fast ≠ NULL and fast.next ≠ NULL: slow = slow.next // one step fast = fast.next.next // two steps if slow == fast: CYCLE DETECTED

Time: O(n). Space: O(1). This is the canonical O(1) space cycle detection. Any approach using a visited set uses O(n) space and is suboptimal.

Reversing a Singly Linked List

Three pointers: prev, curr, next. Iterate through the list, reversing each next pointer.

prev = NULL, curr = HEAD while curr ≠ NULL: next = curr.next // save next curr.next = prev // reverse the pointer prev = curr // advance prev curr = next // advance curr HEAD = prev // new head is the old tail

Time: O(n). Space: O(1). GATE may ask for number of pointer operations required, or trace the algorithm on a given list.

Finding the Middle Node

Same two-pointer idea: slow moves one step, fast moves two. When fast reaches the end, slow is at the middle. For a list of n nodes, slow stops at node ⌈n/2⌉.

2.5   Simulating Data Structures 2-Mark Brain Teasers

GATE regularly tests whether you understand the behavioural contract of data structures deeply enough to implement one using another. These are not trick questions — they test whether you understand LIFO vs FIFO at a mechanical level, and whether you can reason about amortized cost.

Queue Using Two Stacks — Amortized O(1)

Maintain two stacks: S1 (inbox) for enqueue operations, S2 (outbox) for dequeue operations.

  • Enqueue(x): Simply push x onto S1. Always O(1).
  • Dequeue(): If S2 is non-empty, pop from S2 directly. If S2 is empty, move all elements from S1 to S2 (pop from S1, push to S2 one by one), then pop from S2.

Trace: Enqueue 1, 2, 3. Dequeue twice.

Enqueue(1): S1=[1], S2=[] Enqueue(2): S1=[1,2], S2=[] Enqueue(3): S1=[1,2,3], S2=[] Dequeue(): S2 is empty → pour S1 into S2: S1=[], S2=[3,2,1] ← 1 is now on top of S2 Pop S2 top → return 1 ✓ (FIFO behaviour preserved) Dequeue(): S2 is non-empty → pop directly S2=[3,2] return 2 ✓

Notice: elements are poured from S1 to S2 only when S2 is empty. Each element crosses from S1 to S2 exactly once in its lifetime.

Amortized Analysis — Why It's O(1)

Consider n enqueue operations followed by n dequeue operations. Each element is pushed onto S1 once (cost 1), transferred to S2 once (cost 1), and popped from S2 once (cost 1). Total cost for n elements = 3n operations. Amortized cost per operation = 3n / n = O(1) per operation.

The expensive pour happens rarely (only when S2 runs empty), and its cost is distributed across all the cheap dequeues that follow it. A single pour of k elements pays for k future dequeues — each dequeue is "pre-paid" by its own enqueue.

Worst Case vs Amortized

A single dequeue call can be O(n) in the worst case (when S2 is empty and S1 has n elements). But amortized over all operations, each dequeue costs O(1). GATE questions on this always ask for amortized cost — answer O(1), not O(n).

Stack Using Two Queues — O(n) Unavoidable

Simulating a stack (LIFO) with queues (FIFO) is fundamentally more expensive because a queue gives you no way to directly access its last element — you must dequeue everything to reach it.

There are two approaches, both with one O(n) operation:

Expensive Push — O(n)

On push(x): enqueue x to Q2. Pour all of Q1 into Q2. Swap Q1 and Q2. Q1 always has newest element at front. Pop = dequeue from Q1. O(1) pop, O(n) push.

Expensive Pop — O(n)

Push = enqueue to Q1. O(1). Pop = dequeue all but last element from Q1 into Q2, dequeue the last (= stack top), swap Q1 and Q2. O(n) pop.

Why the Asymmetry?

A queue only exposes its front. The "top" of a stack is the most recently added element — which is at the back of a queue. There is no O(1) way to access the back of a queue. This forces at least one O(n) traversal, making the simulation inherently more expensive than the Queue-using-Stacks case where amortized O(1) is achievable.

Min-Stack — O(1) Minimum Retrieval

Design a stack that supports push, pop, and getMin() — all in O(1) time. The naive approach of scanning the stack for the minimum is O(n). The trick: maintain an auxiliary stack that tracks the minimum at every level.

Algorithm

Maintain two stacks: main (normal stack) and minStack (tracks current minimum).

push(x): Push x onto main. If minStack is empty or x ≤ minStack.top(), push x onto minStack.
pop(): Pop from main. If popped value == minStack.top(), also pop from minStack.
getMin(): Return minStack.top(). Always O(1).

Trace: push(5), push(3), push(7), push(2), pop(), getMin()

Operation Main Stack Min Stack Explanation push(5) [5] [5] 5 ≤ nothing → push to minStack push(3) [5,3] [5,3] 3 ≤ 5 → push to minStack push(7) [5,3,7] [5,3] 7 > 3 → do NOT push to minStack push(2) [5,3,7,2] [5,3,2] 2 ≤ 3 → push to minStack pop() [5,3,7] [5,3] popped 2 == minStack.top() → pop minStack too getMin() [5,3,7] [5,3] return minStack.top() = 3 ✓

After popping 2, the minimum correctly reverts to 3 — the minimum of what remains. minStack always holds the running minimum for the current stack depth.

Duplicate Handling

When pushing a value equal to the current minimum (x == minStack.top()), push it onto minStack too (using ≤, not <). If you use strict <, then when you pop a duplicate minimum, you'd incorrectly remove the minimum from minStack even though another copy remains in main. Always use ≤ in the push condition.

Chapter 03
Stacks — Theory, Implementation & Applications

3.1   The Stack — LIFO Principle Core

Definition

A stack is a linear data structure that follows the Last In, First Out (LIFO) principle. The element inserted most recently is the first one to be removed. Access is restricted to one end — the top.

Think of a stack of plates. You can only add a plate to the top, and you can only take a plate from the top. You cannot access the bottom plate without removing everything above it. This restriction — while seeming like a limitation — is exactly what makes stacks powerful for certain problem classes.

Stack state after: push(10), push(20), push(30) TOP → [ 30 ] ← most recently pushed, first to be popped [ 20 ] [ 10 ] ← pushed first, will be popped last

Three fundamental operations, all O(1):

  • Push(x) — Add element x to the top of the stack. Error if stack is full (overflow).
  • Pop() — Remove and return the top element. Error if stack is empty (underflow).
  • Peek() / Top() — Return the top element without removing it.
Stack Overflow vs Underflow

Overflow: Attempting to push onto a full stack (array implementation with fixed size). Underflow: Attempting to pop from an empty stack. Both are error conditions. GATE has presented code snippets and asked "under what condition does this fail?" — recognize these two conditions immediately.

3.2   Implementation — Array vs Linked List

Array Implementation

Maintain a variable top that holds the index of the current top element. Initialize top = −1 for empty stack.

Push(x): if top == MAX−1: OVERFLOW top = top + 1 A[top] = x Pop(): if top == −1: UNDERFLOW x = A[top] top = top − 1 return x

Array implementation is cache-friendly (contiguous memory), has O(1) operations, but has a fixed maximum size determined at creation time.

Linked List Implementation

The HEAD of the linked list acts as the top of the stack. Push = insert at HEAD. Pop = delete from HEAD. Both O(1). Size is dynamic — no overflow (until memory is exhausted).

Array Stack

Fixed size. Cache friendly. Overflow possible. O(1) all ops. Good when max size is known.

LL Stack

Dynamic size. No overflow. Each node has pointer overhead. O(1) all ops. Good when size is unknown.

3.3   Applications of Stacks GATE Favourite

1. Expression Evaluation & Conversion

Mathematical expressions are written in infix notation (operator between operands: A+B). Computers evaluate them more efficiently in postfix notation (operator after operands: AB+) or prefix (operator before: +AB). Stacks drive both the conversion and the evaluation.

Infix to Postfix conversion algorithm (Shunting-yard):

Initialize empty stack (for operators), empty output queue For each token in infix expression: If token is an OPERAND: → append directly to output If token is an OPERATOR: → while stack top has higher or equal precedence AND is not '(': pop from stack → append to output → push current operator onto stack If token is '(': → push onto stack If token is ')': → pop from stack → append to output, until '(' is found → discard the '(' After all tokens: → pop remaining operators to output

Precedence: ^(exponent) > *, / > +, −   |   Left-to-right associativity for +,−,*,/   |   Right-to-left for ^

Convert A + B * C − D to postfix. Trace the stack:

Token Stack Output A [] A + [+] A B [+] AB * [+,*] AB (* has higher precedence than +, push) C [+,*] ABC − [−] ABC*+ (pop * and + because − ≤ precedence of both) D [−] ABC*+D END [] ABC*+D− Result: ABC*+D−

Postfix Evaluation — evaluate ABC*+D− where A=2,B=3,C=4,D=5:

Postfix: 2 3 4 * + 5 − Token Stack 2 [2] 3 [2, 3] 4 [2, 3, 4] * [2, 12] (3*4=12) + [14] (2+12=14) 5 [14, 5] − [9] (14−5=9) Result: 9

Rule: when you see an operator in postfix, pop two operands, apply the operator, push the result.

2. Balanced Parentheses / Bracket Matching

Scan the expression. Push every opening bracket. When a closing bracket is seen, pop the stack and check if it matches. If stack is empty when you need to pop, or stack is non-empty at the end → unbalanced.

3. Function Call Stack

Every programming language uses a call stack to manage function calls. When a function is called, a stack frame is pushed containing: return address, local variables, parameters. When the function returns, its frame is popped and execution resumes at the return address. Recursion depth = stack depth.

Stack Overflow in Recursion

Infinite recursion (or recursion too deep) fills the call stack until no more frames can be pushed → stack overflow. This is the runtime error, not a logic error in your algorithm. GATE has asked about maximum recursion depth given a fixed stack size and frame size.

4. Undo/Redo Operations

Every action is pushed onto an "undo stack." Undo = pop from undo stack and push to "redo stack." Redo = pop from redo stack.

5. DFS — Depth First Search

Graph DFS uses a stack (explicitly or via recursion's call stack) to track nodes to visit next. Every time you visit a node, push its unvisited neighbors. This is covered in detail in the Algorithms notes.

3.4   Auxiliary Space & Recursion Depth Advanced — Bridges DS & Algorithms

When analyzing algorithms, we distinguish between two measures of memory usage. Understanding this distinction is essential for correctly computing space complexity — a recurring GATE question type.

Total Space

All memory used by the algorithm, including memory occupied by the input itself. For a function processing an array of n elements, total space includes that array.

Auxiliary Space

Extra memory used by the algorithm beyond the input. This is the memory the algorithm allocates as it runs — temporary variables, recursion stack frames, helper data structures.

GATE Always Asks for Auxiliary Space

When GATE asks "what is the space complexity of algorithm X?", it almost always means auxiliary space. An in-place sorting algorithm (like Heapsort) has O(1) auxiliary space even though the array itself is O(n) — because the algorithm doesn't allocate extra memory proportional to input size.

How Recursion Depth Determines Auxiliary Space

Every recursive function call creates a stack frame on the call stack. A stack frame contains the function's local variables, parameters, and the return address. If a function recurses to depth d before hitting the base case, then d stack frames exist simultaneously — consuming O(d) auxiliary space.

This is the hidden space cost of recursion that beginners miss. An algorithm that looks like it uses no extra memory may actually use O(n) auxiliary space if it recurses n levels deep.

Case Study: Linked List Reversal — Recursive vs Iterative

// RECURSIVE reversal Node* reverse(Node* head) { if (head == NULL || head->next == NULL) return head; // base case Node* rest = reverse(head->next); // recurse on rest of list head->next->next = head; head->next = NULL; return rest; } // For a list of n nodes: recursion goes n levels deep. // n stack frames exist simultaneously at deepest point. // Auxiliary space: O(n)
// ITERATIVE reversal (same result) prev = NULL; curr = head; while (curr != NULL) { next = curr->next; curr->next = prev; prev = curr; curr = next; } // Only 3 pointer variables regardless of list length. // Auxiliary space: O(1)

Same result. Same O(n) time. But the recursive version uses O(n) auxiliary space (call stack) while the iterative version uses O(1). For n = 10,000 nodes, the recursive version creates 10,000 stack frames — a potential stack overflow for large lists.

The Recursion–Stack Connection

The call stack IS a stack data structure, managed by the OS/runtime. When you recurse, you're implicitly doing push operations. When the base case returns and unwinds, you're doing pops. Any recursive algorithm can be rewritten iteratively by making the call stack explicit — managing your own stack data structure. This is exactly how iterative DFS works: you push/pop nodes on an explicit stack instead of relying on recursive call frames.

Chapter 04
Queues — All Variants

4.1   The Queue — FIFO Principle Core

Definition

A queue is a linear data structure following the First In, First Out (FIFO) principle. Elements are inserted at one end (the rear) and removed from the other end (the front).

Think of a line at a ticket counter. The first person to join the line is the first to be served. No cutting ahead. This fairness is exactly what FIFO provides.

Queue: enqueue(10), enqueue(20), enqueue(30) → dequeue returns 10 FRONT REAR ↓ ↓ [ 10 ] ← dequeue from here [ 30 ] ← enqueue here [ 20 ] [ 30 ]
  • Enqueue(x) — Insert x at the rear. O(1).
  • Dequeue() — Remove and return from the front. O(1).
  • Front() — Peek at front element without removing. O(1).
  • IsEmpty() — Check if queue has no elements. O(1).

4.2   Array Implementation — The Circular Queue Problem GATE Critical

A naive array implementation with front and rear indices has a fatal problem: after repeated enqueue/dequeue operations, rear reaches the end of the array even though front has advanced, leaving empty slots at the beginning that can't be used. This is called false overflow.

False Overflow: array of size 5, after dequeue(10), dequeue(20) Index: [ 0 ] [ 1 ] [ 2 ] [ 3 ] [ 4 ] State: [ --- ] [ --- ] [ 30 ] [ 40 ] [ 50 ] ↑ ↑ front rear Slots 0 and 1 are empty but rear can't go further → False overflow!

Solution: Circular Queue

Use modular arithmetic to wrap the indices around. After reaching the end of the array, the next index wraps back to 0.

Circular Queue Index Update

Rear = (Rear + 1) % MAX    |    Front = (Front + 1) % MAX

Full vs Empty Condition — The Ambiguity

In a circular queue of size MAX, both empty and full conditions can look like front == (rear+1) % MAX, creating ambiguity. Two solutions: (1) Keep one slot always empty — queue is full when (rear+1)%MAX == front, usable capacity = MAX−1. (2) Maintain a separate count variable. GATE specifies which approach to use — always check. The "one slot wasted" approach is the most commonly tested.

Circular queue of size 5 (usable = 4), initial front = rear = 0:

Enqueue(A): rear=(0+1)%5=1, A[1]=A Enqueue(B): rear=(1+1)%5=2, A[2]=B Enqueue(C): rear=(2+1)%5=3, A[3]=C Enqueue(D): rear=(3+1)%5=4, A[4]=D Is full? (4+1)%5 == 0 == front → YES. Queue is full (4 elements, one slot wasted) Dequeue(): returns A[front+1]=A[1]=A, front=(0+1)%5=1 Enqueue(E): rear=(4+1)%5=0, A[0]=E → wraps around correctly

4.3   Queue Variants Core

Deque — Double-Ended Queue

Definition

A deque (pronounced "deck") allows insertion and deletion at both ends. It generalizes both stacks and queues. Can function as either a stack (use one end only) or a queue (insert one end, delete other).

Input-Restricted Deque

Insertion at one end only. Deletion at both ends. Enqueue from rear; dequeue from either front or rear.

Output-Restricted Deque

Deletion at one end only. Insertion at both ends. Enqueue from either end; dequeue from front only.

Priority Queue

Definition

A priority queue is an abstract data type where each element has a priority. Elements are dequeued in order of their priority, not their insertion order. Highest priority element is served first regardless of when it was inserted.

A priority queue is most efficiently implemented using a heap data structure (covered in the Non-Linear DS notes). The heap property ensures the highest-priority element is always at the root, giving O(log n) insertion and O(log n) extraction.

Queue Applications

  • BFS (Breadth First Search) — The canonical queue application. Processes nodes level by level. A queue ensures you always explore the closest nodes first.
  • CPU Scheduling — Ready queue, round-robin scheduling.
  • Printer spooling — Print jobs processed in order of arrival.
  • Buffering — Network packet buffers, I/O buffers.

Chapter 05
Hashing — Hash Functions & Tables

5.1   The Core Idea Core

The goal of hashing is to achieve O(1) average-case search, insertion, and deletion — faster than the O(log n) offered by balanced BSTs and far better than the O(n) of unsorted arrays. The key insight: compute the storage location directly from the key, rather than searching for it.

A hash function h(k) maps a key k to an index in a hash table of size m. If h(k) = i, key k is stored at index i. To retrieve key k, compute h(k) and look at that index directly — O(1).

Hash Table

An array of size m (called the hash table) where element with key k is stored at index h(k). The hash function h : U → {0, 1, ..., m−1} maps the universe of keys U to table indices.

The Problem: Collisions

A collision occurs when two different keys hash to the same index: k1 ≠ k2 but h(k1) = h(k2). Since the key universe U is usually much larger than m, collisions are inevitable (pigeonhole principle). The hash function design and collision resolution strategy determine hashing performance in practice.

5.2   Hash Functions GATE Tested

Division Method

Formula

h(k) = k mod m   — map key k to index k mod m.

Simple and fast. The choice of m matters significantly. m should be a prime number not close to a power of 2. A prime m avoids systematic clustering. If m is a power of 2, h(k) depends only on the lowest-order bits of k, ignoring the rest — poor distribution.

m = 7 (prime). Keys: 15, 22, 29, 36.

h(15) = 15 mod 7 = 1   |   h(22) = 22 mod 7 = 1   ← collision!   |   h(29) = 1   ← collision!   |   h(36) = 1

All four keys hash to 1. This illustrates that even a prime m doesn't prevent clusters when keys form an arithmetic sequence with a common factor related to m.

Multiplication Method

Formula

h(k) = ⌊m × (k × A mod 1)⌋   where A is a constant 0 < A < 1. Knuth suggests A ≈ (√5 − 1)/2 ≈ 0.618.

The advantage is that m does not need to be prime — any value works. The multiplication distributes keys more uniformly than division when keys have patterns. Less sensitive to the choice of m.

Universal Hashing

Choose a hash function randomly from a family of hash functions. This guarantees that for any fixed set of keys, the expected number of collisions is low, regardless of adversarial key patterns. Used in high-security or theoretical contexts.

Properties of a Good Hash Function

  • Uniform distribution: Each index is equally likely. Minimizes clustering.
  • Deterministic: Same key always maps to same index.
  • Fast to compute: O(1) computation, not O(k) where k is key length.
  • Minimizes collisions: Different keys should map to different indices as often as possible.

5.3   Load Factor GATE Tested

Definition

Load Factor α = n / m, where n = number of keys stored in the table, m = size of the hash table.

The load factor directly controls the expected performance of hashing. It tells you how "full" the table is on average. Its impact differs depending on the collision resolution technique:

  • For chaining, α can exceed 1 (chains can grow). Average search time = O(1 + α).
  • For open addressing, α must be < 1 (table cannot be overfull). Performance degrades sharply as α → 1.
Expected Number of Probes

For open addressing with uniform hashing, the expected number of probes for a successful search is approximately (1/α) × ln(1/(1−α)), and for an unsuccessful search is 1/(1−α). As α → 1, unsuccessful search probes → ∞. This is the mathematical reason you never fill an open-addressing hash table completely.

Quadratic Probing — The α < 0.5 Hard Constraint

For quadratic probing to guarantee finding an empty slot when one exists, two conditions must hold simultaneously: (1) the table size m must be prime, and (2) the load factor α must be strictly less than 0.5. If α ≥ 0.5 with prime m, some insertions may fail even though the table is less than half full — the probe sequence cycles back to already-visited slots without covering all positions. This is a known mathematical property, not an implementation bug. If your problem uses quadratic probing and α ≥ 0.5, the algorithm is not guaranteed to work.

Chaining Formula — Why It Looks Different

Chaining's expected successful search cost is 1 + α/2, not (1/α)ln(...). This is because with chaining you always find the right chain in O(1) (one hash computation + one table lookup), then scan on average α/2 elements to find your key (half the chain length). The formula reflects the structure of the algorithm, not just the load.

Chapter 06
Collision Resolution Techniques

6.1   Separate Chaining Core

Definition

In separate chaining, each slot in the hash table is a pointer to a linked list. All keys that hash to the same index are stored in that slot's linked list. Colliding keys coexist in the chain.

Separate Chaining: m=5, insert 7,2,12,17,22 using h(k)=k mod 5 Index 0: [] Index 1: [] Index 2: [2]→[12]→[22]→NULL ← h(2)=h(12)=h(22)=2 Index 3: [] Index 4: [] Index 7%5=2: chained. Index 17%5=2: chained. Corrected: h(7)=2, h(17)=2 → all go to index 2
  • Search: Compute h(k), traverse the chain at that index. Expected O(1 + α).
  • Insert: Compute h(k), insert at head of chain. O(1) (ignoring hash computation).
  • Delete: Compute h(k), traverse chain, delete node. O(1 + α) expected.
Advantages

Simple. α can be > 1. Never full. Deletion is straightforward. Performance degrades gracefully.

Disadvantages

Pointer overhead for each node. Poor cache performance (linked list nodes scattered in memory). Extra memory for pointers even in empty slots.

6.2   Open Addressing Most Tested

Definition

In open addressing, all keys are stored within the hash table itself — no external chains. When a collision occurs, an alternate slot is found by probing — examining a sequence of slots until an empty one is found. α must be strictly < 1.

The probe sequence for key k is: h(k, 0), h(k, 1), h(k, 2), ... where the i-th probe is h(k, i).

Linear Probing

Formula

h(k, i) = (h'(k) + i) mod m   where h'(k) is the primary hash function and i = 0, 1, 2, ...

On collision, try the next slot, then the next, and so on linearly. Simple and cache-friendly (sequential memory access).

Primary Clustering: The major problem with linear probing. Contiguous sequences of occupied slots grow longer over time. Any key that hashes into the middle of a cluster must probe through the entire cluster to find an empty slot, making the cluster grow even larger. Performance degrades significantly as α increases.

Insert 7, 14, 21 into hash table of size 7, h'(k) = k mod 7:

h(7) = 0 → insert at 0. Table: [7, _, _, _, _, _, _] h(14) = 0 → collision! probe i=1: slot 1 empty → insert at 1. h(21) = 0 → collision! probe i=1: slot 1 occupied. probe i=2: slot 2 empty → insert at 2. Table: [7, 14, 21, _, _, _, _] A cluster of length 3 has formed at indices 0,1,2.

Quadratic Probing

Formula

h(k, i) = (h'(k) + c₁i + c₂i²) mod m   where c₁, c₂ are constants (often c₁=0, c₂=1 giving h'(k) + i²).

Instead of probing linearly (i, i+1, i+2...), probe quadratically (i, i+1, i+4, i+9...). This breaks up primary clusters. However, it introduces secondary clustering: two keys with the same initial hash value h'(k) will follow the exact same probe sequence — they still cluster, just less densely than linear probing.

Quadratic Probing May Not Find All Slots

Unlike linear probing which always finds an empty slot (if one exists), quadratic probing is not guaranteed to probe all slots. For quadratic probing to guarantee finding an empty slot when one exists, m must be prime AND α < 0.5. GATE has directly tested this constraint.

Double Hashing

Formula

h(k, i) = (h₁(k) + i × h₂(k)) mod m   where h₁ and h₂ are two independent hash functions.

The step size for probing is itself determined by a second hash function — so different keys with the same primary hash get different probe sequences. This eliminates both primary and secondary clustering. Double hashing produces the closest approximation to truly random probing in practice.

Constraint: h₂(k) must never evaluate to 0 (otherwise we'd probe the same slot forever) and must be coprime to m. A common choice: h₁(k) = k mod m, h₂(k) = 1 + (k mod (m−1)).

6.3   Deletion in Open Addressing GATE Trap

Deletion in open addressing requires special care. You cannot simply mark a slot as empty after deletion. Consider the following scenario:

Keys A, B, C all hash to index 3. Due to linear probing, A is at 3, B at 4, C at 5.

Now delete B: mark slot 4 as empty.

Now search for C: h(C) = 3, slot 3 has A (not C), probe slot 4 → empty! Search stops. Concludes C is absent. WRONG. C is at slot 5 but we stopped early.

Solution: Lazy Deletion. Instead of marking deleted slots as "empty", mark them as "deleted" (tombstone). During search, probe past tombstones (don't stop). During insert, reuse tombstone slots. This preserves the probe sequence integrity while allowing slot reuse.

Tombstone Accumulation

Over time, many deletions fill the table with tombstones. This degrades search performance because you probe past tombstones unnecessarily. Solution: periodically rehash the entire table — rebuild it fresh without tombstones. This is an O(n) operation but amortized over many operations.

Hash Table Worst Case

Hash table worst case O(n) occurs when all keys hash to the same slot — all keys form one long chain or require probing through the entire table. This happens with a bad hash function or adversarial input. In practice with a good hash function, O(1) average is reliable.

7.3   GATE Question Patterns for This Topic

  • Address computation: Given array, base, size, index — compute address using row or column major formula. Very frequently tested, usually 1 mark.
  • Pointer manipulation tracing: Given a sequence of LL operations, trace the final list state. Tests order-of-operations understanding.
  • Stack expression questions: Convert infix to postfix, or evaluate a postfix expression. 2 mark questions, trace required.
  • Circular queue capacity: "How many elements can a circular queue of size n hold?" — answer is n−1 (one slot wasted), not n.
  • Collision counting: Given m, h(k), and a set of keys — how many collisions occur? Insert one by one and count.
  • Probing sequence: Given h(k), linear/quadratic/double probing, trace where a key is inserted or found.
  • Load factor and performance: Given n and m, compute α. Given α, compute expected number of probes using the formulae.
  • Floyd's algorithm: Trace slow/fast pointers on a given linked list to detect cycle meeting point.
