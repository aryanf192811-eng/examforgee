Chapter 01
Asymptotic Notations — Precise Definitions

1.1   The Five Asymptotic Notations Core

Asymptotic analysis describes how an algorithm's resource usage (time or space) grows as the input size n → ∞. It abstracts away constants and lower-order terms to focus on the dominant growth rate. The five notations each capture a different relationship between two functions.

Big-O — Upper Bound

f(n) = O(g(n)) means: ∃ positive constants c and n₀ such that for all n ≥ n₀, f(n) ≤ c·g(n).
"f grows no faster than g." g is an upper bound on f (up to a constant factor).

Big-Ω — Lower Bound

f(n) = Ω(g(n)) means: ∃ c > 0 and n₀ such that for all n ≥ n₀, f(n) ≥ c·g(n).
"f grows at least as fast as g." g is a lower bound on f.

Big-Θ — Tight Bound

f(n) = Θ(g(n)) means: f(n) = O(g(n)) AND f(n) = Ω(g(n)).
Equivalently: ∃ c₁, c₂ > 0 and n₀ such that for all n ≥ n₀, c₁·g(n) ≤ f(n) ≤ c₂·g(n).
"f grows at exactly the same rate as g." Θ is the strongest and most precise statement.

Little-o & Little-ω — Strict Bounds

f(n) = o(g(n)): f grows strictly slower than g. lim_{n→∞} f(n)/g(n) = 0.
f(n) = ω(g(n)): f grows strictly faster than g. lim_{n→∞} f(n)/g(n) = ∞.

O ≠ Θ — The Most Common Confusion

O is an upper bound — it says "at most." Saying an algorithm is O(n²) doesn't mean it takes n² steps — it could be O(n) and still technically be O(n²). When GATE says "what is the time complexity?" it is asking for the tight bound Θ, not just an upper bound O. A correct answer is precise: Merge Sort is Θ(n log n), not just O(n²) (though both are technically true).

Is 2^(n+1) = O(2ⁿ)?

2^(n+1) = 2·2ⁿ. With constant c=2: 2·2ⁿ ≤ 2·2ⁿ for all n≥1. Yes, 2^(n+1) = O(2ⁿ). The "+1" in the exponent only multiplies by 2 — a constant factor.

Is 2^(2n) = O(2ⁿ)?

2^(2n) = (2ⁿ)². For no constant c does (2ⁿ)² ≤ c·2ⁿ for all large n. No — 2^(2n) ≠ O(2ⁿ).

Recurrences & Master Theorem

2.1   Solving Recurrences — Three Methods GATE Tested

Method 1 — Substitution (Guess and Verify)

Guess the form of the solution, then use mathematical induction to prove the guess is correct. Used when you have a good intuition about the answer.

Solve T(n) = 2T(n/2) + n by substitution:

Guess: T(n) = O(n log n). Assume T(k) ≤ ck log k for all k < n. T(n) = 2T(n/2) + n ≤ 2·c(n/2)log(n/2) + n (by inductive hypothesis) = cn·log(n/2) + n = cn(log n − log 2) + n = cn log n − cn + n = cn log n + n(1 − c) ≤ cn log n (for c ≥ 1) Induction holds. T(n) = O(n log n). ✓

Method 2 — Recursion Tree

Draw the recursion tree. At each level, compute the work done. Sum across all levels to get total work.

T(n) = 2T(n/2) + n:

Level 0 (root): n work, 1 node Level 1: n/2+n/2 = n work, 2 nodes Level 2: 4×n/4 = n work, 4 nodes ... Level k: n work, 2ᵏ nodes Depth: log₂n levels (until subproblems reach size 1) Total work: n × log n = Θ(n log n)

Method 3 — Master Theorem

Master Theorem — T(n) = aT(n/b) + f(n), a≥1, b>1

Let c* = log_b(a) (the "critical exponent").

Case 1: f(n) = O(nᶜ) for some c < c* → T(n) = Θ(n^c*)
(subproblems dominate — f(n) is polynomially smaller than n^c*)

Case 2: f(n) = Θ(n^c* · logᵏn) for some k ≥ 0 → T(n) = Θ(n^c* · logᵏ⁺¹n)
(equal contribution at every level)

Case 3: f(n) = Ω(nᶜ) for some c > c*, AND regularity condition af(n/b) ≤ δf(n) for some δ<1 → T(n) = Θ(f(n))
(root dominates — f(n) is polynomially larger than n^c*)

Master Theorem Gaps

The Master Theorem does NOT apply when: (1) T(n) = T(n−1) + f(n) — subtraction, not division. (2) f(n) is not polynomially related to n^c*. Example: T(n) = 2T(n/2) + n/log n — falls in a "gap" between Cases 1 and 2 where f(n)=o(n) but not O(n^(1-ε)). Use recursion tree or substitution instead.

3.2   Bubble, Selection & Insertion Sort — Analysis GATE Tested

Bubble Sort

Repeatedly passes through the array, swapping adjacent elements that are out of order. After pass k, the k largest elements are in their final positions ("bubbled" to the end).

for i = 0 to n−2: for j = 0 to n−2−i: if A[j] > A[j+1]: swap(A[j], A[j+1]) Total comparisons (worst): (n−1)+(n−2)+...+1 = n(n−1)/2 = Θ(n²) Best case (already sorted) with early termination flag: Ω(n)

Selection Sort

In pass i, find the minimum element in A[i..n−1] and swap it into position i. Always makes exactly n(n−1)/2 comparisons regardless of input — no best case advantage.

Total comparisons: (n−1)+(n−2)+...+1 = n(n−1)/2 regardless of input order Total swaps: exactly n−1 (one swap per pass, except possibly 0 swaps for already-placed elements) → Selection Sort makes the FEWEST swaps of all O(n²) sorts
Selection Sort — Minimum Swaps Property

Selection Sort makes at most n−1 swaps. This makes it optimal for situations where swaps are expensive (e.g., writing to flash memory). GATE tests this: "Which sort makes the fewest writes?" → Selection Sort.

Insertion Sort

Builds a sorted subarray left to right. For each new element, finds its correct position in the sorted part by shifting elements right.

Comparisons: Best case (sorted): n−1 comparisons, 0 shifts → Θ(n) Worst case (reversed): n(n−1)/2 comparisons → Θ(n²) Average case: ~n²/4 → Θ(n²) Key property: number of comparisons = number of inversions + (n−1)
Inversion

A pair (i,j) is an inversion if i < j but A[i] > A[j]. A sorted array has 0 inversions. A reverse-sorted array has n(n−1)/2 inversions (maximum). The number of swaps in Bubble Sort and comparisons in Insertion Sort = number of inversions.

3.3   Merge Sort & Quicksort — Deep Analysis GATE Critical

Merge Sort

Divide the array in half, recursively sort each half, merge the sorted halves. The merge step takes Θ(n). Recurrence: T(n) = 2T(n/2) + Θ(n) → T(n) = Θ(n log n).

MergeSort(A, l, r): if l ≥ r: return mid = (l+r)/2 MergeSort(A, l, mid) MergeSort(A, mid+1, r) Merge(A, l, mid, r) // O(n) merge step Properties: Time: Θ(n log n) always — best, average, worst Space: O(n) auxiliary — temporary array for merge Stable: YES — equal elements maintain original relative order Online: NO — needs entire array upfront
Merge Sort is the ONLY stable O(n log n) sort that's always guaranteed

Heapsort is O(n log n) but NOT stable. Quicksort is O(n log n) average but NOT stable and has O(n²) worst case. Merge Sort is the only comparison-based sort that is both guaranteed Θ(n log n) in all cases AND stable. This distinction is directly tested in GATE.

Quicksort

Choose a pivot, partition the array so elements ≤ pivot are on the left and elements ≥ pivot are on the right, recursively sort both partitions.

Best/Average case: pivot splits array roughly in half each time T(n) = 2T(n/2) + Θ(n) → Θ(n log n) Worst case: pivot is always min or max (sorted/reverse-sorted input) T(n) = T(n−1) + Θ(n) → Θ(n²) Worst-case input: already sorted array with first/last element as pivot Average-case analysis: random pivot → expected Θ(n log n) Partition strategies: Lomuto: last element as pivot, simpler to implement Hoare: two pointers from both ends, fewer swaps on average
Quicksort in Practice — Why It's Often Fastest

Despite its O(n²) worst case, Quicksort is often faster than Merge Sort in practice because: (1) O(1) auxiliary space vs O(n) for Merge Sort. (2) Better cache performance — in-place operations on the same array. (3) Small constants in the average-case Θ(n log n). Production libraries use introsort — quicksort that switches to heapsort when recursion depth exceeds 2 log n to avoid worst case.

Chapter 04
Non-Comparison Sorting — Breaking the Ω(n log n) Barrier

4.1   Counting Sort GATE Tested

Counting Sort — Idea

For integers in range [0, k], count occurrences of each value, compute cumulative counts (positions), and place elements directly into sorted output. No comparisons — uses the integer values as indices.

CountingSort(A, n, k): C[0..k] = 0 // count array for i = 1 to n: C[A[i]]++ // count occurrences for i = 1 to k: C[i] += C[i−1] // cumulative counts = final positions for i = n downto 1: // fill output (reverse for stability) B[C[A[i]]] = A[i] C[A[i]]-- return B Time: Θ(n + k) Space: Θ(n + k) Stable: YES (reverse loop preserves order)
Counting Sort Requires Bounded Integer Range

Counting sort only works when k = O(n) — otherwise the Θ(n+k) becomes dominated by Θ(k) which could be much larger than n. For k = n, time is Θ(n). For k = n², time is Θ(n²) — worse than comparison-based sorts. Never use counting sort for large key ranges.

4.2   Radix Sort & Bucket Sort GATE Tested

Radix Sort

Radix Sort

Sort multi-digit integers digit by digit, from least significant digit (LSD) to most significant digit. Use a stable sort (Counting Sort) at each digit position. For n numbers with d digits each in base b:

Time: Θ(d · (n + b)) For d = Θ(log_b k) where k is max value: Time = Θ(log_b(k) · (n + b)) Optimal when b = n: Time = Θ((log n / log n) · 2n) = Θ(n) (When n-bit numbers sorted in base n: d=1 digit, n+n values)

Sort {329, 457, 657, 839, 436, 720, 355} by Radix Sort (base 10, d=3):

Pass 1 (units digit): 720, 355, 436, 457, 657, 329, 839 Pass 2 (tens digit): 720, 329, 436, 839, 355, 457, 657 Pass 3 (hundreds): 329, 355, 436, 457, 657, 720, 839 ← sorted

3 passes of Counting Sort on 7 elements with digits 0-9. Time: Θ(3 × (7+10)) = Θ(51) = Θ(1) for fixed d — scales as Θ(n) for the general case.

Bucket Sort

Bucket Sort

For uniformly distributed input in [0,1): create n buckets covering equal-sized intervals, distribute elements into buckets, sort each bucket (Insertion Sort), concatenate. Expected time: Θ(n) when input is uniformly distributed.

Time: Θ(n) expected (uniform distribution), Θ(n²) worst case Space: Θ(n) for buckets Stable: depends on the sort used within each bucket

Chapter 05
Lower Bound — Why Sorting Can't Be Faster Than Ω(n log n)

5.1   Decision Tree Lower Bound GATE Tested

This is one of the most elegant results in algorithms. We prove that any comparison-based sorting algorithm must make at least Ω(n log n) comparisons in the worst case — regardless of how clever the algorithm is. It's a lower bound on the entire class of algorithms, not just specific ones.

Decision Tree Model

Any comparison-based sorting algorithm can be modeled as a decision tree: a binary tree where each internal node is a comparison (Aᵢ ≤ Aⱼ?), with left branch for "yes" and right branch for "no," and each leaf represents a complete permutation of the input (the sorted output).

  1. 1
    An array of n elements has n! possible orderings (permutations). A correct sorting algorithm must distinguish between all of them — so the decision tree must have at least n! leaves.
  2. 2
    A binary tree of height h has at most 2ʰ leaves. So: 2ʰ ≥ n!, which means h ≥ log₂(n!).
  3. 3
    By Stirling's approximation: log₂(n!) = Θ(n log n). Specifically: log₂(n!) ≥ n/2 · log₂(n/2) = Ω(n log n).
  4. 4
    The height h of the decision tree equals the worst-case number of comparisons. Therefore, any comparison-based sort requires Ω(n log n) comparisons in the worst case.
The Consequence — Merge Sort and Heapsort Are Optimal

Since Merge Sort and Heapsort both run in Θ(n log n) worst case, and the lower bound is Ω(n log n), these algorithms are asymptotically optimal for comparison-based sorting. No comparison-based algorithm can do asymptotically better in the worst case. Non-comparison sorts (Counting, Radix, Bucket) escape this lower bound by not using comparisons.

Lower Bound Applies to WORST CASE, Not All Cases

The Ω(n log n) lower bound is for worst-case complexity. An algorithm CAN do better on specific inputs — Insertion Sort is Θ(n) on nearly-sorted input. The lower bound says: for any comparison-based algorithm A, there exists a permutation on which A makes Ω(n log n) comparisons. Timsort exploits this by using Insertion Sort on already-sorted runs and Merge Sort to combine them.

Searching Algorithms — Linear, Binary & Interpolation

6.1   Linear Search & Binary Search GATE Tested

Linear Search

Scan from left to right until element found or array exhausted. Works on unsorted arrays.

Best case: O(1) — element at position 0 Average case: Θ(n) — element at position n/2 on average Worst case: O(n) — element at last position or absent

Binary Search

Requires a sorted array. At each step, compare with the middle element. Eliminate half the remaining search space.

BinarySearch(A, l, r, key): while l ≤ r: mid = l + (r−l)/2 // avoid integer overflow vs (l+r)/2 if A[mid] == key: return mid else if A[mid] < key: l = mid + 1 else: r = mid − 1 return −1 // not found Time: Θ(log n) — recurrence T(n) = T(n/2) + 1 Space: O(1) iterative, O(log n) recursive

Binary search in sorted array [2,5,8,12,16,23,38,56,72,91], search for 23:

l=0, r=9 → mid=4 → A[4]=16 < 23 → l=5 l=5, r=9 → mid=7 → A[7]=56 > 23 → r=6 l=5, r=6 → mid=5 → A[5]=23 == 23 → FOUND at index 5 3 comparisons for n=10. ⌊log₂(10)⌋+1 = 4 max comparisons.

Interpolation Search

Interpolation Search

Like binary search but uses an interpolated probe position instead of always choosing the midpoint:
pos = l + [(key − A[l]) / (A[r] − A[l])] × (r − l)
For uniformly distributed data: Θ(log log n) average. Worst case: O(n) (exponentially skewed data).

Chapter 07
Order Statistics — Finding the k-th Smallest Element

7.1   Selection Problem & Quickselect GATE Tested

Order Statistic

The k-th order statistic is the k-th smallest element. Finding it is called the selection problem. Naïve approach: sort the array in Θ(n log n) and return element at index k−1. But we can do better.

Quickselect — O(n) Average

Use Quicksort's partition step. After partitioning around a pivot, the pivot is in its correct position. If pivot is at index k−1, done. If k−1 < pivot_index, recurse on left. Else recurse on right. Unlike Quicksort, only one recursive call is made.

Quickselect(A, l, r, k): if l == r: return A[l] pivot_idx = Partition(A, l, r) if k == pivot_idx: return A[k] elif k < pivot_idx: return Quickselect(A, l, pivot_idx−1, k) else: return Quickselect(A, pivot_idx+1, r, k) Time: T(n) = T(n/2) + Θ(n) → Θ(n) average (random pivot) T(n) = T(n−1) + Θ(n) → Θ(n²) worst case (bad pivot)

Median of Medians — Guaranteed O(n) Worst Case

The Blum-Floyd-Pratt-Rivest-Tarjan algorithm (BFPRT) guarantees O(n) worst case selection by choosing a "good" pivot:

  1. 1
    Divide n elements into ⌈n/5⌉ groups of 5.
  2. 2
    Find the median of each group (insertion sort each group — O(1) per group).
  3. 3
    Recursively find the median of the ⌈n/5⌉ medians — this is the pivot.
  4. 4
    Partition around this pivot and recurse on the appropriate side.
Recurrence: T(n) = T(n/5) + T(7n/10) + Θ(n) → T(n) = Θ(n) The pivot is guaranteed to be in the range [30%, 70%] of the array, ensuring no partition is larger than 70% → linear time guaranteed.
GATE Pattern — Min, Max, Both Min and Max

Finding minimum: n−1 comparisons (optimal). Finding maximum: n−1 comparisons. Finding both simultaneously: can be done in ⌊3n/2⌋ − 2 comparisons — better than 2(n−1) by pairing elements first. GATE asks for exact comparison counts on these problems.

8.3   GATE Question Patterns

  • Asymptotic classification NAT: "Arrange in increasing order: n!, 2ⁿ, n³, n log n, √n." Use the growth hierarchy. n log n < n³ < 2ⁿ < n!.
  • Master Theorem: "T(n) = 3T(n/4) + n log n. What is T(n)?" c*=log₄3≈0.79. f(n)=n log n = Ω(n^0.79+ε). Regularity: 3f(n/4) ≤ (3/4)f(n). Case 3: T(n)=Θ(n log n).
  • Sorting property MCQ: "Which sorting algorithm is both stable AND in-place AND O(n log n) worst case?" No such algorithm exists — Merge Sort (stable, O(n log n) worst) uses O(n) space. Heapsort (in-place, O(n log n) worst) is unstable.
  • Comparison count NAT: "How many comparisons does Selection Sort make on an 8-element array?" n(n−1)/2 = 8×7/2 = 28, regardless of input.
  • Inversion count: "How many comparisons does Insertion Sort make on [5,4,3,2,1]?" Inversions = 10 = n(n−1)/2. Comparisons = inversions + (n−1) = 10+4 = 14? No — comparisons = inversions for reverse-sorted = n(n-1)/2 = 10. Check: n(n-1)/2 = 4×5/2 = 10.
  • Lower bound question: "Is O(n) sorting possible in a comparison model?" No — proven Ω(n log n) lower bound. Possible in non-comparison model with bounded keys.
  • Recurrence solving: T(n) = T(n-1) + log n. Neither subtraction recurrence fits Master Theorem. Use summation: T(n) = Σᵢ log i = log(n!) = Θ(n log n).
  • Worst case trigger: "For which input does Quicksort (with first element as pivot) perform worst?" Already sorted or reverse-sorted arrays — pivot always ends up at boundary position.
