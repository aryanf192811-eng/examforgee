Chapter 01
Entity-Relationship Model — Entities, Attributes & Relationships

1.1   Entities & Attributes Core

The ER model is a conceptual design tool — it describes what data exists and how it is related, before any tables are created. Think of it as the blueprint before the construction. GATE tests the ER model at two levels: recognizing correct notation in diagrams, and applying mapping rules to generate relational schemas.

Entity vs Entity Set

Entity: A single real-world object that is distinguishable from others. Example: the student "Aryan" with RollNo 101.
Entity Set: A collection of entities of the same type sharing the same properties. Example: the set of all students.
In ER diagrams, entity sets are drawn as rectangles. Individual entities are instances, not shown in the diagram.

Attribute Types

Attribute TypeNotationMeaningExample
SimpleOvalAtomic, indivisibleRollNo, Age
CompositeOval with child ovalsCan be divided into sub-partsName → {FirstName, LastName}
Multi-valuedDouble ovalCan hold multiple values for one entityPhoneNumbers, Degrees
DerivedDashed ovalComputed from other attributesAge (derived from DOB)
KeyUnderlined ovalUniquely identifies each entityRollNo
Partial KeyDashed underlined ovalUniquely identifies weak entity within ownerLoanNo (within a Customer)
Derived Attributes Are NOT Stored

A derived attribute (dashed oval) is computed on demand — it is not physically stored in the database. GATE trap: "Which attribute requires a new column in the table?" — derived attributes do NOT generate a column. Composite attributes generate columns for their leaf-level parts only (not the composite attribute itself).

1.2   Strong vs. Weak Entities GATE Critical

Strong Entity

Has its own primary key. Exists independently. Drawn as a single rectangle. Example: Student, Course, Employee.

Weak Entity

No primary key on its own. Depends on an owner (strong) entity for existence. Drawn as a double rectangle. Identified by its discriminator + owner's key.

Identifying Relationship & Discriminator

The relationship connecting a weak entity to its owner is the identifying relationship — drawn as a double diamond. The attribute that partially distinguishes weak entity instances among those with the same owner is the discriminator (also called partial key), drawn with a dashed underline.

Full key of a weak entity = Owner's primary key + Discriminator.
Example: Dependent is weak, owned by Employee. Discriminator = DependentName. Full identifier = (EmployeeID, DependentName).

Identify what is strong/weak: A bank has Customers. Each Customer has multiple Loans. A Loan cannot exist without a Customer.

Customer → Strong entity. Has its own key (CustomerID).

Loan → Weak entity. No meaning without Customer. Identifying relationship: Has. Discriminator: LoanNumber (unique per customer, not globally). Full key: (CustomerID, LoanNumber).

Total Participation of Weak Entity

A weak entity always has total participation in its identifying relationship — it cannot exist without its owner. GATE frequently asks: "What is the participation of a weak entity in its identifying relationship?" Answer: always total (double line). The owner entity may have partial or total participation depending on whether all owner instances must have the weak entity.

1.3   Relationships — Cardinality & Participation GATE Critical

Cardinality Ratios

1:1 — One entity in A is associated with at most one entity in B, and vice versa.
1:N — One entity in A is associated with many in B; each entity in B is associated with at most one in A.
M:N — Many entities in A associate with many in B.

Participation Constraints

Total participation (double line): Every entity in the set MUST participate in at least one relationship instance. Equivalent to a NOT NULL foreign key constraint or mandatory participation.
Partial participation (single line): Some entities may not participate. Optional.

ConstraintER NotationRelational equivalent
Total participationDouble line on entity sideNOT NULL foreign key / mandatory FK
Partial participationSingle line on entity sideNullable foreign key
1:N relationshipArrow on "1" side, no arrow on "N" sideFK placed in the "N" side table
1:1 relationshipArrow on both sidesFK can go in either table (prefer total-participation side)
M:N relationshipNo arrowsRequires a new bridge/junction table

Degree of a Relationship

Binary (degree 2): two entity sets — most common. Ternary (degree 3): three entity sets — GATE occasionally tests these. Unary (degree 1): an entity relates to itself (e.g., Employee manages Employee). Key distinction: a ternary relationship is not the same as three binary relationships — they can capture different semantics.

Chapter 02
ER-to-Relational Mapping — The Seven Rules

2.1   The Seven Mapping Rules GATE Tested

Converting an ER diagram to a relational schema is algorithmic. GATE tests both the output (which tables are created) and the reasoning (why that choice). The seven rules below cover every case.

  1. 1Strong Entity Set → one relation with all simple attributes. Composite attributes: use only leaf-level components. Multi-valued attributes: separate table (see Rule 7). Primary key = entity's key attribute.
  2. 2Weak Entity Set → one relation with its own attributes + the primary key of the identifying (owner) entity. Primary key = (owner's PK + discriminator).
  3. 31:1 Relationship → add a foreign key from one side into the other. Prefer the side with total participation — place the FK there (avoids NULLs). Alternatively, merge both entity sets into one table if both have total participation.
  4. 41:N Relationship → add a foreign key in the N side table referencing the 1-side primary key. Include any relationship attributes in the N-side table. Do NOT create a new table (unless the N-side has partial participation and you want to avoid NULLs — then a separate table is optional).
  5. 5M:N Relationship → always create a new junction table. Its primary key = composite of both entity sets' primary keys. Include any relationship attributes as additional columns in this junction table.
  6. 6Multi-valued Attribute → create a new table with: the entity's primary key (FK back to entity) + the multi-valued attribute. Primary key = (entity PK + attribute value).
  7. 7N-ary (Ternary+) Relationship → create a new table. PK = combination of all participating entity keys (may be reduced if cardinality constraints allow).

2.2   Mapping Worked Examples GATE Tested

Map: Student (RollNo, Name, PhoneNos[multi-valued]) — takes → Course (CID, Title). Cardinality: M:N. Both partial participation.

Rule 1: Student(RollNo, Name) ← composite ignored, PK=RollNo Rule 1: Course(CID, Title) ← PK=CID Rule 6: Student_Phone(RollNo, Phone) ← multi-valued; PK=(RollNo,Phone) FK: RollNo → Student Rule 5: Takes(RollNo, CID) ← M:N junction; PK=(RollNo,CID) FK: RollNo → Student; CID → Course

Result: 4 tables. Note: PhoneNos generates a separate table, not a column in Student.

Map: Department (DID, Name) — has → Employee (EID, Name). Cardinality: 1:N (one dept has many employees). Employee has total participation.

Rule 1: Department(DID, Name) Rule 1: Employee(EID, Name, DID) ← FK added to N-side (Employee) FK: DID → Department NOT NULL (total participation of Employee)

Result: 2 tables. FK goes in Employee (the N-side). NOT NULL because Employee totally participates — every employee must belong to a department.

Map weak entity: Employee (EID, Name) — has → Dependent (DependentName, DOB). Identifying relationship. Dependent is weak.

Rule 1: Employee(EID, Name) Rule 2: Dependent(EID, DependentName, DOB) PK = (EID, DependentName) FK: EID → Employee ON DELETE CASCADE ← deleting employee removes dependents
1:1 with Both Partial Participation — 3 Options All Valid

When a 1:1 relationship has partial participation on both sides, you have three valid choices: (a) FK in table A referencing B, (b) FK in table B referencing A, (c) merge A and B. GATE may ask which is most efficient — the answer is usually the one that minimizes NULLs, which means put the FK on the side that participates more (or is more frequently associated).

Chapter 03
File Organization — Storage, Records & Structures

3.1   Records, Blocks & File Structures Core

Key Terminology

Block (Page): The unit of I/O transfer between disk and memory. Typical size: 512B to 16KB. A database reads/writes one block at a time — this is the cost model.
Record: One row of a table stored on disk. Can be fixed-length or variable-length.
Blocking Factor (bfr): Number of records that fit in one block = ⌊Block Size / Record Size⌋.
Number of Blocks: ⌈Number of Records / bfr⌉.

Block size = 1024 bytes. Record size = 100 bytes. 500 records. How many blocks?

bfr = ⌊1024 / 100⌋ = ⌊10.24⌋ = 10 records/block Blocks = ⌈500 / 10⌉ = ⌈50⌉ = 50 blocks

Spanned vs. Unspanned Records

Unspanned

A record must fit entirely within one block. Simple to implement. Wastes space when record size doesn't divide block size evenly. Most common in practice.

Spanned

A record can span across two consecutive blocks. Requires a pointer at the end of one block linking to the continuation in the next. Better space utilization. Used when records are large or variable-length.

Chapter 04
Indexing — Dense, Sparse, Primary, Secondary & Clustering

4.1   Index Classification GATE Tested

An index is a separate, smaller data structure that maps search key values to disk block addresses — allowing the database to find records without scanning the entire file. GATE tests both the classification and the numerical cost of index lookup.

Index TypeBuilt onFile orderingOne entry per
Primary IndexOrdering key fieldFile IS ordered on this fieldBlock (sparse — one entry per block)
Clustering IndexOrdering non-key fieldFile ordered on non-key fieldDistinct value of the field
Secondary IndexNon-ordering fieldFile is NOT ordered on this fieldRecord (dense — one entry per record)
Dense vs. Sparse Index

Dense index: One index entry for every record in the file. Always points directly to the record. Required for secondary indexes (since file is not ordered).
Sparse index: One index entry for every block (or every distinct key group). Smaller index, but requires extra search within a block after finding the block. Only possible when file is ordered on the search key — i.e., for primary and clustering indexes.

File: 10,000 records, block size = 1024 B, record size = 100 B, key field = 10 B, block pointer = 6 B. How many index entries for a primary (sparse) index? How many index blocks?

bfr = ⌊1024/100⌋ = 10 records/block Data blocks = ⌈10000/10⌉ = 1000 blocks Primary index has ONE entry per data block: Index entries = 1000 Index record size = key(10) + pointer(6) = 16 B Index bfr = ⌊1024/16⌋ = 64 entries/index block Index blocks = ⌈1000/64⌉ = 16 index blocks Binary search on 16 index blocks: ⌈log₂16⌉ = 4 I/Os Then 1 data block read = 5 I/Os total (vs. binary search on data directly = ⌈log₂1000⌉ = 10 I/Os)

4.2   Secondary Index — Dense & Multi-Level GATE Tested

Secondary Index on Non-Key Field

When the search field has duplicate values (non-key), a secondary index needs one entry per distinct value, with that entry containing a pointer to a bucket of record pointers (since multiple records share the value). This two-level indirection is important: secondary index → bucket → records.

Multi-Level Index

If the first-level index is too large to fit in memory, build an index on the index. At each level, entries reduce by a factor of the index blocking factor (fan-out). Levels needed = ⌈log_bfr_i(number of first-level entries)⌉. This is the foundation of B-trees.

Secondary Index Does NOT Imply Dense

Secondary index on a key field is dense (one entry per record — because the file is not ordered on this field, you cannot skip any). But secondary index on a non-key field can have one entry per distinct value — not one per record. GATE traps you by asking "dense or sparse?" for secondary indexes on non-key fields: the answer depends on whether it's key or non-key.

Chapter 05
B-Trees & B⁺-Trees

5.1   B-Tree Properties Core

B-Tree of Order p — Node Structure

Every internal node stores at most p−1 keys and at most p pointers (tree pointers to child nodes). The keys within a node are in sorted order. Between consecutive keys kᵢ and kᵢ₊₁, a pointer leads to a subtree containing all keys k where kᵢ < k < kᵢ₊₁.

Root: at least 2 children (if not a leaf).
Internal nodes (non-root): at least ⌈p/2⌉ children.
All leaves at same depth — B-tree is perfectly height-balanced.

B-Tree Node Layout (Internal)

P₁ | K₁ | P₂ | K₂ | P₃ | ... | Kₙ | Pₙ₊₁
where Pᵢ = tree pointer (child pointer), Kᵢ = search key value.
Each key Kᵢ also stores a pointer to the actual data record (so B-tree internal nodes point to records for every key stored).
Node size constraint: p×(pointer size) + (p−1)×(key size) + (p−1)×(record pointer size) ≤ Block size.

5.2   B⁺-Tree — The Standard in Practice GATE Critical

Key Difference: B⁺-Tree vs B-Tree

B-tree: Every node (internal AND leaf) stores pointers to actual data records. A key can appear at most once in the tree.
B⁺-tree: Only leaf nodes store pointers to data records. Internal nodes store only keys as routing guides (no data pointers). All keys in internal nodes are also duplicated in the leaf level. Leaf nodes are linked in a sorted doubly-linked list for efficient range scans.

B⁺-Tree Node Types INTERNAL NODE (order p): ┌────┬────┬────┬────┬────┬────┬────┐ │ P₁ │ K₁ │ P₂ │ K₂ │ P₃ │ K₃ │ P₄ │ ← Tree pointers only, NO data pointers └────┴────┴────┴────┴────┴────┴────┘ (p−1 keys, p tree pointers per node) LEAF NODE (order p_leaf): ┌────┬────┬────┬────┬────┬────┬────┬──────┐ │ K₁ │ R₁ │ K₂ │ R₂ │ K₃ │ R₃ │ K₄ │ Next │ ← Rᵢ = record pointer, Next = next leaf └────┴────┴────┴────┴────┴────┴────┴──────┘ (up to p_leaf−1 key+recordptr pairs + 1 next-leaf pointer)

Order Calculation — The Most Tested GATE Numerical

Internal Node Order Formula

Each internal node stores p tree pointers and (p−1) keys. The node must fit in one disk block:
p × P_size + (p−1) × V_size ≤ Block_size
where P_size = size of a tree pointer (block/page pointer), V_size = size of a key (search field value).
Solve for p: p ≤ (Block_size + V_size) / (P_size + V_size). Take the floor.

Leaf Node Order Formula

Each leaf node stores up to (p_L − 1) pairs of (key, record_pointer) plus one next-leaf pointer:
(p_L − 1) × (V_size + PR_size) + P_size ≤ Block_size
where PR_size = size of a record pointer (points to the actual record on disk), P_size = next-leaf pointer size.
Solve: p_L ≤ (Block_size − P_size) / (V_size + PR_size) + 1. Take the floor.

Block size = 1024 B. Search key (V) = 9 B. Block pointer (P) = 6 B. Record pointer (PR) = 7 B. Find the order of internal nodes and leaf nodes of a B⁺-tree.

INTERNAL NODE: p × 6 + (p−1) × 9 ≤ 1024 6p + 9p − 9 ≤ 1024 15p ≤ 1033 p ≤ 68.87 p = 68 ← internal node order Each internal node: 68 pointers, 67 keys. Min keys in non-root internal node: ⌈68/2⌉ − 1 = 33 keys. LEAF NODE: (p_L − 1) × (9 + 7) + 6 ≤ 1024 (p_L − 1) × 16 ≤ 1018 p_L − 1 ≤ 63.625 p_L ≤ 64.625 p_L = 64 ← leaf node order Each leaf node: up to 63 (key, record-ptr) pairs + 1 next-leaf pointer. Min key-record pairs in a leaf: ⌈(64−1)/2⌉ = 32 pairs.

Height and Search Cost

Maximum Records in a B⁺-Tree of Height h

Max leaf nodes at height h = p^(h−1) (root at level 1).
Max records = p^(h−1) × (p_L − 1).
Height for n records ≈ ⌈log_{⌈p/2⌉}(n / (p_L−1))⌉ + 1 (using minimum fill).
Search cost = h disk I/Os (one per level) + 1 for the actual record.

PropertyB-TreeB⁺-Tree

B-Tree Order p vs B⁺-Tree Internal Order vs Leaf Order

GATE questions sometimes give "order p of a B⁺-tree" without specifying internal vs leaf. The standard convention: order p refers to internal nodes (p tree pointers per node). Leaf nodes have a potentially different order p_L because they store record pointers instead of tree pointers. Always compute both separately when record pointer size ≠ tree pointer size.

Chapter 06
Tuple Relational Calculus — Non-Procedural Queries

6.1   TRC — Syntax, Semantics & Safe Expressions GATE Tested

Relational Algebra (File 02) is procedural — you specify how to get the result step by step. Relational Calculus is non-procedural — you specify what you want, not how. Both are equivalent in expressive power (Codd's theorem). GATE tests TRC for expression reading and safety.

TRC Expression Form

{ t | P(t) }
Read: "The set of all tuples t such that predicate P(t) is true."
t is a tuple variable that ranges over a relation.
P(t) is a formula built from: atomic formulas, connectives (∧, ∨, ¬), quantifiers (∃, ∀).
t.A accesses attribute A of tuple t.
t ∈ R or R(t) means tuple t is in relation R.

Atomic Formulas

  • t ∈ R — tuple t is a member of relation R
  • t.A θ s.B — comparison between attributes of two tuples (θ ∈ {=, ≠, <, ≤, >, ≥})
  • t.A θ c — comparison between an attribute and a constant c

Find all students with CGPA > 9.0:

{ t | t ∈ Student ∧ t.CGPA > 9.0 }

Find names of students who are enrolled in course C101:

{ t.Name | t ∈ Student ∧ (∃ e)(e ∈ Enrollment ∧ e.RollNo = t.RollNo ∧ e.CID = 'C101') }

Find students enrolled in ALL courses (universal quantification):

{ t | t ∈ Student ∧ (∀ c)(c ∈ Course → (∃ e)(e ∈ Enrollment ∧ e.RollNo = t.RollNo ∧ e.CID = c.CID)) }

Read: "All students t such that for every course c, there exists an enrollment e linking t to c."

Safety of TRC Expressions

A TRC expression is safe if it is guaranteed to produce a finite result. An unsafe expression might attempt to generate an infinite set (e.g., all tuples NOT in a relation — which could be infinite).
Safe condition (domain independence): Every value in the result must appear somewhere in the database (in some relation referenced in the formula). This prevents selecting from the "infinite" universe of all possible values.

Unsafe expression example:

-- UNSAFE: selects all tuples NOT in Student → potentially infinite { t | ¬(t ∈ Student) } -- SAFE equivalent: restrict domain to a known finite relation { t | t ∈ Person ∧ ¬(t ∈ Student) } ← safe if Person is finite

Chapter 07
Domain Relational Calculus — Variables over Attribute Domains

7.1   DRC — Syntax & Comparison with TRC GATE Tested

DRC Expression Form

{ <x₁, x₂, ..., xₙ> | P(x₁, x₂, ..., xₙ) }
Read: "The set of all tuples <x₁,...,xₙ> such that predicate P holds."
Domain variables xᵢ each range over the domain of a single attribute (not over whole tuples).
<x₁,...,xₙ> in the result list specifies which domain variable values to include in the output.

Find names and CGPA of students with CGPA > 9 — in TRC vs DRC:

-- TRC version: { t.Name, t.CGPA | t ∈ Student ∧ t.CGPA > 9 } -- DRC version: { <n, c> | ∃r ∃d (Student(r, n, d, c) ∧ c > 9) } -- where Student(RollNo, Name, Dept, CGPA) -- r=RollNo, n=Name, d=Dept, c=CGPA -- r and d are existentially quantified (we don't output them)

Find names of students enrolled in course 'CS101':

{ <n> | ∃r ∃d ∃c (Student(r, n, d, c) ∧ ∃cid (Enrollment(r, cid) ∧ cid = 'CS101')) }
FeatureTRCDRC

Converting RA to TRC to DRC — The Pattern

Projection (π) → list the projected attributes in the result variables. Selection (σ) → condition in the predicate. Join (⋈) → two relation memberships with an equality condition linking them. Union (∪) → OR (∨) between two membership conditions. Difference (−) → AND NOT (∧ ¬). Division (÷) → universal quantifier (∀).

GATE Traps & Numerical Reference

8.1   All the Traps in One Place

#TrapCorrect Understanding
1Derived attribute → column in tableDerived attributes (dashed oval in ER) are NOT stored as columns. They are computed on demand. Age derived from DOB: only DOB gets a column, not Age.
2Composite attribute → single columnA composite attribute (Name → {First, Last}) generates columns only for its leaf components — First and Last. The composite Name itself does NOT become a column.
3Weak entity can have partial participation in identifying relationshipA weak entity ALWAYS has total participation in its identifying relationship. By definition: it cannot exist without its owner. This is double line on the weak entity side — always.
41:N relationship → create new tableFor 1:N, the standard mapping is to put a FK in the N-side table — no new table needed. A new table is only justified when N-side has partial participation and you want to avoid NULLs, but it's not the default.
5M:N → FK in one of the existing tablesM:N relationships ALWAYS require a new junction/bridge table. You cannot represent an M:N relationship by adding a FK to either existing table — that can only represent 1:N or 1:1.
6Primary index is densePrimary index is SPARSE — one entry per data block (not per record). The file is ordered on the key, so you can binary search to the right block and then scan within it. Dense primary index is wasteful and unnecessary.
7Secondary index is sparseSecondary index on a KEY field is DENSE (one entry per record — because the file is not ordered on this field, you cannot skip records). Secondary on NON-KEY can have one entry per distinct value (with pointer to bucket of record pointers).
8B-tree and B⁺-tree are the sameB-tree: every node (internal and leaf) has data record pointers. B⁺-tree: ONLY leaves have data record pointers; internal nodes have tree pointers only. B⁺-tree has higher fan-out and efficient range scans via leaf linking. They are different structures.
9Internal and leaf node order are always the same in B⁺-treeInternal and leaf nodes have different formulas because they store different pointer types. Internal: p×P + (p−1)×V ≤ B. Leaf: (p_L−1)×(V+PR) + P ≤ B. When P (tree pointer) ≠ PR (record pointer), p ≠ p_L.
10TRC and DRC have different expressive powerTRC, DRC, and Relational Algebra are all equally expressive (Codd's theorem). Any query expressible in one is expressible in the others. The difference is syntactic style, not power.
11Any TRC expression is safeUnsafe expressions exist — { t | ¬(t ∈ R) } is unsafe because it denotes all tuples NOT in R, which could be infinite. An expression is safe iff every value in the result is in the active domain (appears in some relation in the formula).
12Multi-valued attribute → column with multiple valuesA multi-valued attribute (double oval) always generates a SEPARATE table with a composite primary key (entity PK + the attribute value). Never put multiple values in a single column — that violates 1NF.

8.3   GATE PYQ Patterns

  • ER mapping MCQ (1 mark): "Which of the following correctly maps a M:N relationship?" → requires new table with both PKs as composite PK. Option traps: FK in one table (wrong), no new table (wrong).
  • Weak entity identification (1 mark): "Which entity is weak in this ER diagram?" → look for: double rectangle, double diamond identifying relationship, discriminator (dashed underline). Always has total participation.
  • B⁺-tree order NAT (2 marks): Given block size, key size, pointer sizes — compute internal order p using p×P + (p−1)×V ≤ B. Then leaf order p_L using (p_L−1)×(V+PR) + P ≤ B. Common mistake: using same formula for both.
  • Index blocks NAT (2 marks): Given file stats — compute bfr, data blocks, index entries, index block size, index blocks. Then binary search cost = ⌈log₂(index_blocks)⌉ + 1 I/Os.
  • TRC expression reading MCQ (1 mark): "What does { t | t ∈ Employee ∧ t.Salary > 50000 } return?" → set of all employee tuples with salary > 50000. Identify what the free variable t ranges over and what the predicate filters.
  • Safety MCQ (1 mark): "Which expression is safe?" → check: does the result set come from values already in the database? Expressions with bare negation (¬ without prior membership constraint) are typically unsafe.
  • Primary vs secondary index MCQ (1 mark): File ordered on field X. Index built on field X = primary (sparse). Index built on field Y (not ordering field) = secondary (dense for key Y, bucket-based for non-key Y).
  • Mapping 1:1 with participation (2 marks): Given a 1:1 relationship with one entity having total participation — FK should go in the total-participation entity's table. Reason: avoids NULL values (every instance on total side has a partner).
